Self-describing card faces in the world-deck renderer

A focused UX pass on Step 9 of the world-deck slice: make player cards explain themselves on the face instead of requiring prior knowledge of the rules.

2026-06-04 module: game / describe / render branch: world-deck-slice touched 4 files (+1 new test)

▦ What prompted it

The world-deck slice (steps 1–9) was already implemented and committed (HEAD b922503). The renderer worked, but a card's behaviour was only legible if you already knew the rules. The user named Sprint as the prime example: its face rendered the literal word Choose… with no hint of the two branches. The ask was to propose a solution first, not jump to code.

⌕ Root cause found while reading the code

The opacity had two sources, and a latent bug sat between them:

  • summariseEffect in render.ts collapsed the two interesting effect kinds to nothing: Modal → 'Choose…', Sequence → 'Multi-step'. Singles were terse abbreviations rendered at 9px (Progress 1, Return 0–2 world).
  • Hazard faces showed cryptic tokens — +Listen, -1hp, +W:Door, Skip draw.
  • A second, divergent description path existed: branchLabel in TableScene.ts hardcoded Sprint's modal options by index (if (idx === 0) return 'Draw cards…'). Those strings were divorced from the card's actual effect and would have silently lied the moment the catalog changed.

◇ Decision: the display surface (asked the user)

I presented three options for where the full text should live — a hover/select detail panel, bigger self-describing faces, or both — with ASCII previews. The user picked bigger self-describing faces: enlarge the cards and always render the full description, accepting the trade that six cards leave less room.

The foundation under all three options was the same and I treated it as non-negotiable regardless of the surface chosen: one pure description module, plus deleting the hardcoded modal labels.

⚙ What was built

  • src/game/describe.ts (new) — single source of every English string for card behaviour. Imports core types only, so it stays on the pure side of the renderer boundary and is unit-tested headless. describeEffect recurses into Modal (header + a bullet per branch) and Sequence (steps prefixed "then …"); describePenalty/describeReward render full sentences; previewPlay computes the live target math, looking through Modal/Sequence to the Progress step via a small dealProgressOf helper.
  • src/game/render.ts — cards grown 120×160 → 150×196; player faces render the whole describeEffect block, hazard faces render full penalty/reward sentences and a "N / to clear" cost. Deleted summariseEffect, penaltyLabel, rewardLabel.
  • src/game/TableScene.ts — deleted the index-keyed modal labels; the chooser now derives each label from the card's real Modal effect. Re-centred rows for the wider cards. Added a hover handler that shows previewPlay in the hint line during hazard targeting.
  • src/game/describe.test.ts (new) — 14 tests across every effect/penalty/reward kind and the recursive Modal/Sequence cases.

Surprise: the TargetSpec had already thrown away what the label needed

The modal chooser worked off TargetSpec, which only carries targeting shape (hazard / returnWorld / none), not the underlying effect ("draw 2"). That is why the original author resorted to hardcoded index labels — there was nothing richer to read at that point. The fix was to look the card up in state.hand and read card.effect.branches[idx] directly, since the full Effect is still on the card. describeEffect operates on Effect, not TargetSpec.

Verification went all the way to pixels

typecheck, lint, and bun test (128 pass, 14 new) were necessary but not sufficient for a layout change. The Playwright MCP failed (Chromium distribution 'chrome' is not found), so I drove the system /usr/bin/chromium via a one-off playwright script against the running dev server (port had bumped to 5174). Screenshots confirmed Sprint reads its two branches on the face, and selecting Explore + hovering Rubble shows "Deals 1 → clears Rubble" in the hint. No console errors.

⚐ Context worth keeping for next time

  • Description text is a renderer concern, kept pure. describe.ts lives in src/game/ (not core) because it is presentation English, but it imports core types only and has no Phaser — so it is the testable seam between "what a card is" (core) and "how it reads" (Phaser). If the sim ever wants human-readable logging, it can reuse it.
  • Six self-describing cards fit, barely. 150px cards at 156px spacing span ~780px on the 900px table; a full six-wide row sits close to the edges. Hand is capped toward 6 and splits into a world row + player row, so a six-wide single row is rare in practice. If polish later wants breathing room, this is the constraint to revisit.
  • The Gaps callout in a plan is a checklist, not just prose. "Plain-text target previews" was named in-scope but silently skipped in the first implementation pass and only caught when re-reading the plan this session.
  • previewPlay returns null for non-Progress plays (Med Kit, or Sprint's draw branch), which is what keeps the hint from showing nonsense math on a heal.
