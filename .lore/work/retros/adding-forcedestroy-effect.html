Adding the ForceDestroy card effect end-to-end

What it took to add one new CardEffect kind to the engine: queue on discard, destroy a random player card from the next hand. Bird-building's Gripping Talon. 2026-06-07.

The setup

The session opened pointed at the brainstorm theme-mechanical-differentiation.html, where the bird world's "displace" threat-verb was flagged as the one idea that genuinely needs a new effect ("the bird carries your stuff off — no discard/steal-a-random-player-card effect exists today"). The request: add a ForceDestroy effect, put it on Gripping Talon's onDiscard, destroying a random card from the player's next hand.

First move was reading the request as a doc edit because of where we'd left off. It was a code task; the brainstorm was only the breadcrumb. No artifact (spec/plan) was written — this went straight from request to implementation after reading the engine.

What the change actually touched

"One new effect" landed across eight non-test files plus four test files. The spread came from two things: a new effect needs a slot in the discriminated union (which ripples into every exhaustive switch), and "next hand" is a deferred effect that can't resolve where it's invoked.

The timing problem: "next hand" can't resolve at discard

A discard fires onDiscarded mid-turn through handleDiscardHazard. The "next hand" does not exist yet — it gets built at the following EndTurn in refillHand. So the effect had to split into two stages, mirroring how skipDrawNext already works (a boolean queued at discard, consumed at the next refill, emitting DrawSkipped when it fires):

1.Queue. ForceDestroy increments pendingForceDestroy. No card touched, no event.
2.Resolve. resolveForceDestroy runs at turn start after the refill, drains the counter against the new hand, emits one CardDestroyed per card taken.

skipDrawNext was the template followed throughout — both for the queued-state-field pattern and for the "emit the event at resolution, not at queue time" choice.

Where it got bumpy

Exhaustive switch with no default forced a touch

describeEffect in describe.ts switches over every CardEffect kind with no default branch. Adding a union member broke its compile until a ForceDestroy case was added. This was the helpful kind of break — the type system pointed at the one renderer file that needed a string. By contrast, available.ts (structuralSpec / isPlayable) and sim/policy.ts all have default cases or switch on TargetSpec rather than CardEffect, so they silently did the right thing (ForceDestroy never becomes a playable player card, so it correctly reports none/false).

Adding a required GameState field broke 6 test literals

pendingForceDestroy being required (not optional) meant every hand-built full GameState literal failed typecheck. tsc --noEmit was the authoritative finder: 6 errors across cards.test.ts, describe.test.ts, feedback.test.ts, and hud.test.ts (3 sites). Test helpers that spread a createWorld base (the makeState pattern) were unaffected — only the from-scratch object literals needed the field. Grepping skipDrawNext: over-reported candidate sites; tsc gave the real list.

The JSON change is never typechecked

World JSON is cast as unknown as RawCardSource in worldManifest.ts, so changing Gripping Talon's onDiscarded to an effect kind that didn't exist yet would not have been caught at the data layer. It only works because the union member was added in types.ts; the runtime applyEffect switch is what actually validates the kind. The cast bypasses the data-side safety net entirely.

A design fork resolved without asking

"A random card from the player's next hand" is literally ambiguous — the hand holds both player and world cards. Resolving it as written would let the effect destroy a world hazard, which helps the player and contradicts a threat. resolveForceDestroy was scoped to player cards only, with two further unspecified calls made and documented in code comments:

  • Fizzles, doesn't linger. Excess charges over available player cards are drained to 0, not carried into a later turn.
  • Stacks. Two discards queue two charges and take two distinct player cards.

Renderer reality: the destruction lands without juice

State-diff renderer, not event-driven

Tracing whether CardDestroyed produces any feedback turned up that TableScene never reads the event stream at all. It reconciles card objects against game_.state.hand in drawAll() after every dispatch: cards whose id is no longer in the hand have their Phaser container destroy()ed (after killing tweens). So a force-destroyed card just vanishes on the next repaint. There is no CardDestroyed handler anywhere in src/game — the existing DestroyCardInHand effect (Regroup, Steady) has the same silent removal. Feedback was explicitly deferred ("we'll add feedback later").

Verification

tsc --noEmit clean, eslint on changed files clean, full suite 326 pass / 0 fail (6 new tests). New tests: resolveForceDestroy unit coverage in draw.test.ts (no-op, single destroy, world-cards-spared, fizzle, multi-charge-distinct) and a discard→EndTurn integration in reduce.test.ts (charge queued but current hand untouched; charge resolves against the refilled hand). The integration test needed world cards seeded in worldDraw so the livelock guard ("no world cards anywhere + no way to make them") wouldn't fire WorldLost and mask the destruction being tested.

Shape of "add an effect" in this engine

For the next person adding a CardEffect kind, the surfaces that came up:

  • Union member in types.ts + the applyEffect case in effects.ts are the irreducible minimum.
  • describeEffect in describe.ts will fail compile until handled (no default). That's the trip-wire that reminds you the card needs display text.
  • A deferred effect (acts on a future hand/turn) needs a queued GameState field + a resolution point in the turn cycle. skipDrawNext is the existing reference pattern; startTurn in energy.ts is where post-refill resolution hooks in.
  • A new required GameState field breaks from-scratch literals in tests; createWorld-spread helpers survive. Let tsc enumerate the sites.
  • World JSON effect changes are unchecked casts — the union member must exist or it fails only at runtime.
  • If the effect should feel like anything on screen, that's separate work: the renderer is state-diff based and currently ignores the event stream.
