Implementation plan: player exhaust + world self-transform

Two new card-lifecycle behaviors for the Shattered Worlds engine, plus the one shared primitive they need (self-removal from hand).

status: executed date: 2026-06-07 modules: core-engine, core-model, game-interaction, worlds-data
Source artifacts
  • Origin: brainstorm — theme mechanical differentiation ("Volcano: the floor is lava" = burn/exhaust; "Zombie: corpses / multiply" = self-transform).
  • Precedent: retro — adding the ForceDestroy effect end-to-end (the canonical checklist for touching the effect system).
  • Constraint: reference — theme-authoring rules (closed effect vocabulary; "a new kind is a core change to flag, not a theme edit").

Decisions settled with the user

Player exhaust → boolean flag on the card

A played card with exhaust: true is destroyed instead of discarded. Whole-card / always-on, not a conditional Sequence step. Chosen over a dedicated ExhaustSelf effect kind.

Consequence: not an effect kind, so it skips the entire effect-switch checklist (applyEffect, available.ts). Pure reducer + type change.

World self-transform → degrade onto worldDraw top

An undealt-with world card removes itself at end of turn and pushes a worse card onto the top of worldDraw (drawn next turn). Chosen over in-place hand replacement.

Consequence: the "spawn" half is the existing AddWorldCardToTop. Only the "remove self" half is new — one effect kind, DestroySelf.

Shared mechanism: selfId threaded into applyEffect

onEndOfTurn runs with no Action, so an effect there has no way to name "the card that fired me." We add an optional selfId parameter to applyEffect, passed through the Sequence/Modal recursion and supplied by the end-of-turn loop. The player-play path does not need it (exhaust is a flag), keeping selfId single-purpose: "the world card whose hook is firing, against the current hand." This avoids two divergent notions of "self" (the card is already in discard by the time a played player card's effect runs).

A3Exhaust-on-play in the reducerdeps: A2

In handlePlayCard, branch on card.exhaust when building the post-play state.

  • Not exhaust (today's behavior): remove from hand, prepend to playerDiscard.
  • Exhaust: remove from hand, do not add to any zone, and append a CardDestroyed event after the existing CardPlayed event.

Event ordering (do not reorder the existing pipeline): the only structural change is the stateAfterPlay branch and one appended event. Keep spendEnergy (which may emit EnergyChanged) and applyEffect exactly as today, and append CardDestroyed after the effect events. Correct order: CardPlayed → EnergyChanged (if any) → effect events → CardDestroyed. Reuse the existing CardDestroyed event — no new event type.

// 1. only this branch changes vs today:
const handAfterPlay = state.hand.filter((c) => c.id !== cardId)
const stateAfterPlay: GameState = card.exhaust
  ? { ...state, hand: handAfterPlay }                       // gone, not discarded
  : { ...state, hand: handAfterPlay,
      playerDiscard: [card, ...state.playerDiscard] }

const events: GameEvent[] = [{ type: 'CardPlayed', cardId }]
// 2. spendEnergy(stateAfterPlay) -> EnergyChanged ... UNCHANGED
// 3. applyEffect(stateAfterSpend, card.effect, action) -> effect events ... UNCHANGED
// 4. append the destroy AFTER effect events:
if (card.exhaust) events.push({ type: 'CardDestroyed', id: cardId })

A4Surface "Exhaust" on the card facedeps: A1

The flag lives on the card, not the effect, so describeEffect(effect) can't see it. The badge belongs where the card object is rendered.

  • Add an "Exhaust" badge/line in card rendering (src/game/interaction/cardObjects.ts) gated on card.exhaust, or thread the flag into whatever builds the face text.

Research note: the renderer is state-diff based and has no CardDestroyed handler — an exhausted card simply disappears from hand (same as DestroyCardInHand today). The badge is therefore the only player-facing signal that the card won't come back, which is exactly why it is required: without it, exhaust is invisible until the player notices the card never returns.

Alternatives considered

OptionWhy not
ExhaustSelf as a dedicated effect kind (composable in Sequence) More surface (effect switches) and conditional exhaust isn't a current need. User chose the simpler whole-card flag. Revisit if a card ever needs to exhaust only on one Modal branch.
A single TransformInto { template } effect bundling remove-self + spawn Bakes in placement and duplicates AddWorldCardToTop. DestroySelf is the smaller, composable primitive and matches the codebase's "small effects, compose with Sequence" style.
In-place hand replacement for the transform Would need a new mint-into-hand path (no hand destination exists in Dest). User chose degrade-onto-worldDraw-top, which reuses existing machinery.
Pure AddWorldCardToTop on the corpse's onEndOfTurn (the brainstorm's "data-only" multiply) That spawns a Zombie but leaves the corpse in hand — accumulation, not transformation. The user wants the corpse to become the zombie, which requires self-removal.

Risks & watch-items

  • Silent renderer: no CardDestroyed handler exists; destroyed cards just diff out of hand. Exhaust feedback is limited to the A4 badge. Acceptable, matches existing DestroyCardInHand.
  • Unchecked world JSON cast (B6): a bad kind fails only at runtime; the catalog-build test is the guard.
  • EoT loop snapshot (B4): the no-re-fire guarantee depends on the loop iterating the entry-time snapshot. Comment it; assert it in B7.
  • Livelock guards in handleEndTurn count world cards in hand / world-introducing player cards. Transform keeps a world card present (it just moves to worldDraw), so the guards stay satisfied — but B7's "world deck not empty" path should be sanity-checked.
