## Codebase Context

- `src/core/engine/game.ts` already exposes the seam this plan should preserve: `dispatch(action) -> { state, events }`.
- `src/core/model/types.ts` already defines a broad `GameEvent` union covering card play, progress, discard/destruction, act advancement, and terminal win/loss facts. The missing capability is persistence across dispatches and lifecycle envelopes around a session.
- `src/game/scenes/TableScene.ts` is the live runtime hotspot. It creates the session with `createGame(...)` in `create()` and currently discards the `dispatch(...)` return value in its private `dispatch(action)` method.
- `src/sim/run.ts` uses `reduce(...)` directly and ignores events today. That is a useful guardrail: the stream foundation must remain optional so headless and test consumers can continue to resolve valid runs without subscribing.
- `.lore/work/design/core-render-architecture.html` already commits the project to a pure, deterministic core and a renderer/runtime layer that reads ordered semantic events as an animation script.
- `.lore/work/brainstorm/shattered-worlds-meta-progression.html` supplies the first real downstream pressure: feats, witness knowledge, echo-of-last-run behavior, and other progression systems all need authoritative act/run history outside Phaser.

## Planning Assumptions

Because the user was unavailable for follow-up questions, this plan proceeds with the narrowest architecture that satisfies the spec without changing core ownership boundaries:

1. Run lifecycle envelopes stay outside `src/core/`.
2. The existing `GameEvent` union remains gameplay-only; lifecycle and fan-out metadata are added by a runtime wrapper.
3. The runtime stream groups gameplay events by accepted dispatch so causal ordering and dispatch atomicity stay explicit.
4. Session identity is generated by the runtime wrapper through an injectable factory, with a default implementation provided outside the core.

## Implementation Steps

### Step 1: Define the runtime event-stream contract

**Files**: `src/game/runtime/gameplayEventStream.ts`, `src/game/runtime/gameplayEventStream.test.ts`  
**Addresses**: REQ-EVENTS-1, REQ-EVENTS-2, REQ-EVENTS-3, REQ-EVENTS-4, REQ-EVENTS-7, REQ-EVENTS-8, REQ-EVENTS-9  
**Expertise**: none needed

Create a Phaser-free runtime contract that wraps authoritative core events without moving them into `src/core/`. Define:

- a discriminated union such as `RunStreamItem = RunStarted | GameplayBatch | RunEnded`;
- `RunStarted` with `sessionId`, `worldId`, `seed`, and applied setup modifiers;
- `GameplayBatch` with `sessionId`, the accepted `Action`, the ordered `GameEvent[]`, and the corroborating post-dispatch `GameState`;
- `RunEnded` with `sessionId`, terminal outcome, and final act reached.

Keep the top-level runtime stream type explicit about dispatch grouping rather than flattening events immediately. That preserves the existing "state plus ordered events" seam and gives save/checkpoint policy a stable unit of observation. Lock the minimum downstream-usable facts into this step: subscribers must be able to recover session correlation, card-play facts, progress dealt, discard/destruction facts, act boundaries, and terminal outcome from the emitted items without inspecting renderer state.

### Step 2: Build the multi-subscriber observation boundary

**Files**: `src/game/runtime/gameplayEventStream.ts`, `src/game/runtime/gameplayEventStream.test.ts`  
**Addresses**: REQ-EVENTS-5, REQ-EVENTS-10, REQ-EVENTS-11  
**Expertise**: none needed

Add a small subscribe/unsubscribe API around the runtime stream contract. Keep it synchronous and side-effect free with respect to gameplay resolution:

- subscribers observe emitted stream items after the core has already resolved the action;
- the wrapper does not let subscriber return values influence gameplay;
- no subscriber registration lives inside `GameCore` or `src/core/`.
- delivery iterates a snapshot of current subscribers so subscribe/unsubscribe during emission cannot reorder or skip peers for the current item;
- if a subscriber throws, the wrapper still finishes delivering that item to the remaining snapshot listeners, then surfaces the first error explicitly to the caller rather than swallowing it.

Cover the bus with tests for multi-listener fan-out, delivery order, and "no subscribers attached" behavior so the stream remains optional.

### Step 3: Wrap `createGame` with session lifecycle emission

**Files**: `src/game/runtime/gameplaySession.ts`, `src/game/runtime/gameplaySession.test.ts`, `src/game/runtime/gameplayEventStream.ts`  
**Addresses**: REQ-EVENTS-1, REQ-EVENTS-3, REQ-EVENTS-5, REQ-EVENTS-6, REQ-EVENTS-7, REQ-EVENTS-8, REQ-EVENTS-9, REQ-EVENTS-10  
**Expertise**: none needed

Introduce a wrapper around `createGame(...)` that owns session lifecycle emission while delegating gameplay truth to the existing core:

- emit the run-start envelope exactly once when a single-world session is created;
- forward each accepted dispatch as one gameplay stream item containing the ordered `GameEvent[]` returned by the core;
- detect transition to terminal `state.status` and emit the run-end envelope exactly once;
- inject a `makeSessionId` factory so tests can pin IDs without coupling gameplay determinism to ID generation.

Keep the wrapper interface close to `GameCore` so existing call sites need minimal change.

### Step 4: Wire the renderer to the observed session boundary

**Files**: `src/game/scenes/TableScene.ts`, `src/game/runtime/gameplaySession.ts`, `src/game/tests/gameplaySessionIntegration.test.ts`  
**Addresses**: REQ-EVENTS-2, REQ-EVENTS-5, REQ-EVENTS-12  
**Expertise**: none needed

Replace the direct `createGame(...)` call in `TableScene` with the session wrapper and make that wrapper the renderer's only gameplay-event path. Keep renderer responsibilities unchanged:

- `TableScene` still owns presentation, redraws, and scene transitions from authoritative session state;
- any renderer code that needs gameplay events subscribes to the session wrapper instead of reading a parallel event path from a raw `createGame(...)` result;
- the stream boundary sits beside the renderer rather than inside it;
- no run-stats, meta-progression, or save implementation lands here yet.

This step should only prove that the live runtime now emits stream data from one dispatch flow that multiple future consumers can subscribe to, without creating a second gameplay-truth path inside `TableScene`.

### Step 5: Preserve headless and test optionality

**Files**: `src/sim/run.ts` (only if needed for compile surface), `src/sim/tests/*` (only if needed), `src/game/runtime/gameplaySession.test.ts`  
**Addresses**: REQ-EVENTS-11  
**Expertise**: none needed

Verify that the sim and other headless consumers can continue to ignore the stream entirely. Prefer leaving `src/sim/run.ts` untouched; only change shared types or exports if the new runtime module would otherwise force a dependency on Phaser or on active subscribers.

The intended outcome is that stream-aware consumers opt in, while existing pure-core consumers still run valid worlds with no subscription code.

### Step 6: Add requirement-driven tests and validation gates

**Files**: `src/game/runtime/gameplayEventStream.test.ts`, `src/game/runtime/gameplaySession.test.ts`, `src/game/tests/gameplaySessionIntegration.test.ts`, `src/core/tests/reduce.test.ts` (only if payload gaps require it)  
**Addresses**: REQ-EVENTS-1 through REQ-EVENTS-11  
**Expertise**: none needed

Add tests that prove the foundation works before follow-up subscriber specs exist:

- emitted gameplay batches preserve the exact core event order from one accepted action;
- multiple subscribers observe identical session history;
- start and end envelopes correlate one single-world session from creation to terminal outcome;
- end emission happens exactly once;
- adding subscribers does not alter core output for the same seed and action sequence;
- emitted items expose the facts needed to derive card-play tracking, progress dealt, discard/destruction tracking, act boundaries, and terminal outcome without touching renderer state;
- emitted contract types stay renderer-free: no Phaser objects, timing tokens, tween instructions, particle choices, or other presentation-only payloads appear in the runtime stream;
- optional consumers can be absent without breaking resolution.

Target 90%+ coverage for the new runtime modules through unit tests even though the repository does not currently expose a dedicated coverage gate. Validate the finished implementation with the repository gates in `CONTRIBUTING.md`: `bun run lint`, `bun run typecheck`, `bun run test`, and `bun run build`.

## Open Questions

- **`CardPlayed` payload depth**: the current core event only carries `cardId`. If a future feat needs card cost or other card metadata directly from the stream, decide whether to enrich `CardPlayed` in core or rely on the corroborating `GameState` included in the dispatch payload.
- **Applied modifier shape**: the plan assumes the run-start envelope can carry an empty or minimal `modifiers` payload until progression/setup systems are specified.
- **Failed dispatch visibility**: this plan assumes only accepted actions emit gameplay stream items, matching the existing reducer contract that throws on illegal actions rather than returning partial events.
