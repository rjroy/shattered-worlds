src/data/worldHelpManifest.ts

Types to define

export interface WorldMechanicNote {
  title:  string   // e.g. "Zombies grow"
  detail: string   // 1–2 sentences explaining the mechanic
}

export interface WorldHelpData {
  mechanics: readonly WorldMechanicNote[]
}

export const worldHelpManifest: Record<string, WorldHelpData>

No core/engine imports

This file is UI-only. It must not import from src/core or src/game/view. The only allowed imports are the types defined in this file itself.

zombie-big-box entry — 4 mechanic notes

  • Zombies grow — A Zombie in your hand deals 1 damage and adds another Zombie to the world deck at end of every turn. Clear them before they multiply.
  • Corpses transform — A Corpse becomes a Zombie at end of turn. Corpses cannot be discarded; they must be cleared or endured.
  • Costly discards — Discarding a Zombie costs 5 HP. Know your limits before you cut your losses.
  • The Walker and the Door — The Walker (Act 3, cost 10) is the final hazard. Clearing it adds Summon Door to your hand. Play Summon Door to place the Door (a Hidden hazard, cost 4). Clear the Door to survive the world.

Exact wording is flexible; the four topics must be covered.

bird-building and highway-volcano entries — placeholders

One mechanic note each. Non-empty title and detail strings acknowledging that full content is deferred. Example:

{ title: 'Mechanic notes coming',
  detail: 'This world\'s mechanics are still being tuned. '
        + 'Notes will be added after rebalancing.' }

New test Write worldHelpManifest completeness test
src/core/tests/worldHelpManifest.test.ts

Mirrors the pattern in src/core/tests/worldDisplayManifest.test.ts exactly. Two it() blocks:

import { describe, expect, it } from 'bun:test'
import { worldManifest } from '../../data/worldManifest'
import { worldHelpManifest } from '../../data/worldHelpManifest'

describe('worldHelpManifest', () => {
  const worldIds = Object.keys(worldManifest)  // keys only — never call builders

  it('has an entry for every world in worldManifest', () => {
    for (const id of worldIds) {
      expect(worldHelpManifest[id]).toBeDefined()
    }
  })

  it('every entry has at least one mechanic note with non-empty title and detail', () => {
    for (const id of worldIds) {
      const entry = worldHelpManifest[id]!
      expect(entry.mechanics.length).toBeGreaterThan(0)
      for (const note of entry.mechanics) {
        expect(note.title.length).toBeGreaterThan(0)
        expect(note.detail.length).toBeGreaterThan(0)
      }
    }
  })
})

Calling the builder functions (e.g. worldManifest['zombie-big-box']()) throws CatalogError in the test environment because the Phaser asset cache is absent. The test must not call them.

Guard — throw on missing manifest entry

const helpData = worldHelpManifest[worldId]

  throw new Error(`No help entry for worldId: ${worldId}`)
}
const displayData = worldDisplayManifest[worldId]
if (displayData === undefined) {
  throw new Error(`No display entry for worldId: ${worldId}`)

Container setup

Matches the win/loss overlay pattern:

const container = scene.add.container(CANVAS_W / 2, CANVAS_H / 2)
container.setDepth(1000)
container.setVisible(false)

Background rectangle — must call setInteractive()

Uses a slightly higher alpha than win/loss (0.92 vs 0.8) so text is legible over the game board.

const bg = scene.add.rectangle(0, 0, CANVAS_W, CANVAS_H, 0x080a12, 0.92)
bg.setInteractive()  // swallows pointer events; blocks game board input (topOnly default)
container.add(bg)
topOnly reliance: Phaser's input system defaults to topOnly = true, meaning only the highest-depth interactive object under the pointer fires events. The background rect at depth 1000 therefore blocks cards and buttons at depth 0 without any per-card disableInteractive() loop. If topOnly is ever disabled globally, this behavior changes — see REQ-HELP-3.

Section 1 — HOW IT WORKS

Positioned near top of panel. All Y coordinates are relative to the container center (0,0) at canvas center (450, 300). Approximate anchor for section 1: y ≈ −230.

  • Section header: "HOW IT WORKS" — gold, uppercase, small font
  • Content line: `HP: ${WORLD_CONSTS.startHp} · Energy: +1 per turn · Hand: ${WORLD_CONSTS.maxHandSize} cards · Acts: ${totalActs}` — muted text, single line

Section 2 — KEYWORDS (two-column layout)

Left column starts at x ≈ −380, right column at x ≈ 10. Each column is 380px wide (wordWrap: { width: 380 }). Rows are positioned with variable heights — advance y by textObject.height + rowGap after each entry, not by a fixed stride. The Hazard entry wraps to ~3 lines; other entries are single-line.

Left column entries (in order): Energy, Progress, Hazard, Hidden. Right column: Creature, Slow, ForceDestroy.

Keyword names use TEXT.textKeyword ('#88ccff'). Definitions use TEXT.textMuted. Render each entry as two separate text objects (name + definition) OR as a single text object with the name bolded via Phaser's BBCode/style tags — either approach is acceptable.

Variable row heights: The Hazard entry has a three-part definition (clear / discard / end-of-turn). At 370px wrap width it will be significantly taller than the single-line entries. Both columns must be laid out by tracking a running y offset, not by assuming a fixed row height.

Section 3 — IN THIS WORLD

Header: `IN THIS WORLD — ${displayData.name}`. Then for each note in helpData.mechanics:

  • Title: note.title — accent color (TEXT.textKeyword), bold
  • Detail: note.detail — muted, word-wrapped to ~750px width

Advance y by each rendered text height + gap after each note.

X dismiss button

A simple scene.add.text or CommonButton labeled "×", positioned top-right of the panel (approximately x = 380, y = −265). Wire the dismiss internally:

closeBtn.setInteractive({ useHandCursor: true })
closeBtn.on('pointerup', () => container.setVisible(false))
container.add(closeBtn)

Add children to the container in this order: background rect first (behind everything), then section content, then close button last (on top).

New instance properties (add after the win/loss screen declarations)

In create() — after winScreen / lossScreen creation (lines 204–205)

this.helpOverlay = createHelpOverlay(
  this,
  this.worldId_,
  this.game_.state.totalActs,
)

const questionStyle = textStyle({
  fontSize: '16px',
  fontStyle: 'bold',
  color: TEXT.textLight,
})
this.questionBtn = new CommonButton(this, 860, 22, '?', questionStyle)
  .on('pointerup', () => this.helpOverlay.setVisible(true))

this.input.keyboard?.on('keydown-ESC', () => {
  if (this.helpOverlay.visible) this.helpOverlay.setVisible(false)
})
Import additions needed: createHelpOverlay is exported from overlays.ts — add it to the existing import. TEXT is not yet imported in TableScene.ts — add it to the presentation import line.

In drawAll() — after the win/loss visibility block (lines 326–330)

this.questionBtn.setVisible(state.status === 'playing')
if (state.status !== 'playing') this.helpOverlay.setVisible(false)

The first line hides the ? button on win/loss screens. The second dismisses the help overlay if the player wins or loses while it is open — without it, the overlay persists over the win/loss screen. During normal gameplay (status === 'playing') the help overlay's visibility is managed by user action only and must not be touched.

No changes to drawAll()'s card reconciliation loop

The overlay's input blocking is handled by the background rect's setInteractive(). No card or button needs disableInteractive() while help is open.
