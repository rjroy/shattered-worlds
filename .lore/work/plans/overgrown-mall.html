Decisions this plan makes (review these first)

The spec leaves these open or in tension with prior artifacts. Each is cheap to change now, expensive mid-step.

D1This plan supersedes the prior plan's P6 design for DealProgressScaled. The shard-response-archetypes plan (P6, never started) designed it with per: 'emptyHandSlot' as a string and a mandatory live preview via dealProgressOf. The Mall brainstorm's Q3 changed the counter to a discriminated-union spec object ({ kind: 'KeywordInHand', keyword: 'Spore' }), and REQ-MALL-21 relaxes the preview to "existing card text/describe path at minimum". This plan implements the Q3 shape with static rule text ("Add 1 Progress, +1 per Spore in hand") and no live preview; EmptyHandSlot becomes a future union member (out of scope per spec). On approval of this plan, mark the prior plan's P6 section superseded with a pointer here, so two competing designs don't coexist in approved plans.
D2Intrusion hue: #35e0a0 (verdant green, cyan-leaning), tunable. The green-on-green risk is real: zombie-big-box owns #9bff6a (yellow-leaning acid green). Existing hues: starter #b98aff, bird #7cc6ff, volcano #ff6a3c. A cyan-leaning verdant green keeps REQ-MALL-20's "overgrowth-green" while sitting ~55° of hue away from zombie's acid. Exact value is tuning; the requirement is the at-a-glance distinction.
D3Keywords: optional on the template, required on the minted card. PlayerCardTemplate gains keywords?: readonly Keyword[] (existing JSON loads unchanged); mintCard stamps keywords: readonly Keyword[] (defaulting []) onto every minted PlayerCard, giving parity with WorldCard so KeywordInHand counts state.hand uniformly with no undefined checks. Per the ForceDestroy retro, let tsc --noEmit enumerate any test literals that break — don't grep.
D4The new None describe text cannot leak onto world cards. Verified: world-hook rendering guards with if (effect.kind === 'None') return [] at CardView.ts:138, before describeEffect is called. The pinned text "play to prune (leaves the run)" therefore appears only on player card faces (CardView.ts:217 path). Known edge: a None step inside a Sequence would render the text; no data does this and none is planned — accepted, no special-casing.
D5Counter resolution lives in effects.ts. A small exported resolveCounter(state, spec): number beside applyEffect — exported so tests hit the counting logic directly. No new file; one counter kind doesn't justify a module. Revisit when EmptyHandSlot lands.
D6sim/policy.ts is in scope even though the spec never mentions it. The add-an-effect checklist (ForceDestroy retro, prior plan's cross-cutting rules) includes it, and it has the same silent-default trap as available.ts. Both new behaviors get explicit policy handling: the sim must be able to play a Spore (prune) and play Bloom at a hazard. Without this, sim-driven tests and balance probes silently never exercise the Mall's signature verbs.
D7Spore visibility = the world-card keyword line, applied to player cards. World cards render keywords at CardView.ts:274 (9px line under the name); the player branch has no equivalent. Add the same keyword line to the player branch and shift the effect-text block down when keywords are present (effect text currently starts at -CARD_H/2 + 28, which would collide). This satisfies REQ-MALL-21 with an existing pattern, no new UI vocabulary.
D8Inset keys: registered placeholders, per the spec (not the omit-the-key precedent). Bird/volcano shipped by omitting insetKey; REQ-MALL-19 instead requires every mall-inset-* key present in JSON and registered (W2: an unregistered key is a runtime missing-texture). All mall-inset-* keys point at a single shared placeholder webp until real art lands. This keeps the JSON final — art drops in by replacing files, not editing data.
