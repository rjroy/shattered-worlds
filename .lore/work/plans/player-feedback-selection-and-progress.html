Implementation plan: target clarity before commit and partial hazard progress

Renderer-only feedback work, sequenced over a structural reuse-refactor of TableScene.drawAll().

Source artifact: specs/player-feedback-selection-and-progress.html (REQ-FEEDBACK-1 … 12). Driven by the spec-ready brainstorm (Q1–Q5). Validation in Step 12 checks every requirement back against this spec.
The spec's hard constraint (FEEDBACK-11): nothing here may touch src/core, GameState, or the reducers. All work lands in src/game.
Key architecture decision (confirmed with user, 2026-06-05). Today drawAll() destroys and recreates every card container on every cycle (TableScene.ts:202-205). That fights the ring animations (FEEDBACK-8/9), which need objects that survive a state change. We refactor drawAll() to reuse card containers (diff by card id: update in place, create new, destroy removed) rather than the minimal cache-and-tween workaround. Consequences that ripple through the whole plan:
  • The cost ring becomes a persistent child of its card. FEEDBACK-8 (fill) and FEEDBACK-9 (drain) collapse into one code path: updateCostRing(container, targetFraction) always tweens from the ring's currently displayed value to the new fraction. Banking progress sets a higher target (fills); end-of-turn sets target 0 (drains). This is the spec's "one clock running in two directions" (FEEDBACK-8) achieved literally.
  • No progress cache and no event-snapshot is needed for the drain. The spec's FEEDBACK-9 suggested the renderer cache last-known progress because it assumed the ring object is destroyed; with reuse, the ring's own displayed value is the pre-reset value. We deliver the same observed behavior more simply. (Documented as a faithful deviation from the spec's suggested mechanism.)
  • Hover emphasis (FEEDBACK-1) and the connector (FEEDBACK-2) attach to persistent objects, so they no longer race the rebuild. Risk: a hover-scaled card must have its scale restored on hover-out and must not be stomped by a concurrent drawAll() — handled by tracking the hovered id and re-applying base transform in the update path.
  • Possible side benefit: the open clicking/input bug from the visual-identity rollout (re-creating interactive containers every frame is a Phaser smell) may improve. Not guaranteed — Step 3 and Step 12 add explicit input verification rather than assuming a fix.

S1 Pure feedback module: ring fraction + connector-style resolver unit parallel-safe
Covers: FEEDBACK-7 (math), FEEDBACK-3 (mapping), FEEDBACK-10 (single-source agreement)

Create a new Phaser-free module so the load-bearing logic is unit-testable independent of the renderer (satisfies the spec's "pure function" demands and the structural constraint).

New: src/game/feedback.ts, src/game/feedback.test.ts
What it contains
  • ringFraction(progress: number, cost: number): number — returns cost > 0 ? Math.min(1, Math.max(0, Math.min(progress, cost) / cost)) : 0. Guards divide-by-zero.
  • type ConnectorStyle = 'progress' | 'destroy' | 'return' and selectConnectorStyle(effect: CardEffect): ConnectorStyle | null — keyed off effect kind, looking through Modal/Sequence the same way dealProgressOf does (describe.ts:112): a DealProgress reachable → 'progress'; DestroyCardInHand → 'destroy'; ReturnWorldCards → 'return'. Imports core types only.
  • The connector resolver and the cost ring must read progress the same way previewPlay does (FEEDBACK-10). Keep previewPlay as the one place that turns (card, target, progress) into a sentence; ringFraction only formats the same numbers. The agreement test below enforces this by construction.

Why a new module and not describe.ts: describe.ts is the English-string layer; ring/connector math is a different concern. Both stay Phaser-free, so both are headless-testable. Cohesion over colocation.

Validation gate New unit tests pass: ringFraction returns 0 at progress 0, 0.5 at 1/2, 1.0 at progress≥cost, clamps (progress 3, cost 2 → 1.0), and is correct at the cost-10 boss; selectConnectorStyle maps DealProgress→progress, DestroyCardInHand→destroy, ReturnWorldCards→return (incl. through Modal/Sequence); agreement test (sample at minimum progress = cost−1 "not yet", progress = cost "clears", progress = cost+1 "already over"): ringFraction reaches 1.0 exactly when previewPlay reports "clears". bun test green.

S5 Ring fill + end-of-turn drain animation (one clock) manual after S4
Covers: FEEDBACK-8 (fill on bank), FEEDBACK-9 (drain at end of turn)

Make updateCostRing animate from the ring's currently displayed fraction to the new target, instead of snapping. Because the ring persists (S3), banking and the per-turn reset are the same tween in opposite directions.

Edit: src/game/render.ts (updateCostRing), src/game/TableScene.ts (drive the update during drawAll)
Concrete shape
  • Track the ring's displayed fraction on the ring object. In updateCostRing, if target ≠ displayed, killTweensOf the ring then scene.tweens.add a tween over a shared duration (~200–400ms, spec FEEDBACK-8) that interpolates a { value } proxy and redraws the arc on onUpdate. First render (no prior displayed value) snaps.
  • On a progress play, the post-dispatch drawAll recomputes a higher fraction → ring fills. On EndTurn, progress is wiped to {} (reduce.ts:129), the held hazard's card persists (reduce.ts:127), the post-dispatch drawAll computes fraction 0 → the same ring drains 2→0. No cache, no event snapshot.
  • Why no events (repeat of the architecture note, here at the point of use): the spec's FEEDBACK-9 suggested consuming the TurnEnded event and caching the pre-reset progress because it assumed the ring is destroyed. With the reuse refactor (S3) the ring's own displayed fraction is the pre-reset value, so we tween from it to 0. The implementer does not need to wire dispatch's returned events for this to work.
  • Drain safety condition (correctness, do not skip): a world card whose ring is mid-tween must not be destroyed and recreated, or the displayed fraction is lost and the drain snaps. S3's reconcile only destroys containers for cards not in state.hand. A hazard still holding progress at turn-end stays in hand (reduce.ts:127), so its container is guaranteed to persist through the drain. The only world cards whose containers are destroyed mid-frame are resolved hazards (HazardResolved → left the hand); those exit immediately and never drain. Assert this invariant survives any later change to the reconcile.
  • Use the same duration/easing both directions so they read as one clock (FEEDBACK-8). Drain is core teaching (FEEDBACK-9 / brainstorm Q1/Q3), so it must be clearly perceptible, not a 1-frame snap.
  • dispatch (TableScene.ts:638) may optionally capture the returned events for finer control, but the unified tween-from-displayed approach does not require them. Keep it simple unless a feel problem shows up in manual testing.
Validation gate Manual: bank progress on a hazard → ring animates upward, not a snap. End a turn with a hazard holding partial progress → ring visibly drains to zero (animated), legible as "the work I banked just evaporated." Fill and drain feel like the same clock.

Review & delegation notes

  • Serialize TableScene.ts edits. S3, S6, S7, S9, S10 all touch it. The rollout retro showed concurrent edits work only when agents target different methods; the safe default here is sequential. If parallelizing, partition by method and never overlap match strings.
  • Fresh-eyes review after S3 (the refactor) and again at S12, per the testing-review rule — run by a fresh sub-agent, not the implementer. The S3 review's explicit scope: (1) every card id present before a cycle still has exactly one container after, none orphaned or duplicated; (2) pointer handlers are attached once and not re-bound per cycle; (3) killTweensOf is called before any container is destroyed; (4) no remaining code path destroys a card that is still in state.hand. "Tests pass" is necessary but not the review's job.
  • Tween hygiene everywhere: killTweensOf before re-tweening; never retain a Tween reference across cycles (rollout dead-end).
  • Pure logic stays in feedback.ts so it is testable headless and the structural boundary holds.
