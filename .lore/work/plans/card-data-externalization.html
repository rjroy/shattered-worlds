CARD-DATA Implementation plan — Card data externalization & world-scoped templates draft

Implementation plan: card-data-externalization

Date: 2026-06-04  |  Status: draft  |  Target: bun test green at every phase gate

Source artifact (validate against this) Spec: card-data-externalization.html — 17 requirements (REQ-CARD-DATA-1..17) plus a 10-point AI Validation checklist. Architectural constraints drawn from core-render-architecture (pure sync core, lint-enforced boundary), world-deck-slice (card model, mint invariant, error contract), and visual-identity (worldId in core, selectTheme fallback, constant card back).

Decisions & deviations

Resolved before drafting. Two items deviate from the spec's literal wording and need explicit sign-off at plan review.

Confirmed with user

  • JSON location: src/game/assets/worlds/starter.json and src/game/assets/worlds/zombie-big-box.json, co-located with existing art. Phaser loads them by hashed URL (Vite ?url import in the manifest); the headless sim and bun test import the parsed JSON module directly. Single source of truth, reconciles with the src/game/assets/<world>/ convention from REQ-VIS-17. The spec's literal path assets/worlds/*.json is not used.

Deviations from the spec's literal text (need review sign-off)

  • deviationcreateWorld / createGame take a WorldData run-setup argument, not just catalog. REQ-CARD-DATA-8 lists createWorld(catalog, seed), but neither the deck composition (REQ-CARD-DATA-6) nor the starter-deck multiset is part of CardCatalog — REQ-CARD-DATA-7 fixes CardCatalog = Record<string, CardTemplate> (templates only). The active worldId, its deckComposition, and the starter-deck composition must reach createWorld some other way. Proposed run-setup bundle: WorldData = { worldId: string; starterDeck: CardCount[]; deckComposition: DeckComposition } (the driver fills starterDeck from starter.json and the rest from the world JSON), giving createWorld(catalog, world, seed) and createGame(catalog, world, seed). mintCard(catalog, state, id) and dispatch/reduce(catalog, state, action) match the spec exactly.
  • deviationRenderer change is a seam, not a visible change, for player cards. REQ-CARD-DATA-13 requires the front renderer to key off card.sourceWorldId. The current asset model only defines a per-world world-card front (theme.worldCardfrontKey); player cards always use the generic cardfront, and per-world player art is explicitly Out of Scope. So we wire the sourceWorldId→selectTheme lookup and unit-test it, but a 'starter' player card keeps rendering the generic front (which is exactly what AI-Validation #9 asserts). No new art keys are added.
  • decisionavailableActions is NOT given a catalog parameter. Confirmed it never mints (src/core/available.ts has no mintCard/CATALOG reference). It stays a pure selector over state. The research note flagged this as an open question; resolved as "no".
  • decisionassembleCatalog lives in src/core/ (pure, no I/O). REQ-CARD-DATA-7 says "the game layer assembles" — satisfied because only the loading happens in the game layer. The pure merge+validation function is shared by all three drivers (Phaser game, headless sim, tests) so collision detection and template validation have one implementation. It imports no Phaser and no JSON files; callers pass already-parsed objects in. This respects the lint-enforced core boundary.

P1 Provenance & error scaffolding no dependencies

Add the data-model and error pieces that have no dependency on externalization. The module-level CATALOG stays in place; this phase is behavior-preserving except for the new provenance field and the now-explicit "unknown template" throw. Keeping it isolated lets the largest blast-radius work (2b) start from a green, provenance-complete base.

Files & changes

  • src/core/types.ts — add sourceWorldId: string to the PlayerCard interface (REQ-CARD-DATA-9). Change CardTemplateId from the 17-name union to type CardTemplateId = string (REQ-CARD-DATA-17); keep the alias for documentation. WorldCard is untouched (REQ-CARD-DATA-12 — no sourceWorldId).
  • src/core/errors.ts — add UnknownTemplateError (carries the bad templateId and the GameState, mirroring IllegalActionError's shape) for REQ-CARD-DATA-16, and CatalogError (collision / load / shape failures) for REQ-CARD-DATA-3 and -7. Used in later phases; defined here so the contract is settled early.
  • src/core/cards.ts — mintCard: when minting a player card, stamp sourceWorldId: state.worldId (REQ-CARD-DATA-9/11). Replace the silent CATALOG[templateId] undefined-deref with an explicit if (template === undefined) throw new UnknownTemplateError(...) (REQ-CARD-DATA-16). Signature still mintCard(state, templateId) in this phase.
  • src/core/world.ts — createWorld: build the skeleton with worldId: 'starter', mint the starter deck (now stamped 'starter'), then set worldId to the active world ('zombie-big-box' still hardcoded this phase) before minting acts (REQ-CARD-DATA-10). World cards carry no provenance, so act minting order is unaffected.
  • src/core/cards.test.ts, src/core/effects.test.ts — fix any PlayerCard literals / helper mint calls for the new field. Most build via mintCard and need no change.

P2a Author JSON + pure assembleCatalog (data duplicated) depends on P1

Move the card data into JSON and build the pure assembly path, while the TS CATALOG still exists. The phase's key test asserts the assembled-from-JSON catalog is deep-equal to the live CATALOG — proving JSON fidelity before anything depends on it. Nothing in the running game switches over yet.

New data files (REQ-CARD-DATA-1/4/5/6)

  • src/game/assets/worlds/starter.json — { "worldId": "starter", "cardTemplates": {...}, "starterDeck": [...] } for Sprint, Explore, Barricade, Med Kit, Panic, Adrenaline. No deckComposition (REQ-CARD-DATA-5), but it carries a starterDeck key — the [{templateId,count}] multiset currently in STARTER_SPEC (Sprint×2, Explore×3, Barricade×2, Med Kit, Panic, Adrenaline). This is the resolution to Q1: REQ-CARD-DATA-1 forbids that array in .ts, and starterDeck is a distinct key from deckComposition so REQ-CARD-DATA-5 still holds. Effect objects are copied verbatim from cards.ts so they deserialize into the exact Effect union shape. Authoring caution: omit optional keys entirely (e.g. a DealProgress with no bonus leaves the key out — never "bonus": null); readonly arrays accept the mutable arrays JSON produces. The P2a fidelity guard catches any shape drift.
  • src/game/assets/worlds/zombie-big-box.json — { "worldId": "zombie-big-box", "cardTemplates": {...}, "deckComposition": { "acts": [...] } }. cardTemplates covers the 7 world cards (Strange Sounds, Rubble, Screams, Zombie, Find Baseball Bat, The Walker, Door) and the world's reward-minted player cards (Listen, Baseball Bat, Regroup, Summon Door). deckComposition.acts reproduces ACT_1/2/3_SPEC exactly as { templateId, count } pairs.

New core module (REQ-CARD-DATA-7)

  • src/core/catalog.ts — schema types and the pure assembler. No Phaser, no JSON imports, no I/O: 
    export type CardCatalog = Record<string, CardTemplate>
export type CardCount = { templateId: string; count: number }
export interface DeckComposition { acts: { cards: CardCount[] }[] }
// Run-setup bundle for createWorld: active world + starter multiset.
export interface WorldData {
  worldId: string
  starterDeck: CardCount[]
  deckComposition: DeckComposition
}
export interface RawCardSource {
  worldId: string
  cardTemplates: Record<string, CardTemplate>
  starterDeck?: CardCount[]        // present only in starter.json
  deckComposition?: DeckComposition // present only in world JSON
}
// Merge sources into one catalog; throw CatalogError on duplicate templateId.
export function assembleCatalog(sources: RawCardSource[]): CardCatalog
    Collisions between starter and world templates throw at assembly time, never silently overwrite (REQ-CARD-DATA-7). Not implemented: per-field JSON schema validation is deferred per the spec's Out-of-Scope list — assembleCatalog checks ids for collision but does not validate every template field. The JSON imports are typed with an explicit as RawCardSource cast (avoids resolveJsonModule literal-narrowing surprises).

Driver-shared data accessor

  • src/game/worldData.ts — imports the two JSON modules directly (for sim + tests) and exposes typed parsed objects, e.g. STARTER_SOURCE and ZOMBIE_SOURCE typed as RawCardSource. The Phaser scene will not use this in P3 (it loads via this.load.json per REQ-CARD-DATA-2); this module serves the non-Phaser drivers. Importing JSON here, not in core, keeps the core boundary clean.
  • tsconfig.json — add "resolveJsonModule": true (currently absent) so TS type-checks the JSON imports under bun test.

Tests added (REQ-CARD-DATA-7)

  • src/core/catalog.test.ts — assembling [starter, zombie] yields all template ids; injecting a duplicate id throws CatalogError.
  • Fidelity guard: assembleCatalog([STARTER_SOURCE, ZOMBIE_SOURCE]) deep-equals the still-present CATALOG from cards.ts. This is the safety net for the 2b cutover; it is deleted in 2b along with CATALOG.

P2b Cutover: thread explicit catalog, delete the global & the TS data depends on P2a · widest blast radius

The atomic cutover. Every minting path now receives catalog explicitly; createWorld/createGame also receive WorldData; the module-level CATALOG and all inline card/act data are deleted. All call sites and tests are updated in this same phase so the build stays green. This is the phase to treat as unrecoverable-if-half-done — review the signature changes against the spec before starting.

Core signature changes (REQ-CARD-DATA-8)

  • src/core/cards.ts — mintCard(catalog: CardCatalog, state, templateId): [PlayerCard|WorldCard, GameState]. Look up catalog[templateId]; throw UnknownTemplateError if absent. Delete PLAYER_TEMPLATES, WORLD_TEMPLATES, and CATALOG (REQ-CARD-DATA-1/8). Keep the CardTemplate/PlayerCardTemplate/WorldCardTemplate types (now imported by catalog.ts).
  • src/core/world.ts — createWorld(catalog, world: WorldData, seed). Delete ACT_1/2/3_SPEC, STARTER_SPEC, and the local ActSpec/StarterSpec types. Mint the starter deck by expanding world.starterDeck (each {templateId,count}) while worldId === 'starter'; then set active worldId from world.worldId; then build acts by expanding world.deckComposition.acts. This preserves the P1 ordering (starter cards stamped 'starter', acts minted under the active world).
  • src/core/effects.ts — thread catalog through applyEffect(catalog, state, effect, action), applyPenalty(catalog, state, penalty), grantReward(catalog, state, reward), and gainCard(catalog, state, template, dest). The single mintCard call (line 90) passes catalog through.
  • src/core/reduce.ts — reduce(catalog, state, action); the three handlers thread catalog into applyEffect/applyPenalty. availableActions calls stay catalog-free.
  • src/core/game.ts — createGame(catalog, world, seed); createWorld(catalog, world, seed) at init; the closure's dispatch(action) calls reduce(catalog, current, action). GameCore interface unchanged (still dispatch(action) outward — catalog is captured).
  • src/core/contract.ts — export the new public types (CardCatalog, CardCount, DeckComposition, WorldData) and assembleCatalog so drivers consume them through the contract boundary. The existing CardTemplateId export stays (now resolving to the string alias) — game-layer imports of it keep working.
  • src/game/describe.ts — no change required, verify only. describeEffect/Penalty/Reward are pure switches on effect.kind / penalty.kind / reward.kind and operate on plain data, so deserialized-from-JSON effect objects read identically to the old inline ones. Per the retro, do not add a JSON-keyed description path; the assembled CardTemplate.effect remains the single source.

P3 Phaser preload + scene-init wiring (load JSON, surface failures) depends on P2b

Wire the Phaser scene to load both JSON files in preload() and assemble the catalog in create() before any game state exists. This resolves the preload()-can't-read-state.worldId constraint the visual-identity rollout retro flagged: worldId arrives as scene init data. This is the trickiest wiring point in the plan — only this scene file changes, and edits stay within distinct methods.

Files & changes (REQ-CARD-DATA-2/3)

  • src/game/assetManifest.ts — add two ?url JSON imports and entries, e.g. 'world-starter' → starter.json?url, 'world-zombie-big-box' → zombie-big-box.json?url. (Manifest already declares /// <reference types="vite/client" />.)
  • Scene init data needs a Boot scene (resolves Q3). The current main.ts uses scene: [TableScene] (constructor reference, auto-started, no key, no data path). Phaser cannot inject init data into an auto-started array entry. Concrete fix:
    • src/game/TableScene.ts — give the scene a key: super({ key: 'Table' }) in its constructor.
    • src/game/BootScene.ts (new, ~10 lines) — super({ key: 'Boot' }); its create() calls this.scene.start('Table', { worldId: 'zombie-big-box', seed: 42 }). This is where the run's world + seed are chosen.
    • src/game/main.ts — scene: [BootScene, TableScene]. Boot runs first and is the only auto-started scene; it hands off to Table with init data. The starter file is always loaded regardless of worldId; worldId selects the active world JSON.
  • src/game/TableScene.ts:
    • init(data) — store this.worldId_ and this.seed_ from init data (default to 'zombie-big-box'/42).
    • preload() — replace the hardcoded selectTheme('zombie-big-box') with selectTheme(this.worldId_) for art keys; add this.load.json('world-starter', manifest['world-starter']) and this.load.json('world-' + this.worldId_, manifest['world-' + this.worldId_]). Register this.load.on('loaderror', ...) to record a failure flag (REQ-CARD-DATA-3).
    • create() — before building state: if the loaderror flag is set, or this.cache.json.get(...) returns undefined for either required key, log via console.error and throw CatalogError before any state exists. The console.error is what satisfies REQ-CARD-DATA-3's "error message visible to the developer" — no user-facing error UI is required at this stage. Otherwise read both JSON objects, assembleCatalog([starter, world]), build the WorldData run-setup (worldId + deckComposition from the world JSON, starterDeck from starter.json), then createGame(catalog, world, this.seed_). Keep selectTheme(this.game_.state.worldId) for the active theme.

Resolved during plan review

The fresh-eyes review surfaced three would-be open questions; all are now resolved inline in the phases above. Recorded here so the rationale is recoverable:

  • Starter deck composition (was Q1, was a P2b blocker). Resolved: a starterDeck: [{templateId,count}] key in starter.json (distinct from deckComposition, so REQ-CARD-DATA-5 holds), carried into createWorld via the WorldData run-setup. This is where STARTER_SPEC goes when it leaves .ts (REQ-CARD-DATA-1). See P2a / P2b.
  • WorldData signature (was Q2). Accepted as the resolution of the "CardCatalog can't carry composition" gap — see Decisions & deviations. The alternative (a richer catalog object) contradicts REQ-CARD-DATA-7's fixed Record<string, CardTemplate> type, so it is rejected.
  • Scene-start mechanics (was Q3, was a P3 blocker). Resolved: a one-purpose BootScene injects {worldId, seed} init data into a keyed TableScene; main.ts becomes scene: [BootScene, TableScene]. The bare game.scene.start alternative does not work against an auto-started array entry. See P3.

Remaining for the implementer: none blocking. Confirm resolveJsonModule is acceptable in tsconfig.json (the JSON imports use an explicit as RawCardSource cast regardless), and that adding a BootScene file is acceptable (per the "ask before creating new files" rule — this plan proposes it, the new files are: src/core/catalog.ts, src/game/worldData.ts, src/game/BootScene.ts, two JSON assets, and new *.test.ts files).
