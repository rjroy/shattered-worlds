Implementation plan: Player resource — Energy

DRAFT source: specs/player-energy-resource.html

Implements REQ-ENERGY-1 … 16: a per-turn energy field on GameState, an authored per-card energy cost, the turn-start gain (+1, uncapped carry-over), the affordability gate, deduction on play, a loss-guard correction, an EnergyChanged event, and the HUD + card-badge presentation.

Build order rationale. The model fields (Step 1) are the foundation everything else reads. Data (Step 2) and the energy lifecycle module (Step 3) can land in parallel after that. The reducer wiring (Steps 4–6) depends on both. View (Steps 7–8) depends only on the model fields. Tests are written alongside each step, with a consolidated validation gate at the end.

Step sequence

1 model fields→ 2 card data 3 energy lifecycle→ 4 affordability gate→ 5 deduct on play→ 6 loss guard A→ 7 HUD 8 card badge→ 9 ✅ validate

Steps 2 and 3 depend only on Step 1 and on each other not at all. Steps 7–8 depend only on Step 1 and can proceed in parallel with 3–6. Step 9 is the validation gate over the whole spec.

Key design decisions (resolve before implementing)

D1 — A single turn-start entry point, in a new energy.ts module

REQ-ENERGY-4/5 mandate one shared gain step invoked by both createWorld and the EndTurn handler, with the +1 applied before refillHand. To avoid two call sites independently sequencing "gain then refill" (a classic divergence bug), this plan adds a new file src/core/engine/energy.ts exporting:

  • gainEnergy(state) — returns state with energy + 1 plus an EnergyChanged event. The single increment site.
  • spendEnergy(state, cost) — returns state with energy - cost plus an EnergyChanged event when cost > 0 (no event for cost 0). The single decrement site.
  • startTurn(state) — composes gainEnergy then refillHand, returning the combined state and events. This is the one place the turn-start ordering lives; both createWorld and handleEndTurn call it.

New-file decision (needs approval per "ask first"). Alternative: put these three functions in the existing draw.ts alongside refillHand. Recommendation: a dedicated energy.ts keeps energy arithmetic cohesive and keeps draw.ts draw-only; startTurn imports refillHand from draw.ts (no circular dependency, since draw.ts never imports energy).

D2 — Affordability is a card-level filter; an ignoreEnergy option feeds loss guard A

The affordability gate (REQ-8/9) is a single energyCost > state.energy check in the availableActions loop, layered on top of the existing structural playableSpec result. Loss guard A (REQ-13) must distinguish "no structural play exists" (a real loss) from "the only play is unaffordable right now but energy will rise" (not a loss). Rather than duplicate the structural logic, availableActions gains an optional { ignoreEnergy?: boolean } argument; guard A calls availableActions(afterRefill, { ignoreEnergy: true }) to ask "is progress possible in any future energy state". Spec knowledge stays in one home (available.ts).

D3 — REQ-11 (reducer rejects unaffordable plays) needs no new reducer code

Because unaffordable cards are excluded from playable (Step 4), the existing checkPlayAction already returns "Card … is not playable" for them, and handlePlayCard already throws IllegalActionError on a non-null check. Step 5 only adds the deduction for legal plays. This is recorded so the implementer does not add a redundant second guard.

Critical anchor: the mechanism is the availableActions(state) call at the top of handlePlayCard — it must stay without the ignoreEnergy argument added in Step 4. Do not propagate ignoreEnergy: true into the reducer as a symmetry reflex; doing so would let an unaffordable play through and break REQ-11. Only loss guard A (Step 6) passes ignoreEnergy: true.

STEP 3Energy lifecycle: energy.ts (gain/spend/startTurn) + world init + EndTurn wiringcore-engine

REQ-ENERGY-4, 5, 6, 7, 14

Depends on Step 1. See decision D1.

New: src/core/engine/energy.ts. Edits: src/core/engine/world.ts, src/core/engine/reduce.ts

  • New energy.ts:
    • All three return the project's { state, events } pair (same shape as refillHand's EffectResult), so call sites compose them uniformly.
    • gainEnergy(state) → { state: {...state, energy: state.energy + 1}, events: [{ type: 'EnergyChanged', energy }] }.
    • spendEnergy(state, cost) → energy - cost; emit EnergyChanged only when cost > 0.
    • startTurn(state) → run gainEnergy, then refillHand on the gained state; concatenate events (EnergyChanged first, then draw events). This guarantees REQ-5 ordering (+1 before refill) in one place.
  • world.ts createWorld: initialize energy: 0 in the bootstrap skeleton state (REQ-4, REQ-7 — fresh world resets to 0). Replace the final refillHand(baseState) call with startTurn(baseState) so the opening hand is a turn start: returned state has energy === 1. Events are discarded at init as today.
  • reduce.ts handleEndTurn: replace the refillHand(stateAfterDiscard) call with startTurn(stateAfterDiscard), pushing its events (EnergyChanged + draw events) into the event list. Placement: after the onEndOfTurn loop and player-card discard, before the loss guards — so the +1 is unconditional (REQ-5: not suppressed by skipDrawNext, which refillHand still consumes internally) and carry-over is automatic (REQ-6: nothing ever decrements energy at end of turn). The ActAdvanced path lives inside drawWorld and never touches energy, satisfying REQ-7 by construction.
Watch: the onEndOfTurn early-return (when a world card's end-of-turn effect ends the game) must stay before startTurn — a lost/won turn does not start a new turn, so no energy is gained on a turn that kills the player.

Tests (alongside): in world.test.ts assert createWorld(...).energy === 1. In reduce.test.ts: EndTurn with no plays gains exactly +1; several EndTurns are monotonic +1 with no cap; skipDrawNext: true still gains +1; energy is identical across an ActAdvanced boundary; an EnergyChanged event with the new total appears on a turn-start gain.

STEP 4Affordability gate: exclude unaffordable cards from playablecore-engine

REQ-ENERGY-8, 9, 11 (via D3), 12

Depends on Steps 1 and 3 (state carries energy). See decision D2.

File: src/core/engine/available.ts

  • Add an optional second arg to availableActions(state, opts?: { ignoreEnergy?: boolean }).
  • In the playable loop, after playableSpec returns non-null, skip the card when opts?.ignoreEnergy !== true && card.energyCost > state.energy. Cost-0 cards are always affordable (REQ-8). This is the single affordability site.
  • REQ-12: the cost is per-card and is read from card.energyCost on the PlayerCard instance — it does not vary by modal branch or sequence step, because the gate compares the whole card's cost. No branch-specific logic.
Discardable, canEndTurn, and legalTargets are unaffected — energy gates playing a player card only. World-card discard (REQ has no energy cost) stays as-is.

Tests (alongside): in available.test.ts — a cost-1 card (Listen) with otherwise-met structural requirements is absent from availableActions(state).playable when energy === 0, and present when energy === 1. With ignoreEnergy: true it is present even at energy === 0 (the seam guard A relies on).

STEP 5Deduct energy on a successful playcore-engine

REQ-ENERGY-10, 11, 14

Depends on Steps 1, 3 (spendEnergy), 4 (gate already rejects unaffordable plays).

File: src/core/engine/reduce.ts (handlePlayCard)

  • After the card is confirmed a player card and removed from hand, call spendEnergy(stateAfterPlay, card.energyCost) and thread the resulting state into applyEffect. Push the EnergyChanged event (when cost > 0) into the event list — order it right after CardPlayed and before the effect's events, so the deduction reads as part of resolving the play.
  • REQ-10/11: energy never goes negative because the gate (Step 4) guarantees energy >= cost before this point; spendEnergy performs a plain subtraction. Cost-0 leaves energy unchanged and emits no event.
Sequencing note: deduction happens on stateAfterPlay (card already moved to discard) and the spent state is what applyEffect receives — so any effect that reads energy mid-resolution sees the post-spend value. None of the three shipping costed cards read energy, but this keeps the model coherent for future cards.

Tests (alongside): in reduce.test.ts — energy 1, play cost-1 card legally → state.energy === 0 and an EnergyChanged event present; energy never reads negative; play cost-0 card → energy unchanged, no EnergyChanged. REQ-11: energy 0, dispatch PlayCard for a cost-1 card directly → throws IllegalActionError, state unchanged.

STEP 6Loss guard A: do not declare loss when only blocker is current energycore-engine

REQ-ENERGY-13

Depends on Step 4 (the ignoreEnergy seam).

File: src/core/engine/reduce.ts (handleEndTurn, guard A)

  • Variable provenance: after Step 3 replaces refillHand(stateAfterDiscard) with startTurn(stateAfterDiscard), the local currently named afterRefill holds the startTurn result (energy already gained + hand refilled). Rename it (e.g. afterTurnStart) or annotate it so the two handleEndTurn edits (Steps 3 and 6) don't read as touching a stale name.
  • Guard A currently computes noProgressPossible from availableActions(afterRefill). Change it to use availableActions(afterTurnStart, { ignoreEnergy: true }) so a card that is unaffordable only because energy is currently too low counts as future progress and prevents a false WorldLost. Energy regenerates +1/turn uncapped, so such a card becomes affordable in finite turns.
  • Guard B is unchanged (REQ-13 records this): it tests for the existence of an AddWorldCardToTop card across zones, which energy gating never affects. Leave it exactly as-is; add a one-line code comment pointing at REQ-13 so a future reader does not "fix" it.

Tests (alongside): in reduce.test.ts — construct an end-of-turn state where the only progress-making card is a costed card currently unaffordable, with energy that will rise next turn, and assert status stays playing / no WorldLost. Keep a companion test proving a genuinely dead state (no structural play at all) still loses, so the guard isn't neutered.
