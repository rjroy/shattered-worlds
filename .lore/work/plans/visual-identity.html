Plan · Building The Look

Theme the world, bleed the intrusion, walk the Walker

A dependency-ordered build of the visual-identity spec: one pure-data core seam, two headless-tested pure mappers, a renderer-side theme registry, themed fronts over a constant face-down back, an intensity-driven intrusion backdrop, and the Walker who closes in by act and steps to the foreground when drawn.

Source of truth: specs/visual-identity.html (approved, REQ-VIS-1..18) · respects the core/render boundary · builds on the world-deck slice

▦ The build order

core one pure-data change, lint-boundary safe pure no Phaser — unit-tested headlessly (REQ-VIS-15) phaser renderer glue (src/game/) art-gen uses the art-gen skill

S2 (core), S3 and S4 (pure) have no prerequisites and can land in parallel. The Phaser steps (S5–S9) chain through the preload. S10 checks everything against the spec.

Plan-level decisions (veto any of these)

  • Card back gets a home: the player and world draw piles render as small face-down stacks of cardback.webp beside their existing HUD counts. That is where the constant back lives (REQ-VIS-5). (Confirmed with user.)
  • Theme art is generated from the big-box panel of art-style.webp via the art-gen skill: a reality backdrop, an intrusion overlay, and a transparent Walker silhouette. (Confirmed with user.)
  • VisualTheme holds asset keys, not imported URLs. The pure theme/mapper modules must stay importable under bun test with no bundler, so the key→URL mapping (Vite import of .webp) lives only in the Phaser preload glue. This is what keeps REQ-VIS-15 genuinely headless. (Assumed — load-bearing for testability.)
  • worldId is a required field on GameState, not optional. Required is the typed-discipline choice; it means updating the few full-literal state constructions (the createWorld skeleton; test fixtures fork createWorld so are unaffected). (Assumed.)
  • Intrusion is one overlay, alpha-driven. Rather than two full backdrops, a reality image sits behind an intrusion overlay whose alpha = intrusionForIntensity(intensity), tweened. The overlay is composed (edge/radial weighting) so the board area keeps a mundane boundary (REQ-VIS-10). (Assumed — simplest faithful realization.)
  • Backdrop/Walker/pile rendering is extracted into new modules, not piled into TableScene.ts (already ~650 lines, near the 800 heuristic). (Assumed.)
  • Presence is state-derived each repaint. The renderer already fully repaints on every dispatch (drawAll); backdrop, Walker, and piles read state + intensity() there. Tweens carry the motion; we do not add an event-timeline system (out of scope). (Assumed.)

S3The VisualTheme registry (pure, headless-tested)· no deps pure

The renderer-side theme abstraction and its one concrete theme (REQ-VIS-4). Plain data, no Phaser, so its test runs headless.

new src/game/theme.ts src/game/theme.test.ts
export interface VisualTheme {
  worldId: string
  intrusionHue: string            // '#9bff6a' for big-box (fluorescent green)
  realityPalette: string[]        // desaturated slate base
  frameStyle: FrameStyle         // front frame/chrome colors + stroke
  grain: GrainSpec               // distress params (numbers)
  backdrop: { realityKey: string; intrusionKey: string }
  walker?: { textureKey: string }   // present iff the world has a Walker
}
export function selectTheme(worldId: string): VisualTheme  // unknown -> zombie-big-box
  • Define the theme + sub-types as plain data. Asset references are string keys (realityKey, intrusionKey, textureKey), never imported URLs.
  • Build the zombie-big-box theme: green intrusion hue, slate reality palette, front colors chosen to clear WCAG AA against the themed card background (REQ-VIS-7).
  • selectTheme is total: an unknown worldId returns the zombie-big-box theme.
  • GrainSpec is a static per-theme constant for now — define and populate it, but do not wire grain to intensity. Driving grain/shake from the signal is REQ-VIS-14 (a Should), deferred (see Not in this plan).
  • No Phaser import in this file so theme.test.ts loads with no bundler/DOM.
Gate
  • theme.test.ts green under bun test, no Phaser/DOM: selectTheme('zombie-big-box') returns the theme; an unknown id returns the same default. REQ-VIS-4, 15

S8Backdrop & intensity-driven intrusion· needs S3, S4, S5 phaser

The reality base that slides toward intrusion as the run escalates (REQ-VIS-8, 9, 10).

new src/game/backdrop.ts · edit src/game/TableScene.ts (create layer in create, drive in drawAll)
  • A persistent BackdropLayer created in create() (same init rule as S7: after createGame(42), before the win/loss bringToTop, never in the cardObjects destroy loop): the reality image at the lowest depth (setDepth(-10), behind cards/HUD/overlays), plus the intrusion overlay above it. Neither is setInteractive, so pointer hits still reach cards/buttons.
  • Read intensity through the public seam — this.game_.intensity() — never import the core intensity module directly (that would breach the renderer's read-via-GameCore discipline).
  • Each drawAll: compute target alpha = intrusionForIntensity(this.game_.intensity()) and retarget the overlay's tween fire-and-forget. In Phaser 3.90 that is tween.updateTo('alpha', target, true) on a single long-lived tween (or stop-and-restart) — not this.tweens.add(...) per repaint, which queues a second tween and lets the backdrop lag the authoritative state.
  • Compose the overlay (edge/radial weighting, or a masked-clear board zone) so a mundane boundary stays visible even at max intensity — the apocalypse stays localized.
Gate
  • Backdrop is mundane at fresh Act 1 / full HP; visibly more intruded (green) after advancing acts and losing HP; transition tweens, not snaps.
  • The board area keeps a normal/intrusion boundary at high intensity; clicks on cards/buttons still register. REQ-VIS-8, 9, 10
S9The Walker presence· needs S4, S5, S8 phaser

Distant figure who nears by act and foregrounds when drawn — presentation of the existing Walker card, no new rule (REQ-VIS-11, 12, 13).

new src/game/walker.ts (pure proximity/foreground calculator) · edit src/game/backdrop.ts (owns the Walker sprite, in the backdrop layer)
  • A persistent Walker sprite (from walker texture), shown only when the active theme declares a walker. Position/scale from walkerProximityForAct(state.actIndex); tween to the new tier when actIndex changes (compare last-seen actIndex in the layer), so crossing into an act reads as him closing in.
  • Foreground on draw: when state.hand contains a card with name === 'The Walker', switch to a large foreground presentation tied to that card; hold keeps him there; once he leaves the hand, return to the act-tier position. The act-advance tween is suppressed while foregrounded.
  • Resolve and discard share the same return — the mechanical difference is carried by the card, not a distinct animation.
  • All of it reads state (hand membership, actIndex) in drawAll; no renderer-side timer that could desync from the deck. A plain state.hand.some(c => c.name === 'The Walker') check each drawAll is sufficient for enter/exit detection — no need to parse dispatch events for this (both exits share one visual return anyway).
Gate
  • Distant silhouette in Act 1, looming by Act 3, tweening on ActAdvanced; foregrounds when the Walker card is forced into hand and returns to the backdrop after it leaves — all state-derived. REQ-VIS-11, 12, 13

✕ Not in this plan

Inherited from the spec's non-goals: additional worlds/themes beyond zombie-big-box; an asset-authoring pipeline; per-card bespoke art; audio; an event→timeline animation system (we drive from state + tweens); and any change to rules, the card catalog, or the turn loop. Tuning the intensity curve itself stays in the core and is out of scope here.

REQ-VIS-14 (escalation juice) is deferred — a Should, not a Must. The seam exists (the backdrop already reads intensity), but no secondary effects (grain density, particles, shake) are wired in this slice. GrainSpec is defined as a static constant (S3) so the theme shape is complete; driving it from the signal is a later pass.
