Plan-level decisions (spec gaps pinned here)

Each of these resolves something the spec leaves open. Veto here, not during implementation.

D1 — Record-callout mechanism. Bests fold inside the collector, and a tie is indistinguishable from a fresh record by reading the persisted shape alone. The collector therefore remembers which bests the most recent fold set or beat, exposed as a transient RunStatsReader.lastRunRecords() (never persisted; empty after reload, which is fine — the summary is only shown immediately after the fold). Views stay read-only per REQ-STATS-10.
D2 — Migrated lastRun.activeDurationMs. REQ-STATS-4 adds activeDurationMs to RunRecord; a migrated v1 lastRun doesn't have one. Migration backfills it with the clamped wall-clock duration (max(0, endedAt − startedAt)) — same "accepted historical noise" stance REQ-STATS-3 already takes for pre-v2 counters, and it keeps the field required (no optionality sprawl through every view).
D3 — durationMs keeps its name. REQ-STATS-3 says it carries over as-is; REQ-STATS-4 says lifetime play time accumulates activeDurationMs for completed runs from v2 on. Same counter, new feed — renaming would buy clarity at the cost of a field mapping in migration and every consumer. A doc comment marks the semantic seam instead.
D4 — Peek-card clause is vacuous today. REQ-STATS-13 hedges for partial-opacity peek cards; the shipped carousel renders 3 cards at full opacity. Badges render on all visible cards. If a peek treatment ships later, the "omit on peek" default applies then.
D5 — Import writes go through a composition-root transfer seam. Import is the one designed write path from a view (REQ-STATS-19). A StatsTransfer object created inside createGameplayRuntime wraps the collector and storage; ChronicleScene receives it injected, exactly as TableScene receives the runtime. Scenes still never touch fold/persist functions or the collector directly.
D6 — One collector, two keys, isolated loads. The history ring buffer shares the collector (the RunRecord is finalized once — duplicating tally + active-time logic in a second subscriber would be a bug farm) but lives in its own module (runHistory.ts) with its own key, validator, and load — a corrupt history payload warns and yields an empty buffer while lifetime loads normally, satisfying the REQ-STATS-7 isolation mechanism.
D7 — Visibility seam shape. Per REQ-STATS-4's wording, two options on GameplayRuntimeOptions, parallel to clock: visibility?: () => boolean (true = visible) and subscribeVisibility?: (onChange: () => void) => () => void. Defaults: always-visible / no-op (tests, headless). main.ts wires the document visibility API.

S1 v2 schema, migration, bests REQ-STATS-1, 2, 3, 5, 6 · REQ-STATS-21 (partial)

Depends on: nothing. Foundation for everything else.

Files: src/game/runtime/runStats.ts, src/game/runtime/runStats.test.ts, src/game/runtime/gameplayRuntime.ts (signature touch).

Changes
  • Types. LifetimeStats.version becomes 2. WorldStats gains optional fewestTurnsWin?: number and mostProgressInRun?: number (absent until earned). RunRecord gains required activeDurationMs: number (S2 feeds it real values; S1 wires the field through using clamped wall-clock so S1 is shippable alone).
  • Keys. RUN_STATS_STORAGE_KEY = 'shattered-worlds/run-stats/v2'; keep the old value exported as RUN_STATS_LEGACY_V1_KEY for the migration read and for test fixtures.
  • Storage seam. RunStatsStorage gains removeItem(key: string): void. main.ts's window.localStorage already conforms; update test memory storages.
  • Validation. isLifetimeStatsV2 — as strict as v1's (every counter finite, optional bests finite-when-present, lastRun checks activeDurationMs); retain the v1 validator (renamed isLifetimeStatsV1) for the migration path. Wholesale reject, never partial-fold (REQ-STATS-6).
  • Migration. migrateLifetimeV1toV2(v1): LifetimeStats — all counters and byWorld tallies carry over, bests absent, durationMs as-is, lastRun carried with activeDurationMs backfill (D2).
  • Load order. v2 key validates-as-v2 → use. Else v1 key validates-as-v1 → migrate, remember the v1 key needs removal. Else (malformed / unknown version under either key) → console.warn + fresh, unchanged posture. After the first successful v2 setItem, removeItem(v1 key) exactly once (REQ-STATS-3).
  • Fold v2 (REQ-STATS-2): outcome tallies (runs/wins/losses/abandoned, lifetime + byWorld) and lastRun update for every outcome; activity counters (turns, cardsPlayed, progressDealt, damageTaken, hazardsResolved, hazardsDiscarded, cardsDiscarded) and durationMs fold only for won|lost. The exclusion is a lifetime-fold gate, not a zeroing: the abandoned run's RunRecord itself still carries its fully-tallied activity counters — that's what "lastRun and history record abandoned runs in full" (REQ-STATS-2) means, and the abandon summary (S5) displays those values. Bests: fewestTurnsWin on won runs (strictly lower or first), mostProgressInRun on completed runs (strictly higher or first); abandons never set bests. Fold returns { lifetime, newRecords }; collector stores newRecords transiently; RunStatsReader gains lastRunRecords() (D1).
  • The close-exactly-once / partial-run-ignored behaviors are untouched (REQ-STATS-5) — assert via existing tests still passing plus an explicit regression test.
  • Verbatim v1 payload fixture (captured from the shipped v1 shape, stored under the v1 key) migrates field-by-field: all counters equal, bests absent, lastRun carried (REQ-STATS-3 proof).
  • v1 key removed after first persist; not removed before any persist.
  • Malformed v2, malformed v1, and unknown-version payloads each discarded with a warning.
  • Abandon fold exclusion: outcome counts increment, activity counters and durationMs do not, bests unset, lastRun + history still record.
  • Bests: first-set, beat, not-beaten, and tie-does-not-reset transitions for both record types; lastRunRecords() true only on set/beat.
  • bun test green including all pre-existing runStats tests (updated to v2 expectations).

S2 Active-time duration clock REQ-STATS-4 · REQ-STATS-21 (partial)

Depends on: S1 (the activeDurationMs field).

Files: src/game/runtime/runStats.ts, runStats.test.ts (every existing createRunStatsCollector(…) call site moves to the options object — the signature change lands with all callers updated, or S2's own gate fails), gameplayRuntime.ts, gameplayRuntime.test.ts, src/game/main.ts.

Changes
  • GameplayRuntimeOptions gains the two visibility seams (D7). createGameplayRuntime passes clock, visibility, and subscribeVisibility down to the collector.
  • createRunStatsCollector(storage?, key?) becomes createRunStatsCollector(options: RunStatsCollectorOptions) — { storage?, lifetimeKey?, historyKey?, clock?, visibility?, subscribeVisibility? }. Internal API with two call sites (runtime + tests); no compatibility shim.
  • Accumulator tracks activeMs + visibleSince: number | null. RunStarted: open a segment at the item timestamp if visible. Visibility → hidden: close the segment with activeMs += max(0, clock() − visibleSince) (per-segment clamp preserves the backwards-clock guarantee). Visibility → visible: reopen at clock(). RunEnded: close any open segment at the item timestamp; activeDurationMs = activeMs.
  • main.ts: visibility: () => document.visibilityState === 'visible', subscribeVisibility via a document.addEventListener('visibilitychange', …) wrapper. Tests never touch real page visibility.
  • Fake visibility transitions: hidden span excluded from activeDurationMs; fully-visible run equals wall-clock; run started while hidden accrues nothing until visible.
  • Backwards clock jump mid-run never subtracts time (segment clamp held).
  • Lifetime durationMs accumulates activeDurationMs for completed runs only; abandons add nothing.
  • Default seams (no visibility injected) behave as always-visible — existing tests unaffected.

S3 Run history ring buffer REQ-STATS-7 · REQ-STATS-21 (partial)

Depends on: S1. Parallel with S2 in principle, but both touch runStats.ts (S2 reshapes the collector options, S3 wires history into the collector) — if run concurrently, coordinate on disjoint regions or sequence S2 → S3.

Files: src/game/runtime/runHistory.ts (new), src/game/runtime/runHistory.test.ts (new), runStats.ts (append wiring).

Changes
  • RUN_HISTORY_STORAGE_KEY = 'shattered-worlds/run-history/v1'; payload shape { version: 1, records: RunRecord[] }, most-recent-first.
  • runHistory.ts exports loadHistory (validate-or-discard with warning, same posture as lifetime), appendRun (prepend, evict beyond 100), persistHistory (non-fatal failure). Pure, Phaser-free.
  • Collector (D6): loads history at construction independently of lifetime — a throw/invalid in one never affects the other; on RunEnded appends the finalized RunRecord for every outcome (abandons included, in full) and persists under the history key. Partial runs (attached mid-run) are not appended.
  • Collector exposes history(): readonly RunRecord[] for the S4 transfer seam (not added to the view-facing RunStatsReader — no UI consumes history per spec).
  • Cap at exactly 100: 101st run evicts the oldest; order most-recent-first across persist/load.
  • Corrupted history payload → warning + empty buffer, lifetime loads normally; corrupted lifetime → history loads normally (both directions tested).
  • Abandoned run appears in history with full counters while lifetime activity counters stay flat.

S4 Export/import transfer seam REQ-STATS-18, 19 (logic half) · REQ-STATS-21 (partial)

Depends on: S1, S3.

Files: src/game/runtime/statsTransfer.ts (new), statsTransfer.test.ts (new), runStats.ts (replaceAll), gameplayRuntime.ts (expose statsTransfer).

Changes
  • Export payload: { kind: 'shattered-worlds-stats', exportedAt, lifetime, history } — exactly what import accepts (REQ-STATS-18). exportJson(): string serializes current in-memory state.
  • Two-phase import so migration happens only after confirmation (REQ-STATS-19): inspectImport(json: string) → { ok: true, needsMigration: boolean, payload } or { ok: false, reason } — validates with full storage-load strictness (v2 lifetime, or v1 lifetime flagged needsMigration; history validated or absent-tolerated for older exports), mutates nothing. applyImport(inspected) → migrate if needed, then wholesale replace: collector replaceAll(lifetime, history) updates in-memory state and persists both keys.
  • createGameplayRuntime builds the transfer over its collector + storage and exposes it as runtime.statsTransfer for injection into ChronicleScene (D5).
  • Round-trip: export → wipe storage → inspect + apply → lifetime and history byte-equal.
  • Invalid file (malformed JSON, wrong kind, garbled counters): ok: false with reason; stored data and in-memory state untouched.
  • v1-lifetime payload: needsMigration: true from inspect with storage untouched; migration applied only inside applyImport.

S5 Run summary overlay + TableScene wiring REQ-STATS-8, 9, 10, 11 · closes REQ-WS deferred item

Files: src/game/view/RunSummaryView.ts (new), src/game/view/format.ts (new, pure), src/game/scenes/TableScene.ts, src/game/view/EndScreenView.ts (deleted), tests for each.

Changes
  • format.ts: formatDuration(ms): string (human-readable, e.g. "12m 34s" / "1h 02m"; no raw milliseconds) — shared by summary and Chronicle. Pure, unit-tested.
  • RunSummaryView follows the established overlay conventions (REQ-STATS-11 / help-screen spec): container at (CANVAS_W/2, CANVAS_H/2), depth 1000, hidden by default, full-canvas bg.setInteractive() swallowing board input via topOnly. API: show(data) where data carries outcome, world display name, the REQ-STATS-8 stat list (active duration via formatDuration), the lifetime context line ("Run N" = lifetime().runs post-fold, so first run reads "Run 1"; "+ M wins on this world" when ≥ 1), and record callouts from lastRunRecords(); plus a continue affordance. The dismiss callback is passed as show(data, onDismiss) (per-show, since the same view serves three outcomes).
  • TableScene: replace winScreen/lossScreen with one RunSummaryView (delete EndScreenView.ts — TableScene is its only consumer). In drawAll() when status becomes won/lost: populate from this.runtime_.runStats (the reader — viable because RunEnded is emitted synchronously inside dispatch, so lastRun is current; REQ-STATS-10) and show. While the summary is up the ? button is hidden (setVisible(false) — note: today's code only disables it) and the help overlay is closed. Dismiss → scene.start('WorldSelect') (closes the world-select spec's deferred item).
  • Exit button (TableScene.ts:273-282): instead of jumping to WorldSelect, call this.game_.abandon() (fires RunEnded → fold → stats current), then show the same summary with the abandoned outcome; dismiss → WorldSelect. The handler is gated on status === 'playing' (today's code disables the button at game end, but the gate makes "exit can never present a won run as abandoned" explicit rather than load-bearing on button state). The existing SHUTDOWN abandon stays as the safety net (now a no-op on this path). The summary's input-swallowing background is what blocks further board interaction after abandon (dispatch-after-abandon throws by design).
  • World display name comes from worldDisplayManifest[worldId].name — authored text is the right source for a name; the REQ-STATS-12 manifest ban applies to badge data, not the summary's name lookup.
  • formatDuration unit tests (sub-minute, minutes, hours, zero).
  • Scene-level test with the real collector (never mocked core, existing conventions): win a run headless, assert the summary becomes visible and its text objects contain the REQ-STATS-8 fields and "Run 1".
  • Abandon-via-exit test: session closed as abandoned exactly once, summary shown with the run's fully-tallied counters, fold matched REQ-STATS-2 semantics; exit handler no-ops once status is won/lost.
  • Grep: no scene/view module imports RunStatsCollector or fold/persist functions (REQ-STATS-10).
