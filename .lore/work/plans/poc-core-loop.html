Building the smallest loop that exercises the seam

Nine dependency-ordered steps that turn the approved POC spec into a typed, tested, deterministic, deployable vertical slice. Each step ends at a validation gate that must pass before the next begins.

Decisions this plan makes (veto any on review)

The spec and design intentionally deferred these to implementation. Defaults are chosen below; the plan, not the code, is the place to object.

  • PRNG = sfc32, seeded via splitmix32 expansion. Tiny, dependency-free, good statistical quality. Its cursor is a plain {a,b,c,d} object of four uint32s held in state, satisfying the spec's "not assumed to be a single integer" guidance (REQ-POC-3, 5). RNG is purely functional: every draw returns [value, nextState]; nothing mutates.
  • Boundary enforced by ESLint flat-config no-restricted-imports + no-restricted-globals, scoped to src/core/**. Forbids importing phaser or anything under game/, and forbids window/document/canvas globals. Zero extra dependency vs eslint-plugin-boundaries; sufficient for a two-directory split (REQ-POC-1).
  • Phaser 3 (latest stable 3.x), not Phaser 4. The engine research committed to Phaser 3 and warned that AI tends to generate for a version or two behind; implementation must pin the current 3.x and write against its scene API, verified at install (REQ-POC-15).
  • Vite base = /shattered-worlds/. The git remote is github.com/rjroy/shattered-worlds, so the Pages project site is rjroy.github.io/shattered-worlds/ and assets must resolve under that subpath (REQ-POC-17).
  • CI triggers on master. That is this repo's default branch; the deploy job keys off it (REQ-POC-18).
  • Card ids are stable card-01…card-10, assigned at library build before any shuffle; values are fixed per id. Used as render keys (REQ-POC-4).
  • Draw-pile top = front of the array (index 0). Drawing slices from the front; shuffle (Fisher-Yates) defines the order. A stated convention so tests and renderer agree (REQ-POC-6, 7).
  • Core is a pure reducer with a thin stateful façade. reduce(state, action) → {state, events} is the pure, exhaustively testable heart; createGame(seed) returns a GameCore holder that calls the reducer and stores the snapshot for the renderer. Determinism tests hit the reducer directly (REQ-POC-2).

Strictly ordered: each arrow is a hard dependency. Steps 2–5 are the pure core (no engine); steps 6–8 add the renderer, build, and deploy. Green-bordered nodes are gated by an automated check.

STEP 1Project scaffold & toolchain POC-15POC-1 Depends on: nothing (greenfield). Unlocks every later step.

Goal

Stand up the Bun + Vite + strict-TypeScript project with the lint-enforced core/render boundary wired before any game code exists, so the boundary can never be violated silently.

Files to create

  • package.json — Bun project. Dependency phaser (3.x). Dev deps: vite, typescript, eslint, @eslint/js, typescript-eslint, @types/bun. Scripts: dev (vite), build (vite build), preview (vite preview), test (bun test), typecheck (tsc --noEmit), lint (eslint), sim (bun run src/sim/run.ts).
  • tsconfig.json — strict: true, noUncheckedIndexedAccess, exactOptionalPropertyTypes, bundler module resolution, types: ["bun"], target/lib ES2022+DOM.
  • vite.config.ts — base: '/shattered-worlds/', root index, build to dist/.
  • index.html — single canvas mount + <script type="module" src="/src/game/main.ts">.
  • eslint.config.js — flat config. Base JS + typescript-eslint recommended. Override block matching src/core/**/*.ts: no-restricted-imports (patterns phaser, **/game/**) and no-restricted-globals (window, document, navigator).
  • src/core/, src/game/, src/sim/ — placeholder dirs (a stub index.ts each so lint/typecheck have targets).

Notes

  • The existing .gitignore already covers node_modules, dist, build, and *.tsbuildinfo — no change needed.
  • Run bun install and commit bun.lock (REQ-POC-15, reproducible-builds rule).
  • Pin exact versions; verify Phaser's current 3.x at install rather than trusting memory (engine-research caution).
✓ Validation gate
  • bun install from the committed lockfile reconstructs the graph with no error.
  • bun run typecheck and bun run lint both pass on the stubs.
  • A throwaway probe — adding import 'phaser' to a file under src/core/ — makes bun run lint fail; removing it makes lint pass again. Proves the boundary is live before real code lands.
STEP 2The seam contract & card data POC-2POC-4POC-5POC-8 Depends on: Step 1.

Goal

Define the immutable types that the whole architecture rests on: state, actions, events, the error contract, and the fixed ten-card library. No behaviour yet, just the shapes everything else is checked against.

Files to create

  • src/core/types.ts — the concrete schemas: 
    type CardId = string            // "card-01" … "card-10"
interface Card { id: CardId; value: 1 | 2 | 3 }

type Action =
  | { type: 'PlayCard'; cardId: CardId }
  | { type: 'EndTurn' }

type GameEvent =
  | { type: 'CardPlayed'; cardId: CardId; value: number; runningTotal: number }
  | { type: 'TurnEnded'; total: number }
  | { type: 'CardsDiscarded'; cardIds: CardId[] }
  | { type: 'DeckShuffled' }
  | { type: 'CardsDrawn'; cardIds: CardId[] }

interface GameState {
  drawPile: readonly Card[]   // top = index 0
  hand: readonly Card[]
  played: readonly Card[]     // in play order
  discard: readonly Card[]
  runningTotal: number
  history: readonly number[]
  rng: RngState
}

// RngState is defined here (Step 2), not in rng.ts, so GameState
// typechecks now and Step 3's rng.ts imports it. Shape is fixed.
interface RngState { a: number; b: number; c: number; d: number }
  • src/core/errors.ts — class IllegalActionError extends Error carrying readonly action: Action and readonly state: GameState (the single error contract, REQ-POC-8).
  • src/core/cards.ts — buildLibrary(): Card[] returning exactly five value-1, three value-2, two value-3 cards with ids card-01…card-10 in a fixed deterministic order (sum 13).
  • src/core/contract.ts — type re-exports + the GameCore interface (a pure declaration, no function bodies). The reduce/createGame implementations live in their own files in Step 4, so this file typechecks cleanly on its own now.

Notes

  • All array fields are readonly; updates produce new arrays. This is the immutability the determinism guarantee leans on.
  • Consult the typescript-quality skill: prefer discriminated unions over enums, no any, no non-null assertions.
✓ Validation gate
  • bun run typecheck passes; bun run lint clean.
  • A unit test asserts buildLibrary() returns ten cards, composition 5/3/2, value sum 13, and ten unique ids (REQ-POC-4).
STEP 3Seeded PRNG & deterministic shuffle POC-3 Depends on: Step 2 (Card type).

Goal

The single source of all game randomness, purely functional and serializable, so that same seed + same actions reproduce byte-identical runs.

Files to create

  • src/core/rng.ts:
    • type RngState = { a: number; b: number; c: number; d: number } (four uint32s).
    • createRng(seed: number): RngState — expand the numeric seed through splitmix32 four times to fill state.
    • nextFloat(rng: RngState): [value: number, next: RngState] — sfc32 step, value in [0,1). Pure: never mutates the input.
    • shuffle<T>(items: readonly T[], rng: RngState): [shuffled: T[], next: RngState] — Fisher-Yates threading the rng state through each swap.

Notes

  • Returning [value, nextState] rather than mutating keeps the RNG inside the immutability story and makes it trivially testable without injection (matches the testing rule's "no module mocking" preference).
✓ Validation gate (bun test, no DOM)
  • Determinism: two createRng(sameSeed) states produce identical nextFloat sequences over N draws.
  • Shuffle determinism: same seed + same input array ⇒ identical permutation; a different seed ⇒ a different permutation (REQ-POC-3 validation).
  • Purity: calling nextFloat(state) twice on the same input returns the same value (input unchanged).
STEP 4The reducer: rules, dispatch & events POC-2POC-6POC-7POC-8POC-9POC-10POC-11POC-13 Depends on: Steps 2 & 3. This is the heart of the POC.

Goal

Implement the full turn loop as a pure reducer returning { state, events }, plus the stateful GameCore façade the renderer drives. This is where every game rule lives.

Files to create

  • src/core/draw.ts — internal draw(state, n) helper: pull n from the front of the draw pile; if it empties before n, shuffle the discard (seeded) into a new draw pile, emit DeckShuffled, and continue. Returns new zones + CardsDrawn{cardIds} (+ optional DeckShuffled) and the advanced rng (REQ-POC-7).
  • src/core/reduce.ts — reduce(state, action): { state, events }:
    • PlayCard — if cardId not in hand, throw new IllegalActionError(action, state). Else move card hand→played (append), add value to running total, emit CardPlayed{cardId, value, runningTotal} (REQ-POC-8).
    • EndTurn — strict order: (1) append running total to history; (2) move hand + played into discard; (3) reset running total to 0; (4) draw 5 (reshuffle as needed). Emit TurnEnded{total}, CardsDiscarded{cardIds}, optional DeckShuffled, CardsDrawn{cardIds}, in that order. Legal even when hand+played are empty (records 0) (REQ-POC-9, 10).
  • src/core/game.ts — createGame(seed): GameCore: build library, shuffle (seeded), draw opening 5, total 0, empty history. The façade holds state and exposes dispatch(action) that calls reduce, stores the new snapshot, and returns { state, events } (REQ-POC-2, 6).

Notes

  • Ten cards are conserved at every step: a private invariant check (four zones = ten unique ids) reused by tests and the sim runner (REQ-POC-5).
  • Events carry specific card ids so the renderer can animate per card without diffing states (REQ-POC-11).
  • config parameter from the design is intentionally omitted for the POC (REQ-POC-6).
✓ Validation gate (bun test, no Phaser/DOM/canvas — REQ-POC-13)
  • New game: draw pile 5, discard 0, total 0, history empty, hand of 5.
  • PlayCard sums correctly; playing a card not in hand (unknown / already played / discarded) throws IllegalActionError carrying action + state.
  • EndTurn: history gains exactly one entry equal to the turn total, running total resets to 0, hand refills to 5, and exactly one seeded reshuffle fires (observable via DeckShuffled).
  • Conservation invariant holds across a multi-turn sequence.
  • Replay equivalence: same seed + same action list ⇒ deep-equal final state and deep-equal event log; a second, different seed yields a different initial shuffle (≥2 seeds, REQ-POC-3/13).
  • Synchrony: dispatch(...) result is a plain object, not a thenable (assert no .then) (REQ-POC-2).
STEP 5Headless sim runner — the second driver POC-14 Depends on: Step 4.

Goal

Prove the "one core, two drivers" property: the same core, driven at full speed with no renderer, survives a long random-legal run while conservation holds.

Files to create

  • src/sim/policy.ts — pure action chooser: given a state and an rng draw, if the hand is non-empty choose uniformly among playing each individual hand card or ending the turn; if the hand is empty, end the turn. Because a fresh 5 is dealt after every EndTurn, it never selects an illegal action (REQ-POC-14).
  • src/sim/run.ts — drive createGame(seed) through the policy for a configurable count (default ≥1000), asserting the conservation invariant every step; exit non-zero on any throw or violation. Runnable via bun run sim.

Notes

  • The runner ignores the event log entirely — it consumes only state — demonstrating the renderer is optional to the core (design's Option B property).
  • The policy's own choice randomness uses the core RNG helpers (seeded) so a sim run is itself reproducible.
✓ Validation gate (bun test + bun run sim)
  • A test dispatches ≥1000 random-legal actions through the core alone; completes without throwing and preserves conservation throughout (REQ-POC-14 validation).
  • bun run sim exits 0 and prints a short summary (turns, actions, final history length).
STEP 6Phaser renderer makes it playable POC-12POC-11POC-16 Depends on: Step 4 (core API). Independent of Step 5.

Goal

A plain but playable Phaser scene that observes core state and turns clicks into actions. Visual polish is explicitly out of scope (REQ-POC-12); the bar is "visibly playable".

Files to create

  • src/game/main.ts — Phaser 3 Game bootstrap (config, canvas size, single scene), mounted into index.html.
  • src/game/TableScene.ts — holds one GameCore (from createGame with a seed). Renders:
    • the hand as clickable card objects (click ⇒ dispatch(PlayCard)),
    • the played pile with the live running total,
    • draw and discard pile counts,
    • the turn-total history,
    • an End Turn button (click ⇒ dispatch(EndTurn)).
    After each dispatch it re-reads state and redraws. Event-log animation is allowed but not required.
  • src/game/render.ts (optional) — small pure helpers (card label, layout positions) to keep the scene readable and under the size heuristic.

Notes

  • The renderer is the sole importer of phaser; the core lint boundary guarantees it stays that way.
  • Write against the pinned Phaser 3.x scene API — do not assume an older version's signatures (engine-research caution).
  • The renderer owns the clock: for the POC it may snap straight to state after each dispatch.
✓ Validation gate (observable + automated)
  • bun run typecheck and bun run lint pass; the boundary lint confirms no phaser/DOM import has leaked into src/core/ (REQ-POC-1).
  • bun run dev serves the game; the browser shows hand, running total, pile counts, history, and a working End Turn button; clicking cards and ending turns advances the loop (REQ-POC-12, 16). Confirm via the verify/run flow with a screenshot.

STEP 8CI: gate then deploy to GitHub Pages POC-18POC-13 Depends on: Steps 1–7 (lint, tests, build all green).

Goal

One pipeline on push to master: lint + core tests + build as gates, deploy dist/ to Pages only on success.

Files to create

  • .github/workflows/ci.yml — on: push: branches: [master] (plus PR for the gate-only run). Steps: checkout → oven-sh/setup-bun → bun install --frozen-lockfile → bun run lint → bun run typecheck → bun run test → bun run build. typecheck is a gate too: ESLint catches import-boundary breaks, but only tsc --noEmit enforces strict-mode type errors (REQ-POC-15). Then, gated on success and Pages permissions (pages: write, id-token: write): actions/configure-pages → actions/upload-pages-artifact (path dist) → actions/deploy-pages.

Manual prerequisite (flag to user)

  • In the GitHub repo settings, set Pages → Source = GitHub Actions. This is a one-time console step the workflow cannot do for itself; deploy fails without it.
  • Ensure the workflow grants pages: write and id-token: write permissions and targets a github-pages deployment environment. A first-time Pages-via-Actions deploy can fail for these reasons even with the Source set correctly.
✓ Validation gate
  • The workflow runs green and deploys; the live rjroy.github.io/shattered-worlds/ URL serves the same playable loop as local dev (REQ-POC-18).
  • A deliberately failing test (committed on a throwaway branch / reverted) blocks the deploy job — proving the gate, per the spec's CI validation check.

⚠ Risks & watch-points

  • Phaser version drift. AI commonly writes for an older Phaser 3 API. Pin the current 3.x at install and verify the scene API against the installed version, not memory.
  • Pages base trap. A wrong or missing base yields a white screen with 404s on assets. Step 7 tests it under the real subpath before CI ever runs.
  • Pages source setting. The deploy fails until Pages → Source = GitHub Actions is set by hand once (Step 8 prerequisite).
  • Hidden mutation breaks determinism. Any in-place array edit in the core silently corrupts replay equivalence. readonly fields + the replay test are the guard; treat a replay-equivalence failure as a mutation hunt.
  • bun test + DOM. Core tests must import nothing that pulls in Phaser/DOM. The boundary lint plus a no-Phaser test environment catch a leak immediately.
