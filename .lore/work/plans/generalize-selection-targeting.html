Implementation Plan

Generalize selection targeting

Replace the per-spec-kind awaiting-* phases with one generic targeting state that walks a card's TargetSpec steps, so any Sequence ordering works — Barricade (hazard → return), Cut It Loose (destroy → hazard), and whatever comes next — with zero per-card interaction code.

Renderer-only · core untouched
Source artifacts. This plan restores the intent of the approved design targeting-interaction.html, which specified: "What differs per card is only its TargetSpec, which the renderer reads and drives generically. Adding a card later adds no new interaction code if its spec is a known shape." The per-kind phases were a recorded shortcut (notes/009-renderer.html: "compound spec (Barricade) handled inline"). The immediate trigger is Cut It Loose in src/data/worlds/bird-building.json (Sequence [DestroyCardInHand 1–1, DealProgress 4]), introduced by the shard-response-archetypes plan, whose P1 explicitly warned: "verify checkSpec compound path accepts this step pairing in a test before authoring more of its shape."

Why it fails today

The only inter-step transition in the codebase is a literal Barricade check in TableScene.onCardClick (~lines 584–603): after a hazard pick, "if the spec is compound and steps[1] is returnWorld, hand-build awaiting-return with a targetId rider." Cut It Loose's shape (destroy → hazard) has no transition: playing it enters awaiting-destroy, the first click dispatches PlayCard {cardId, destroyId} with no targetId, and the core's checkSpec rejects it at the hazard step. Supporting tangles:

  • activeStep() conflates modal branch index with sequence step index (sel.idx ?? 0), and hard-codes awaiting-return's "step 1 iff targetId set".
  • No general accumulator for earlier steps' picks — just the one-off targetId rider on awaiting-return, merged back in buildAction.
  • isComplete/hintForSelection hard-code destroyHand as min=0 / "(optional)"; the Confirm button shows for awaiting-destroy, wrong for an exact-count (min=1/max=1) step.

The core needs no changes: structuralSpec, legalTargets(cardId, step), and checkSpec's compound walk are already step-generic, and PlayCard already carries one slot per step kind (targetId, returnIds, destroyId, discardId, choice).

Target design

type StepResult =
  | { kind: 'hazard'; targetId: CardId }
  | { kind: 'destroyHand'; destroyId?: CardId }     // undefined = skipped optional destroy
  | { kind: 'returnWorld'; returnIds: readonly CardId[] }
  | { kind: 'discardPlayer'; discardId: CardId }

type SelectionState =
  | { phase: 'idle' }
  | { phase: 'awaiting-modal'; cardId: CardId }
  | {
      phase: 'targeting'
      cardId: CardId
      choice?: number              // modal branch index, when entered through a modal
      steps: readonly TargetSpec[] // leaf spec → [spec]; compound → its steps, ORIGINAL indices kept
      stepIdx: number              // stepIdx === steps.length ⇒ complete, ready to build
      done: readonly StepResult[]  // results of completed steps, in step order
      current: readonly CardId[]   // picks accumulating for steps[stepIdx]
    }
BehaviorGeneric rule (from the approved design)
Auto-advanceExact-count steps (effective min === max, e.g. hazard, discardPlayer, destroy 1–1) advance the moment current.length === max. No Confirm.
Confirm buttonVariable-count steps only (min < max, or min === 0): visible once current.length ≥ min; clicking advances the step.
Step legalityAlways availableActions(state).legalTargets(cardId, activeStep(sel)); activeStep = choice ?? stepIdx (modal cards key legality by branch index, sequences by step index — now explicit instead of the idx ?? 0 accident).
'none' stepsNever stripped from steps — stripping would desync stepIdx from the core's sequence index, breaking legalTargets and effectAtStep for any sequence with a 'none' step before a targeting step. Instead, advance() auto-skips them (they need no picks; the core applies them on dispatch), and beginTargeting applies the same skip to leading 'none' steps. Indices stay 1:1 with the core's Sequence.steps.
CompletionstepIdx === steps.length ⇒ buildAction folds done into one flat PlayCard.
Hints / preview / connector / highlightAll derived from steps[stepIdx] kind + min/max + counts, and from done/current — never from a phase name.

S1

Rewrite the selection machine around the generic targeting state

depends on: S0

Files: src/game/interaction/selection.ts (rewrite), src/game/tests/selection.test.ts (rewrite alongside)

Replace the five per-kind phases with the targeting state above. All transitions stay pure functions with no GameState dependency (preserved properties from notes/009).

New / changed API
  • beginTargeting(cardId, spec, choice?) — flattens a leaf spec to [spec], a compound to its steps with original indices preserved (see "'none' steps" rule above), then skips past any leading 'none' steps via the same skip logic advance uses. A result that is already complete (spec 'none', or all steps are 'none') is dispatched immediately by the caller. This single entry point also covers what the old 'selected' phase did, so that phase is deleted.
  • chooseModal(sel, choice, spec) — now just beginTargeting(cardId, spec.branches[choice], choice). The default: awaiting-hazard fallback arm is deleted (a branch shape the machine doesn't know must not silently masquerade as hazard targeting).
  • togglePick(sel, id) — unifies addTarget/removeTarget: adds to current (capped at the step's max), removes if already present.
  • advance(sel) — folds current into a StepResult appended to done, bumps stepIdx past any trailing/interior 'none' steps, resets current. Optional destroy (min 0) confirmed with zero picks folds to destroyId: undefined, and buildAction then omits the destroyId field entirely — a direct port of the old awaiting-destroy case (selection.ts ~235–248), keep that guard in the fold, not scattered downstream.
  • autoAdvances(sel) — true when the current step is exact-count and current.length === max (hazard and discardPlayer have implicit min = max = 1).
  • needsConfirm(sel) / stepSatisfied(sel) — needsConfirm is true iff the current step's effective min ≠ max (hazard and discardPlayer are implicitly 1/1); stepSatisfied is exactly current.length ≥ min — never "> 0", or min=0 steps (Barricade's return, optional destroys) would wrongly hide Confirm. Together they replace the hard-coded Confirm-visibility pair in TableScene and the per-phase isComplete cases.
  • isComplete(sel) — phase === 'targeting' && stepIdx === steps.length.
  • activeStep(sel) — sel.choice ?? sel.stepIdx. Deletes the awaiting-return ternary and the branch/step conflation. Document the semantics at the function: for a modal card the value is the branch index (the core's computeLegalTargets for Modal dispatches on branches[step]); for a sequence card it is the step index within the card's original Sequence.steps (guaranteed by the no-strip rule); a modal branch that is itself a sequence has no representable step index — out of scope, matching the core.
  • buildAction(sel) — only when complete: fold done into the flat PlayCard slots (+ choice when set). Four of the five old cases deleted.
  • hintForSelection(sel) — derives text from steps[stepIdx]: hazard → "Select a Hazard target"; returnWorld → dynamic "Select N–M … (X chosen)"; discardPlayer → "Select a player card to discard"; destroyHand → "Select a card to destroy" with "(optional)" only when min === 0 (fixes the min=1 lie).
  • getStepIndex — unused outside selection.ts; deleted.
Constraints to encode as comments + tests
  • The flat PlayCard action admits at most one step of each kind per card. The core's checkSpec has the same constraint; assert it in beginTargeting (dev-time throw or console.warn) rather than discovering it as a silent overwrite in buildAction.
  • Modal branches that are themselves compound are out of scope — the core's computeLegalTargets returns [] for nested Sequence under Modal, so the renderer must not pretend to support it. Document at chooseModal.
selection.test.ts coverage (rewrite)
  • Barricade walk: begin → hazard pick auto-advances; assert done is exactly [{kind:'hazard', targetId}] (the old targetId rider does not carry forward — it lives in done now) and stepIdx === 1 → return step (variable 0–1: needsConfirm true, stepSatisfied true at 0 picks since min=0, so Confirm is available immediately — unchanged from today's Barricade UX) → confirm with 0 and with 1 pick → buildAction emits {targetId, returnIds}.
  • Cut It Loose walk: begin → destroy step is exact-count (no Confirm) → pick auto-advances → hazard pick auto-advances → complete → buildAction emits {destroyId, targetId}. This is the test that fails against the old machine.
  • 'none'-step skipping: Panic-shape ([returnWorld(1,2), none]) — assert steps.length === 2 (not stripped), confirm gated until min 1, toggle add/remove, max 2 cap, and advance after the return step skips the 'none' straight to complete. Adrenaline-shape ([discardPlayer, none]) — discard pick auto-advances and the trailing 'none' skip lands on complete in the same transition. Leading-'none' shape ([none, hazard], synthetic) — beginTargeting starts at stepIdx === 1 and activeStep === 1, so legalTargets is queried with the core-aligned index.
  • Sprint modal: chooseModal into a 'none' branch ⇒ complete with choice; into the hazard branch ⇒ targeting with activeStep === choice.
  • needsConfirm boundaries (the headline Cut It Loose fix gets its own unit tests): destroyHand(1,1) ⇒ needsConfirm false / autoAdvances true at one pick; destroyHand(0,2) ⇒ needsConfirm true, confirm with zero picks builds an action without destroyId, and two picks accumulate before Confirm commits.
  • activeStep, hintForSelection — including: destroyHand(0,…) hint contains "(optional)", destroyHand(1,1) hint does not — cancel from every phase, isComplete edges.
Gate bun run test on selection.test.ts green, including the Cut It Loose walk; tsc clean for this module (downstream TableScene errors expected until S3 and tracked, not suppressed).

S3

Rewire TableScene to drive the generic machine

depends on: S1, S2

Files: src/game/scenes/TableScene.ts

Changes by handler
  • onCardClick collapses to two branches. Idle: discard check, else beginTargeting from the playable spec (dispatch immediately if already complete — covers spec 'none'). Targeting: gate on legalTargets(cardId, activeStep(sel)), then togglePick; if autoAdvances → advance; if then complete → buildAction + dispatch, else repaint. The Barricade block (~584–603) is deleted.
  • nextSelection shrinks to: modal → awaiting-modal + showModalChooser; everything else → beginTargeting. The per-kind switch arms are deleted.
  • onConfirmClick: if stepSatisfied → advanceSelection() (see below). (Today it only ever dispatched — with steps after a variable step now possible, confirm means "finish this step", not "finish the card".) currentSpec() loses its only caller here and is deleted.
  • New private advanceSelection() — the single scene-level step-advance path: calls advance(this.sel), then clearConnector(), then either buildAction+dispatch (complete) or drawAll(). Both onCardClick (auto-advance) and onConfirmClick route through it — no other call site may advance or repaint after a pick.
  • Confirm visibility in drawAll: phase === 'targeting' && needsConfirm(sel) && stepSatisfied(sel) (was: phase name pair; also fixes Confirm showing before min is met on a future min≥1 variable step).
  • currentLegalTargetIds: gate becomes phase === 'targeting' && !isComplete(sel).
  • showTargetPreview: gate on current step kind === 'hazard'; pass sel.choice as the branch index to previewPlay (was: sel.idx).
  • showConnector: gate on current step kind ∈ {hazard, destroyHand, returnWorld} — the same three targeting types as today's phase set, now spec-driven. stepConnectorStyle/effectAtStep keep working unchanged: they already take the numeric step, which activeStep still supplies (REQ-FEEDBACK-2/-3 preserved).
  • onModalChoose: chooseModal now returns a targeting state; if complete (none branch) build + dispatch with choice, else repaint.
Known bug class — honor the lesson The stale-connector bug (feedback implementation notes) bit exactly at step transitions that repaint without a pointerout. Every path that calls advance() or dispatches must call clearConnector() before drawAll(). The advanceSelection() helper above is that single path — the clearConnector() lives inside it so no future step ordering can forget it.
Invariants from the approved design (must survive untouched) Selection state never enters GameState; legality is only ever read from availableActions, never cached across dispatches; Cancel/Esc abort from any phase without dispatching; End Turn stays disabled during selection; no destructive single click from idle; touch parity (no affordance depends on hover — hints and highlights carry the interaction, preview/connector remain hover-only sugar).
Gate bun run lint (core/game boundary intact) and full tsc clean. No references to the old phase names remain anywhere in src/ (grep -r "awaiting-hazard\|awaiting-return\|awaiting-discard\|awaiting-destroy" src/ returns nothing).
