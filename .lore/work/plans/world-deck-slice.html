Implementation Plan

Turning the POC seam into a real game

Eleven dependency-ordered steps that replace the POC's draw/sum loop with the world-deck game: Progress vs Hazard, acts, acquisition, and the Zombie Big-Box content, plus a renderer that makes it playable. The pure core (steps 1–8) lands first and is exhaustively tested headless; the renderer and sim consume it. Each step ends at a validation gate that must pass before the next begins.

Gaps the spec left, reconciled here (object on review)

These are the few places where the spec/design under-specify or lightly conflict. Each is resolved with a stated default; flag any you disagree with before implementation.

  • The effect vocabulary has no sequence wrapper, but Barricade needs one. REQ-WDS-3 calls the effect enum "closed" and lists DealProgress and ReturnWorldCards separately, with only a Modal wrapper. Barricade is "deal +1, then return up to 2" (REQ-WDS-23), and the approved targeting design models it as compound [hazard, returnWorld]. Resolution: add exactly one Sequence effect wrapper (an ordered list of sub-effects) to express Barricade, sanctioned by the targeting design's compound spec. Panic uses the same wrapper (Sequence[ReturnWorldCards 1..1; Draw world 1]), same justification. Still a closed enum, just one entry wider than REQ-WDS-3's literal list.
  • Created cards need deterministic ids. The POC's fixed card-01…10 scheme can't survive a game that mints reward cards. Resolution: GameState carries a monotonic nextId counter; every minted instance (catalog deal at world build, and every GainCard) takes its id from the counter, so ids stay unique and replay-identical without any wall-clock or non-seeded source. Card templates (catalog definitions) are distinct from instances (id-bearing copies in zones).
  • Renderer scope = full interaction grammar, no polish. REQ-WDS-27 says "clear, correct interaction" not visual polish, and the targeting design defers juice, drag, keyboard, and tap-and-hold. Resolution: implement the complete grammar (select → resolve TargetSpec by highlight → commit, Cancel always present, discard zone, modal chooser, multi-select + Confirm, plain-text target previews, win/loss screens), all driven by availableActions; skip shimmer/particles/intensity FX/drag/keyboard. The intensity signal is still built in the core (REQ-WDS-26); the renderer simply does not yet consume it for FX.
  • Opening hand uses the same draw routine. The spec describes the per-turn refill (REQ-WDS-7) but not the opening deal explicitly. Resolution: createWorld shuffles the starter deck into the player draw pile and Act 1 into the world draw pile, then runs the identical refill-toward-6 routine with zero world cards held (→ 4 player + 2 world), so there is one draw code path, never two.

Decisions this plan makes (veto any on review)

Implementation-level choices the spec and designs intentionally left open. Defaults below; the plan, not the code, is the place to object.

  • Replace the POC loop in place; reuse rng.ts and the seam shape. types.ts, cards.ts, draw.ts, reduce.ts, game.ts, contract.ts are rewritten for the world-deck domain. rng.ts (sfc32 + splitmix32, functional, cursor-in-state) is kept verbatim. IllegalActionError is kept as the single error contract. The POC's reduce/cards/game tests are replaced by world-deck tests; git history preserves the POC (REQ-WDS-1, 2).
  • Hand is one mixed array of a discriminated Card union. Card = PlayerCard | WorldCard on a kind field; hand: readonly Card[] holds both (the clog). Held world cards count is just hand.filter(kind==='world') (REQ-WDS-6, 7, 12).
  • Per-Hazard turn-Progress is a state-level map, not a field on the instance. progress: Record<CardId, number> keyed by the Hazard id in hand; reset to {} at EndTurn. Keeps the card instance immutable and the no-carry-over reset trivial (REQ-WDS-9, 10, 13).
  • One legality source: availableActions(state). A new pure core selector returns playable cards with their TargetSpec, the discardable Hazards, legalTargets(cardId, step), and canEndTurn. dispatch validates against the same predicates; the sim policy and renderer both read it. Targeting rules exist exactly once (targeting design; REQ-WDS-8, 27, 29).
  • Action shape. PlayCard{ cardId, targetId?, choice?, returnIds?, destroyId?, discardId? }, DiscardHazard{ cardId }, EndTurn. Any illegal action throws IllegalActionError(action, state) (REQ-WDS-8, 11).
  • Shared primitives, not duplicated logic. dealProgress (with auto-resolve), gainCard(template, dest), addWorldCardToTop, returnToTopThree, drawPlayer/drawWorld, damage, heal live once and are called by both player-card effects and Hazard penalties/rewards (REQ-WDS-14, 17).
  • File layout favors small, cohesive modules (the reducer would otherwise blow past the 100-line heuristic): types.ts, cards.ts (catalog), world.ts (Zombie content + createWorld), draw.ts (refill + act advancement), effects.ts (effects/penalties/rewards + primitives), available.ts (selector), reduce.ts (action handlers), intensity.ts, game.ts, contract.ts.
  • Seeds for golden tests are chosen during step 6, then frozen. The win-path and loss-path golden replays pin a specific (seed, action log); those constants are recorded in the test once a passing path is found, and never regenerated (architecture's replay property; REQ-WDS-22, 28).
  • Build, serve, deploy are untouched. Bun + Vite + strict TS, base: '/shattered-worlds/', the existing CI workflow and committed bun.lock all carry over unchanged (REQ-WDS-30).

STEP 1Domain types & vocabularies WDS-2345625 Depends on: nothing new (replaces POC types). Unlocks every later step.

Goal

Define the immutable schemas the whole slice rests on: the card union, effects/penalties/rewards as closed discriminated unions, keywords, the full action and event sets, GameState, and the TargetSpec / AvailableActions shapes. No behaviour yet, only the shapes everything else is checked against.

Files

  • src/core/types.ts rewrite — the concrete schemas: 
    type CardId = string
type Keyword = 'Hidden' | 'Creature' | 'Slow'   // Hazard tags

type Effect =                                  // player-card effects (closed)
  | { kind: 'DealProgress'; base: number; bonus?: { tag: Keyword; amount: number } }
  | { kind: 'Draw'; player?: number; world?: number }
  | { kind: 'Heal'; amount: number }
  | { kind: 'ReturnWorldCards'; min: number; max: number }
  | { kind: 'DestroyCardInHand'; min: 0; max: 1 }
  | { kind: 'DiscardThenDraw'; player: number }       // Adrenaline
  | { kind: 'AddCard'; template: CardTemplateId; dest: Dest } // spec's AddCard
  | { kind: 'AddWorldCardToTop'; template: CardTemplateId } // Summon Door
  | { kind: 'Modal'; branches: Effect[] }              // Sprint
  | { kind: 'Sequence'; steps: Effect[] }             // Barricade (see Gaps)

type Penalty =
  | { kind: 'Damage'; amount: number } | { kind: 'SkipDrawNextTurn' }
  | { kind: 'GainCard'; template: CardTemplateId }
  | { kind: 'AddWorldCardToTop'; template: CardTemplateId }
  | { kind: 'None' }      // the Door's no-discard rule is the discardable=false flag, not a penalty

type Reward =
  | { kind: 'GainCard'; template: CardTemplateId }
  | { kind: 'AddPlayerCardToTop'; template: CardTemplateId }
  | { kind: 'AddWorldCardToTop'; template: CardTemplateId }
  | { kind: 'SurviveWorld' } | { kind: 'None' }

interface PlayerCard { kind: 'player'; id: CardId; name: string; effect: Effect }
interface WorldCard { kind: 'world'; id: CardId; name: string; cost: number;
  keywords: readonly Keyword[]; discardable: boolean; penalty: Penalty; reward: Reward }
type Card = PlayerCard | WorldCard

type Action =
  | { type: 'PlayCard'; cardId: CardId; targetId?: CardId; choice?: number;
      returnIds?: CardId[]; destroyId?: CardId; discardId?: CardId }
  | { type: 'DiscardHazard'; cardId: CardId }
  | { type: 'EndTurn' }

interface GameState {
  playerDraw: readonly Card[]; hand: readonly Card[]; playerDiscard: readonly Card[]
  worldDraw: readonly WorldCard[]               // current act, ordered
  acts: readonly (readonly WorldCard[])[]        // remaining acts queue
  actIndex: number; progress: Readonly<Record<CardId, number>>
  hp: number; skipDrawNext: boolean
  status: 'playing' | 'won' | 'lost'
  rng: RngState; nextId: number
}

  | { kind: 'none' } | { kind: 'hazard'; tag?: Keyword }
  | { kind: 'modal'; branches: TargetSpec[] }
  | { kind: 'returnWorld'; min: number; max: number }
  | { kind: 'destroyHand'; min: 0; max: 1 }
  | { kind: 'discardPlayer' } | { kind: 'compound'; steps: TargetSpec[] }
interface AvailableActions {
  playable: { cardId: CardId; spec: TargetSpec }[]
  discardable: CardId[]; canEndTurn: boolean
  legalTargets(cardId: CardId, step: number): CardId[]
}
  • src/core/types.ts — the full GameEvent union from REQ-WDS-25 (CardPlayed, ProgressDealt{hazardId,amount,hazardTurnTotal}, HazardResolved, HazardDiscarded, DamageDealt, DrawSkipped, CardGained{id,dest}, CardDestroyed, WorldCardsReturned{ids}, HpChanged{hp}, CardsDiscarded, DeckShuffled, ActAdvanced{act}, CardsDrawn{ids}, TurnEnded, WorldWon, WorldLost).

  • Consult the typescript-quality skill: discriminated unions over enums, no any, no non-null assertions, readonly on every array (the immutability the determinism guarantee leans on).

STEP 3Draw, refill-toward-6 & act advancement WDS-718 Depends on: Steps 1–2. Closes the createWorld opening-deal dependency.

Goal

The one draw routine used by both the opening deal and every refill, plus the act-advancement rule that fires on any world draw, mid-turn or at end of turn.

Files

  • src/core/draw.ts rewrite:
    • drawPlayer(state, n) — pull from playerDraw; if it empties mid-draw, reshuffle playerDiscard (seeded) into a new pile and emit DeckShuffled, then continue (as in the POC).
    • drawWorld(state, n) — pull from worldDraw; when it empties and a world card is still required, advanceAct: shuffle the next queued act into worldDraw, bump actIndex, emit ActAdvanced{act}, continue. If no act remains and the pile is empty, draw nothing (no crash) (REQ-WDS-18).
    • refillHand(state) — the table in REQ-WDS-7: heldWorld = hand.filter(world).length; worldToDraw = max(1, 2 - heldWorld) capped by room-in-hand (toward 6) and world cards remaining; then player draws fill the remainder up to 6. If skipDrawNext is set, suppress player draws this refill but still take the minimum world card; emit DrawSkipped. When the hand is already at 6 from held world cards, draw nothing. Emits CardsDrawn{ids}.
  • src/core/world.ts — finish createWorld by calling refillHand for the opening deal (0 held → 4+2).

Notes

  • Mid-turn world draws (Sprint's draw branch, Panic) reuse drawWorld, so act advancement during a turn is the same code path; a Hazard drawn mid-turn lands in hand and is targetable that turn (REQ-WDS-18).
  • Held/unresolved world cards in hand are never touched by an act boundary.
✓ Validation gate (bun test)
  • Draw table exact: holding 0/1/2 world cards yields 4+2 / 4+1 / 3+1 (REQ-WDS-7).
  • Skip-draw suppresses player draws but still yields the minimum one world card; a hand already at 6 draws nothing.
  • Emptying the current act's world pile advances to the next act (ActAdvanced) while held world cards remain in hand; mid-turn advancement works the same.
  • Driving past Act 3's last world card leaves the world pile empty and the minimum-one draw yields nothing rather than throwing (REQ-WDS-18).
  • Player-pile exhaustion reshuffles the player discard (seeded) and emits DeckShuffled.
STEP 4Effects, penalties & rewards (the primitives) WDS-589141516172021 Depends on: Steps 1–3 (types, catalog, draw).

Goal

Every rule that moves a card, deals Progress, or changes HP, as a set of small shared primitives plus the dispatchers that apply a player-card Effect, a Hazard Penalty (on discard), and a Hazard Reward (on resolve). This is where the mechanics live.

Files

  • src/core/effects.ts new — primitives, each (state, …) → { state, events }:
    • dealProgress(state, hazardId, base, bonus?) — Progress = base + (bonus.amount if hazard has bonus.tag), added to progress[hazardId]; emit ProgressDealt{hazardId, amount, hazardTurnTotal}; then auto-resolve: if progress[hazardId] ≥ cost, grant its reward, remove it from hand, emit HazardResolved{id} (excess wasted). Within the same dispatch (REQ-WDS-5, 8, 9).
    • gainCard(state, template, dest) — mint a fresh instance to player-discard (default), player-draw top, or world-draw top; emit CardGained{id, dest} (REQ-WDS-14).
    • returnToTopThree(state, ids) — remove named world cards from hand, insert into the top three positions of worldDraw ordered by the seeded PRNG; emit WorldCardsReturned{ids} (REQ-WDS-15).
    • destroyInHand(state, id?) — remove from the game (no zone); emit CardDestroyed{id} (REQ-WDS-16).
    • damage(state, n) / heal(state, n) — adjust HP, emit DamageDealt + HpChanged; damage sets status='lost' + WorldLost at HP ≤ 0 (REQ-WDS-17, 21).
    • grantReward(state, reward) — dispatches the reward vocabulary; SurviveWorld sets status='won' + WorldWon (REQ-WDS-20, 21).
    • applyPenalty(state, penalty) — Damage, SkipDrawNextTurn (idempotent boolean, no stacking), GainCard, AddWorldCardToTop, None (REQ-WDS-17).
    • applyEffect(state, effect, action) — the player-effect dispatcher, including Modal (pick action.choice branch), Sequence (apply steps in order, threading state; Barricade deals before returning, so a Hazard resolved in step 1 is simply absent for the return step), ReturnWorldCards/DestroyCardInHand (use action.returnIds/destroyId), DiscardThenDraw (move action.discardId to player-discard, then drawPlayer).

Notes

  • These primitives are called by both applyEffect and applyPenalty/grantReward, so "add a card" and "deal progress" exist once (the architecture's anti-duplication discipline).
  • Validation of legality (is there a target? is the choice in range?) is not here; it belongs to availableActions (Step 5) and is enforced in reduce (Step 6). These functions assume a legal call and focus on state transformation.
✓ Validation gate (bun test)
  • Keyword math: Baseball Bat at a Zombie deals 2+3=5 and resolves the 1-cost Hazard; Explore at a non-Hidden Hazard deals only 1; auto-resolve fires the instant turn-Progress ≥ cost, excess wasted (REQ-WDS-5, 8, 9).
  • Economy: a GainCard reward lands in player-discard; AddPlayerCardToTop/AddWorldCardToTop hit the right pile tops; returns land in the top 3 of the world deck; Regroup removes a hand card from the game entirely (REQ-WDS-14, 15, 16).
  • Penalties: Damage subtracts HP (and can trigger loss); SkipDrawNextTurn is idempotent; GainCard/AddWorldCardToTop move cards; None is inert (REQ-WDS-17).
  • Win wiring: granting SurviveWorld sets won; HP≤0 sets lost (REQ-WDS-20, 21).
STEP 5availableActions — the one legality source WDS-82729 Depends on: Steps 1–4. Consumed by the reducer, the sim, and the renderer.

Goal

A pure selector that answers "what is legal from here?" once, so the renderer's highlights, the reducer's validation, and the sim's policy can never disagree (targeting design's load-bearing decision).

Files

  • src/core/available.ts new — availableActions(state): AvailableActions:
    • playable — for each player card in hand, compute its TargetSpec from its effect and omit it if it has no legal play: a DealProgress/hazard-targeting card is omitted when no Hazard is in hand; Panic (returnWorld 1..1) is omitted unless exactly ≥1 world card is in hand; Adrenaline (discardPlayer) is omitted with no other player card to discard, and its legalTargets(adrenalineId, 0) returns the ids of all player cards in hand other than Adrenaline itself (never world cards); Sprint stays playable (its draw branch is always available) but its hit-Slow branch is marked unavailable when no Slow Hazard exists; Med Kit / Summon Door are none and always playable (REQ-WDS-8, 23).
    • discardable — world cards in hand with discardable === true (excludes the Door) (REQ-WDS-11, 20).
    • legalTargets(cardId, step) — the concrete ids legal for a card's spec at a compound step (e.g. Barricade step 0 = Hazards, step 1 = world cards still in hand).
    • canEndTurn — true while status==='playing'.

Notes

  • The selector returns specs/ids only; it never mutates and never applies effects. It is the single definition of targeting rules (REQ-WDS-27's shared surface).
✓ Validation gate (bun test)
  • A DealProgress card with no Hazard in hand is absent from playable; with a Hazard it appears with a hazard spec.
  • Panic appears only with ≥1 world card in hand; Sprint's hit branch is flagged unavailable with no Slow Hazard but the card remains playable.
  • The Door is never in discardable; other Hazards are.
  • legalTargets for Barricade returns Hazards at step 0 and remaining world cards at step 1.
  • legalTargets for Adrenaline returns the other player cards in hand — not Adrenaline itself, not any world card.
STEP 6The reducer, dispatch & golden replays WDS-12891011121320212228 Depends on: Steps 1–5. The heart of the slice; fresh-eyes review after this gate.

Goal

Wire the action handlers into the pure reduce(state, action) → { state, events }, enforce legality via availableActions, and prove the whole loop deterministic with golden win/loss replays.

Files

  • src/core/reduce.ts rewrite:
    • Guard: if status !== 'playing', any action throws (REQ-WDS-21).
    • PlayCard — validate the card is currently playable and the target/choice is legal using availableActions (no second copy of the rules); else throw IllegalActionError. Remove the card from hand, emit CardPlayed, run applyEffect (auto-resolve happens inside dealProgress), then check win/loss (REQ-WDS-8, 9, 20, 21).
    • DiscardHazard — must name a discardable world card in hand (the Door throws); apply its penalty, remove from hand, emit HazardDiscarded + the penalty event; check loss (REQ-WDS-11, 17).
    • EndTurn — strict order: (1) move every player card in hand to player-discard (world cards stay), emit CardsDiscarded; (2) reset progress to {}; (3) consume skipDrawNext; (4) refillHand. Emit TurnEnded, CardsDiscarded, optional DeckShuffled/ActAdvanced, CardsDrawn in order (REQ-WDS-10, 12, 13).
  • src/core/game.ts rewrite — createGame(seed): GameCore built on createWorld; the façade holds the snapshot and exposes dispatch (calls reduce, stores, returns) and read access to availableActions/intensity.
  • src/core/contract.ts rewrite + index.ts — re-export the new surface (GameCore, createGame, availableActions, intensity, types).

Notes

  • Id-accounting invariant: a shared test helper asserts every id appears in exactly one place (a zone or removed) and every removal/creation is explained by an event (REQ-WDS-2).
  • Golden seeds are searched for once (e.g. by running the sim until a win and a loss are found), then frozen as constants with their recorded action logs (REQ-WDS-22, 28).
✓ Validation gate (bun test, no Phaser/DOM — REQ-WDS-1, 28)
  • No carry-over: partially pay a Hazard, end the turn, and its turn-Progress is 0 (REQ-WDS-10).
  • Hold vs discard: after EndTurn, held world cards remain in hand and unplayed player cards are in player-discard (REQ-WDS-12, 13).
  • Error contract: DealProgress with no Hazard throws; out-of-range Modal choice throws; Panic with no world card throws; DiscardHazard on the Door throws; returning a player-card id throws; any action after won/lost throws (REQ-WDS-8, 11, 15, 21).
  • Golden win: a fixed seed + recorded log reaches won by forcing and paying a Door without ever resolving The Walker, byte-identical on re-run (REQ-WDS-20, 22).
  • Golden loss: a second fixed seed + log reaches lost by lethal damage.
  • Replay equivalence: same seed + actions ⇒ deep-equal final state and event log; a different seed ⇒ different Act 1 order. Id-accounting holds across the sequence (REQ-WDS-2, 28).
  • Fresh-context review of the core (the seam the whole game rests on) before moving on: no hidden mutation, no rules duplicated outside availableActions, no determinism hole.

STEP 8Headless sim — the second driver WDS-292 Depends on: Steps 1–6. Parallelizable with Step 9.

Goal

Files

STEP 9Phaser renderer & targeting grammar WDS-2725 Depends on: Steps 1–6 (core API + availableActions). Independent of Step 8.

Goal

Make the world playable: a Phaser scene that renders the mixed hand and Hazards, and a renderer-only selection state machine that turns clicks into whole core actions, driven entirely by availableActions. Full interaction grammar, no polish (see Gaps).

Files

  • src/game/selection.ts new — the transient selection state machine (IDLE → SELECTED → resolve TargetSpec → COMMIT, with Cancel/Esc from any non-idle state). Holds only renderer state; never enters GameState. Emits a finished Action on commit (targeting design's decision 1 & 3).
  • src/game/render.ts rewrite — layout + factories for a player card vs a world card (visually distinct), a Hazard panel showing cost, current turn-Progress, keywords, penalty, reward; HP; current act + pile counts; the End Turn button; the world-discard zone; a modal chooser; a multi-select counter + Confirm; Cancel; and win/loss screens. Target previews render as plain-text badges (Progress incl. keyword bonus).
  • src/game/TableScene.ts rewrite — owns one GameCore; on each repaint reads state + availableActions(state) and paints affordances (playable, unplayable-dim, discardable, the Door's locked marker). Routes selection-machine commits to dispatch, then re-reads and repaints. End Turn is disabled while a selection is active; destructive actions (discard, Regroup destroy) are always two-step (targeting design's grammar & safety rules; REQ-WDS-27).
  • src/game/main.ts — likely unchanged (bootstrap); adjust scene wiring only if needed.

Notes

  • The renderer is the sole importer of phaser; the core lint boundary guarantees the rules never leak in. Highlights are always a function of current state, re-read after every dispatch (never cached across one).
  • This step is large; consider a sub-agent for the render.ts factories while the scene wiring is built, then integrate. Event-log animation/juice is allowed but not required; for this slice the renderer may snap to state after each dispatch.
  • The intensity signal exists (Step 7) but the renderer does not yet scale FX from it (deferred).

◆ Review & delegation strategy

  • Core is built sequentially by one implementer (steps 1–7). The modules share one tightly-coupled type surface; parallel sub-agents would fight over types.ts. Cohesion beats fan-out here.
  • Fresh-context review at two gates: after Step 6 (the reducer — the seam everything rests on) and after Step 9 (the renderer, which is the largest single surface). These are the highest-blast-radius points.
  • Step 8 (sim) and Step 9 (renderer) run in parallel once the core is frozen at Step 6: both only consume the finished core, and a divergence between them is exactly what the shared availableActions selector is meant to prevent.
  • Within Step 9, the render.ts factories can be delegated to a sub-agent while the scene wiring and selection machine are built, then integrated — the one place fan-out helps.

⚠ Risks & watch-points

  • Id minting must live in state. Any Date.now()/Math.random()/module-level counter for new card ids silently breaks replay equivalence. The nextId-in-state scheme is the guard; treat a replay failure as an id-source hunt first.
  • Rules drift between dispatch and the selector. The architecture names "two paths doing the same thing differently" as a bug class. reduce must validate through availableActions, not with a hand-rolled second check.
  • Mid-turn act advancement is easy to miss. Sprint's draw branch and Panic can empty the world pile mid-turn; both must reuse the same drawWorld/advanceAct path as EndTurn. Test mid-turn advancement explicitly.
  • Barricade compound ordering. Deal Progress in step 1 before the return step; if that deal resolves the targeted Hazard, it must be gone from the return options. The single committed PlayCard carries both targetId and returnIds; the core applies them in order.
  • Hidden mutation breaks determinism. Any in-place array/object edit in the core corrupts replay. readonly fields + the replay and id-accounting tests are the guard.
  • The Walker is intentionally unbeatable on the starter deck. The golden win path must survive via the forced Door (discard Walker → Door penalty → pay 2), never by clearing the 10-cost Walker. Don't "fix" this as a bug (REQ-WDS-22).
  • bun test must not pull in Phaser/DOM. Core tests import only from src/core/; the boundary lint plus a no-Phaser test environment catch a leak immediately.
