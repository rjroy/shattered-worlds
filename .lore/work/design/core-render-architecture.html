Design · Core Architecture

The core / render split

Approved · 2026-06-02 Scope: engine commitment + the seam only. POC feature set is a separate spec.

TypeScript + Phaser, with a pure-TypeScript rules core behind a thin Phaser renderer. The core is a synchronous, deterministic, seedable state machine with zero Phaser imports. The renderer is the sole owner of the clock: it turns player input into core actions, and turns the core's results into animation.

  • The core is a synchronous function, not a live actor. dispatch(action) returns { state, events } immediately. It never waits on animation, never sleeps, never touches the screen. This is what makes it exhaustively unit-testable and runnable at full speed for balance sims.
  • The renderer owns time. It plays the returned events as an animation timeline at human pace, then settles onto the authoritative state. Simulation speed and animation speed are decoupled by design.
  • All randomness lives in the core, seeded. Same seed + same actions yields the same run, byte for byte. The renderer's only randomness is cosmetic (particle jitter) and never feeds back into state.
  • The boundary is lint-enforced, not just a convention. An import rule forbids core/ from importing Phaser or anything in game/. The architecture fails the build the moment it's violated.
  • Escalation is a derived signal, not hand-scripted. The core exposes an intensity value computed from run state; the renderer reads it to dial the juice from mundane to insane. The curve is design; the seam for it is decided here.

This document commits the architecture the prior engine research pointed at, and resolves the one decision that research deferred: how the pure core and the renderer talk to each other. It does not define cards, effects, Destiny mechanics, or the POC scope. Those are spec work.

▦ What lives on which side

The split follows one rule: if it decides what is true in the game, it's core; if it decides how that truth looks or feels, it's renderer. Roughly 70% of a deckbuilder, and the 70% the vision cares most about, is core.

Pure core · src/core/ · no Phaser

  • Domain model — cards, effects, enemies, relics, world definitions, run & Destiny state. Plain immutable data.
  • Rules / resolution — applying an action to a state to produce the next state; win/loss; world transitions.
  • Seedable RNG — the single source of all game randomness.
  • Derived signals — intensity/threat and other read-models computed purely from state.
  • Public contract — the dispatch / read / event surface the renderer (and the sim runner) consume.

Phaser renderer · src/game/ · imports core

  • Scenes & presentation — sprites, layout, fonts, the visible board.
  • Juice — tweens, particle emitters, screen shake, audio, the escalation FX.
  • Input — pointer/keyboard, translated into core actions.
  • The clock — timeline playback of core events; pacing; interruption/skip.
  • Cosmetic randomness — jitter and flourish that never touches state.
→ grey intent flowing into the core (actions) → green results flowing out (state + event log) ┊ dashed the one-way import boundary

⚙ The seam contract

The entire interface between the two halves is small enough to state on one screen. The renderer never reaches inside the core; it sends actions and consumes results.

// src/core/contract.ts — the whole seam

type GameState = /* deep-immutable snapshot: board, deck, run, destiny, rng cursor */
type Action    = /* a player or system intent: PlayCard, EndTurn, ChooseReward… */
type GameEvent = /* a semantic thing that happened: CardPlayed, DamageDealt, CardDestroyed… */

interface GameCore {
  readonly state: GameState
  dispatch(action: Action): { state: GameState; events: GameEvent[] }
  // pure: same (state, action) always yields the same (state, events).
  // the seed lives inside state, so RNG advances deterministically.
}

function createGame(seed: number, config: GameConfig): GameCore

Two things make this contract do all the work:

Actions are intents, events are history. An Action is what someone wants (play this card). The returned GameEvent[] is the ordered list of what actually happened as a result (card left hand, 6 damage dealt, enemy died, relic triggered). The renderer reads that list as an animation script. It already knows the order, the targets, and the magnitudes, so the juice has everything it needs without re-deriving it from a before/after diff.

The core hands back the final state too. The renderer animates the events, but it doesn't have to be correct about the end result, the core already is. If the player skips the animation, the renderer just snaps to state. The authoritative truth and the animation script ship together from a single call.

◆ Why this shape, and not the obvious alternatives

The split itself was settled in research. The open question was the communication style across the seam. Three were on the table.

Seam style How the renderer learns what happened Determinism / headless Fit for maximalist juice Verdict
A · State snapshots only Diffs previous vs current state and infers the animation. Clean Weak — must reverse-engineer cause from effect Rejected
B · Snapshot + event log (chosen) Reads an explicit ordered list of semantic events, plus the final state. Clean — events ignorable Strong — the log is the animation script Chosen
C · Renderer callbacks into core Core invokes renderer hooks mid-resolution, awaiting animation. Broken — core now knows about time Tempting but couples the clock Rejected

A is too thin for this game. A diff tells you the enemy lost 6 HP; it doesn't tell you whether that was one hit or three, which relic fired, or in what order. For a restrained UI that might be fine. For mundane-to-insane escalation, where the sequence and source of effects is exactly what the juice dramatizes, the renderer would be reconstructing a story the core already knew and threw away.

C is the trap. Letting the core call back into the renderer (so it can pause for an animation before the next effect) feels natural and quietly destroys everything valuable about the core. The moment the core awaits the renderer, it is no longer synchronous, no longer pure, no longer runnable at full speed for balance data, and no longer trivially testable. The clock leaks into the rules. Principle 6's data-driven balance depends on simulating thousands of runs headlessly, which is impossible if a single resolution blocks on a tween.

The decision that pays for itself: one core, two drivers

Because the core is synchronous and ignores the event log unless you want it, the exact same core powers both the played game and the balance simulator. The Phaser renderer plays events at human pace; the headless runner discards them and dispatches as fast as the CPU allows, recording pick and win rates. The vision asks for instrumented, data-justified balance (Principle 6) and a prototype that proves fun before deep engineering (Principle 7). A synchronous core is the single architectural choice that serves both, from one codebase, with no rendering in the loop.

⚄ Determinism, RNG, and replay

All randomness flows through a seedable PRNG whose cursor lives inside GameState. This is not just hygiene; it's a vision requirement. Principle 4 says randomness must be owned: a bad outcome must trace to a choice, never to a blind deal. A seeded core makes that auditable, because a run is fully reconstructable from (seed, action log). That same property gives near-free replays, deterministic bug reports (attach the seed and actions), and golden-master tests that pin a whole run's outcome.

The renderer is allowed cosmetic randomness (where a spark flies, how a card wobbles) precisely because it provably cannot affect the game: it never calls dispatch with anything derived from it. The boundary keeps the two kinds of randomness from ever mixing.

✲ The escalation seam

The "mundane to insane" feel is captured architecturally without designing the curve here. The core exposes an intensity read-model, a pure function of run state (threat, stakes, how deep into a world the player is). The renderer reads it and scales the juice: more particles, harder shake, louder mix as it climbs.

This keeps escalation data-driven rather than hand-scripted per moment, which is the same discipline Principle 6 wants for balance, and it means the renderer never invents drama the rules don't justify. What intensity is computed from, and how the FX map onto it, is design work for later. The only commitment now: it is a derived core signal the renderer consumes, not a renderer-side guess.

⊘ Enforcing the boundary

A boundary that lives only in discipline erodes. This one is mechanical:

  • Directory split. src/core/ (pure) and src/game/ (Phaser). The core has no dependency on the engine in its package.json/import graph.
  • Lint rule. An import-boundary rule (e.g. eslint-plugin-boundaries or no-restricted-imports) fails the build if anything under core/ imports phaser or game/. One-way only.
  • Test shape proves it. The core's tests run with no Phaser, no DOM, no canvas. If a core test ever needs the engine to run, the boundary has already been breached and the test suite says so.
  • CI gate. Lint + core unit tests run headless in CI on every change, satisfying the typed-tested-CI half of the portfolio bar.

This is also the reversibility insurance the engine research called for: with zero Phaser in the core, swapping or supplementing the renderer later (a Pixi FX layer, a different shell) is a presentation change, never a rewrite of the game.

▣ Scope boundary of this design

Decided here

  • TypeScript + Phaser commitment.
  • Pure synchronous core / thin renderer split.
  • Seam = dispatch(action) → {state, events}.
  • RNG in core, seeded, replayable.
  • Renderer owns the clock; one core drives both game and sim.
  • intensity as a derived core signal.
  • Lint-enforced, CI-gated boundary.

Deferred to spec / later design

  • Concrete Card / Effect / GameState schemas.
  • Destiny, worlds, and run-economy mechanics.
  • The POC feature set and its success criteria.
  • What intensity is computed from, and the FX mapping.
  • Save format, content authoring pipeline, asset plan.
  • Exact PRNG algorithm and serialization format.

? Open questions for the POC spec to resolve

  • Event granularity. How fine do GameEvents go? Too coarse starves the juice; too fine couples the renderer to rules internals. The POC is where we find the right grain against real animation needs.
  • State delivery: full snapshot vs structural sharing. Returning a fresh immutable state per dispatch is simplest and most testable. If profiling later shows churn, structural sharing (Immer-style) is a drop-in, but we do not pre-optimize.
  • Single repo vs two packages. Start as one repo with two directories and a lint boundary. Promote core/ to its own package only if a second consumer makes it worth the overhead.
  • How the sim runner chooses actions. Random-legal is enough to exercise determinism and surface crashes for the POC; smarter policies for balance tuning come when there's balance to tune.
