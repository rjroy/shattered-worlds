Targeting & card selection

How a click becomes a fully-formed core action: selecting cards, knowing what is targetable, committing, and undoing. One interaction grammar for every card in the slice.

Draft · 2026-06-03

Three decisions carry this design:

  1. Selection is renderer-only state. The half-built act of "I picked Explore, now I'm choosing a target" lives entirely in src/game/. The core never sees a partial selection; it only ever receives a complete Action (PlayCard{cardId, targetId, choice, returnIds…} or DiscardHazard{cardId}). The seam stays pure and synchronous.
  2. Legality and interaction shape come from one pure core selector, not the renderer. The core exposes availableActions(state): which cards are playable, what each one needs (a TargetSpec), and which Hazards are discardable. The renderer projects that truth into highlights, so the UI can never offer something dispatch would reject, and the rules of targeting live in exactly one place. The headless sim (REQ-WDS-29) builds its legal-action policy from the same selector.
  3. One grammar: select → resolve requirements by highlight → commit, with Cancel always present. Every card follows the same shape. What differs per card is only its TargetSpec, which the renderer reads and drives generically. Adding a card later adds no new interaction code if its spec is a known shape.

⚙ The constraint that shapes everything

The approved architecture forbids the core from knowing about time, input, or the screen, and warns specifically against two code paths doing the same thing differently. Targeting sits right on that fault line: it is input (renderer) that must obey rules (core). The resolution is that the renderer holds the transient selection state and the core holds the rules of what is selectable. The renderer asks the core "what is legal from here?" and renders the answer; when the player commits, the renderer hands back a whole action. Nothing about a mid-selection ever enters GameState.

◆ Where does "what can I target" come from?

This is the load-bearing decision. The highlights must agree with what dispatch will accept. Two ways to achieve that:

The core exports a pure read-model describing the legal moves from a state. The renderer highlights from it and never re-implements targeting rules.

// src/core/ — pure, synchronous, no Phaser

type TargetSpec =
  | { kind: 'none' }                                  // Med Kit, Summon Door, Sprint(draw)
  | { kind: 'hazard' }                                // Explore, Listen, Baseball Bat
  | { kind: 'hazard', tag: 'Slow' }                   // Sprint(hit) — gated branch
  | { kind: 'modal', branches: Branch[] }           // Sprint
  | { kind: 'returnWorld', min: number, max: number } // Barricade 0..2, Panic 1..1
  | { kind: 'destroyHand', min: 0, max: 1 }         // Regroup
  | { kind: 'discardPlayer' }                         // Adrenaline's discard step
  | { kind: 'compound', steps: TargetSpec[] }       // Barricade = hazard, then returnWorld

interface AvailableActions {
  playable:    { cardId: CardId; spec: TargetSpec }[]
  discardable: CardId[]        // world cards with discardable=true
  legalTargets(cardId: CardId, step: number): CardId[]
  canEndTurn:  boolean
}

function availableActions(state: GameState): AvailableActions

Cost: a small new pure surface in the core and its tests. Benefit: the renderer and the sim share one definition of "legal," the highlights are provably correct, and a new card is data, not renderer code.

The renderer reads GameState plus card definitions and works out legality itself (this card is a DealProgress, so any world card in hand is a target; this one needs a Slow tag; the Door isn't discardable…).

It works, and it avoids touching the core. But it copies the targeting rules into src/game/, where they will drift from the core's dispatch validation. The architecture explicitly calls that class of duplication a bug source, and the sim runner would then need a third copy to choose legal actions. Rejected.

Why this is worth a core change

REQ-WDS-8 already says dispatch throws on an illegal target, and REQ-WDS-29 needs the sim to only ever choose legal actions. Both are the same question — "what is legal from here?" — and so is highlighting. Answering it once, purely, in the core, serves the renderer, the validator, and the sim from a single source. This is the same discipline as the architecture's derived intensity signal.

▦ The interaction grammar

Every interaction is the same four beats. The card's TargetSpec decides which middle beats appear.

→ enter selection → resolve & commit → abort (no dispatch)

◇ Selecting a player card vs a world card

Click a player card

It lifts and glows; the Cancel button appears. The renderer reads its TargetSpec and runs the matching flow: legal Hazards glow for a hazard spec, a small two-button chooser appears for a modal, hand cards glow for destroyHand, and so on. Committing means clicking a glowing target (or picking a mode, or pressing Confirm). The action is dispatched whole. An unplayable player card (no legal target exists, e.g. Explore with no Hazard in hand) is rendered dim and is not selectable at all, because availableActions omits it.

Click a world card (a Hazard)

A world card plays two roles depending on state. While nothing is selected, clicking it means "I want to deal with this": it becomes selected, Cancel appears, and the world-discard zone highlights as the single commit target. Clicking that zone discards the Hazard and applies its penalty. While a player card is already selected, the same world card is instead a target: if it is legal it glows, and clicking it commits the player card against it. Resolution is never a click on the Hazard itself; you resolve a Hazard by pouring enough Progress into it from the player side. The Door (discardable = false) shows no discard affordance and offers no discard flow; it can only be a Progress target or be held.

▧ The UX changes this requires

  • Discard becomes a zone, not a count. The POC's text pile-counts are replaced by real interactive regions. The world-discard zone is the commit target when a Hazard is selected for discard. The player-discard pile becomes a first-class region too: it is the visual destination for end-of-turn discards and Adrenaline's discard, and the source that visibly reshuffles into the draw pile. (Asymmetry worth stating: world cards can be discarded by player choice, so their zone is an active commit target; player cards have no free discard action, so their pile is a destination and animation anchor, not a general drop target.)
  • A Cancel button appears whenever a selection is active and disappears in IDLE. Escape and a click on empty table also cancel. Re-clicking the selected card cancels (toggle). Cancel never dispatches anything, so the core state is untouched.
  • A Confirm button appears for variable-count steps (Barricade return 0..2, Regroup destroy 0..1) because zero is valid and the player needs a way to say "done." Exact-count steps (Panic's single return, a single Hazard target) auto-commit on the click and show no Confirm.
  • A live counter (0/2) shows during multi-select so the player tracks how many they have toggled.
  • Target previews render the resulting Progress on each legal Hazard before commit, including keyword bonuses and whether the deal would resolve it. This is what makes "what can I do" legible without trial and error.
  • End Turn is disabled while a selection is active. You must commit or cancel first. This prevents ending a turn with a half-built action and prevents a stray End-Turn click during targeting.

⛉ Accidental-action prevention & touch parity

No destructive single click. The two actions that hurt — discarding a Hazard (eats a penalty) and Regroup destroy (permanent) — are never one tap. Both require select, then a deliberate second action on a separate target (the discard zone, or a glowing card), with Cancel available throughout. Resolving and dealing Progress are non-destructive, so they stay fast.

Touch and mouse are the same grammar. Tap = click; there is no reliance on hover. Affordances (playable shimmer, discardable marker, the Door's lock) are always rendered, not hover-revealed, so a touch player sees the same legibility. Target previews show as persistent badges on the cards, not hover tooltips; a tap-and-hold on a card may surface its full text for detail. Hit areas are sized for a fingertip, and the Cancel/Confirm controls anchor to a fixed spot so they do not jump under the thumb.

? Edge cases the grammar already answers

  • No legal target for a card → the card is unplayable and unselectable (dim). The player never reaches a dead-end selection, because availableActions filtered it out.
  • Mid-turn draw changes legality (Sprint's draw mode, Panic) → after the committed action animates, the renderer re-reads availableActions(state) and re-paints affordances. Highlights are always a function of current state, never cached across a dispatch.
  • The Door is held for turns → it simply never offers a discard or a self-action; it sits in hand as a clog and a target until paid. Legal and expected.
  • A commit would be illegal due to a race → it can't, because the renderer only ever offers targets the core selector returned for the current state, and dispatch is synchronous between repaints. If a dispatch ever did throw, that is a bug surfaced loudly, not swallowed.

✓ Decision

Build targeting as a renderer-side selection state machine driven by a pure core availableActions(state) selector that returns each playable card's TargetSpec and the discardable Hazards. The renderer runs one grammar — select → resolve the spec by highlighting → commit, Cancel always available — and hands the core a single complete action. Discard piles become interactive zones; the world-discard zone is the commit target for a Hazard discard; destructive actions are always two-step; the interaction is identical on touch and mouse.

This adds a small pure surface to the core (the selector and TargetSpec), which is worth reflecting in the implementation plan: it is the shared source of legality for the renderer highlights, the dispatch validation, and the sim's action policy. It does not change any rule already in world-deck-slice; it formalizes how the renderer reads them.

⋯ Deferred (not this slice)

  • Drag-to-target as an alternative to click-target. The click grammar is primary and sufficient; drag can layer on later without changing the core seam.
  • Keyboard-only play (number keys to select, arrows to target). The grammar supports it; binding it is polish.
  • Juice on highlight/commit (the intensity-scaled FX). Owned by the renderer, designed later.
  • Undo of a committed action. Out of scope; Cancel covers pre-commit only. A committed action is final for this slice.
