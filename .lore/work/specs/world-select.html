World-select screen

Spec · 2026-06-08 · status: draft
Related: brainstorm, vision

A pre-run screen that replaces the current random world assignment in BootScene. The player sees all available worlds, reads a short mood sketch for each, and chooses their shard before any cards are dealt.

Scene architecture

REQ-WS-1 WorldSelectScene is added to the Phaser game and sits between Boot and Table Must

A WorldSelectScene class is registered in the Phaser config alongside BootScene and TableScene. BootScene.create() launches WorldSelectScene unconditionally — it no longer picks a world or generates a seed. After this change, BootScene's sole responsibility is this.scene.start('WorldSelect'); it retains no world selection or seed logic. It may be removed in a later cleanup pass.

The scene key is 'WorldSelect'.

REQ-WS-2 TableScene interface is unchanged Must

TableScene.init continues to accept { worldId: string, seed: number }. No changes to its public interface. The caller changes (WorldSelectScene instead of BootScene) but the contract does not.

REQ-WS-3 WorldSelectScene launches TableScene with the chosen worldId and a fresh seed Must

When the player confirms a selection, WorldSelectScene calls this.scene.start('Table', { worldId, seed }) where seed is generated as Math.floor(Math.random() * 2 ** 32) — matching the formula used in BootScene today. Each selection produces a fresh seed; there is no seed persistence or display in this version.

Display data model

The existing worldManifest.ts (catalog builders and WorldData) is not changed. A sibling file carries display-only metadata.

REQ-WS-4 A WorldDisplayData type and worldDisplayManifest are defined in src/data/ Must

A new file src/data/worldDisplayManifest.ts exports:

export interface WorldDisplayData {
  name:    string   // Display name, e.g. "The Big Box"
  tagline: string   // One punchy line, e.g. "Corporate hell meets the undead."
  story:   string   // 2–4 sentences of mood. No mechanical explanation.
}

export const worldDisplayManifest: Record<string, WorldDisplayData>

This is the single authoring source for all world UI text. Neither TableScene nor any game-logic module imports it.

The scene should guard the lookup at runtime: if worldDisplayManifest[worldId] is undefined, throw an error immediately (e.g. "No display entry for worldId: <id>"). This surfaces authoring omissions at scene load, not as a silent crash mid-render. The manifest completeness test (REQ-WS-6) is the primary guard; this throw is a belt-and-suspenders fallback.

REQ-WS-5 All three worlds have display manifest entries Must

Entries exist for 'zombie-big-box', 'bird-building', and 'highway-volcano'. All three are selectable with no locked state in this version.

The zombie-big-box entry conveys: big-box store setting, Walker's arrival signalled by environmental dread (flickering lights, strange sounds), zombie invasion. Tone is mood-setting, not tutorial.

Bird-building and highway-volcano entries may use placeholder story text if full lore text isn't written yet. Placeholder text must be a non-empty string (e.g. 'Coming soon…') — an empty string would fail the completeness test in REQ-WS-6. Name and tagline must be real.

REQ-WS-6 A test verifies manifest completeness and shape Must

A unit test confirms:

  • Every worldId key in Object.keys(worldManifest) has a corresponding entry in worldDisplayManifest. The test iterates keys only — it must not invoke the builder functions (calling them throws CatalogError in the test environment because the Phaser asset cache is absent).
  • Each entry satisfies the WorldDisplayData shape (all fields present and non-empty strings).

This test guards against a new world being added to the game manifest without a display entry, which would cause a runtime crash on world-select.

Visual layout

Canvas is 900 × 600. The world-select screen has its own visual identity (dark, pre-game) and does not use any world's VisualTheme for the screen itself — each world card uses its theme's accent color as a tint only.

REQ-WS-7 The screen has a title and subtitle in the header region Must

The upper portion of the screen (above the world cards) shows the game title "SHATTERED WORLDS" and a subtitle such as "Choose your shard". The title uses a large, bold font; the subtitle is smaller and muted. Both are legible against the dark background.

REQ-WS-8 One interactive card per world, arranged horizontally and centered Must

Three world cards are arranged in a single horizontal row, centered on the 900px canvas. Each card is a Phaser.GameObjects.Container containing a background rectangle and text objects. Cards are equally sized and spaced.

Approximate layout: cards ~240px wide, ~350px tall, ~30px gap, ~60px side margins.

REQ-WS-9 Each card displays name, tagline, and story text Must

From top to bottom within each card:

  • Name — world display name, larger bold text
  • Tagline — one line, smaller, styled distinctly (italic or muted color)
  • Story — 2–4 sentence paragraph, smaller text, word-wrapped within the card width

All text content is sourced from worldDisplayManifest.

REQ-WS-10 Card accent color comes from VisualTheme; no backdrop textures are loaded Must

Each card's background uses a dark solid fill with an accent border or tint drawn from theme.intrusionHue — the world's keynote color (a CSS hex string on VisualTheme, e.g. '#9bff6a' for zombie-big-box). Convert it to a Phaser hex integer for rectangle/graphics calls using Phaser.Display.Color.HexStringToColor(intrusionHue).color.

No backdrop or card texture images are loaded by WorldSelectScene. The scene is self-contained with no preload() requirements — all theme data is available synchronously from TypeScript imports.

Interaction

REQ-WS-11 Hovering a card produces a visible emphasis Must

On pointerover, the card receives a visible emphasis: a scale increase (e.g. setScale(1.03)), a brighter border, or both. Scale alone is sufficient. On pointerout, the card returns to its rest state. This provides pointer affordance before committing to a click.

REQ-WS-12 Clicking a card starts the run in that world Must

On pointerdown (or pointerup), the scene calls this.scene.start('Table', { worldId, seed }) with the card's worldId and a freshly generated seed. Any additional click feedback (brief scale-down, flash) is acceptable but not required.

Not in scope

Backdrop thumbnails — Using backdrop textures as card backgrounds. This would require preloading all backdrop images in WorldSelectScene (or an earlier scene). Deferred; theme colors are sufficient for v1.
Animated card reveal / flip transitions — Cards appearing with a stagger or flip animation on scene entry. Deferred.
Back navigation from in-game to world-select — Win/loss screen returning to WorldSelectScene rather than dead-ending. Deferred to a future run-flow spec.
Locked / coming-soon world state — All three worlds are selectable in v1. A locked state is deferred until a formal availability policy is decided.
Seed input or "daily run" — Player-visible seed or shared seed for daily challenges. Deferred.
Keyboard / gamepad navigation — Pointer-only for now.
