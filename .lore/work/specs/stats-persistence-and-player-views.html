Stats persistence and player-facing views

Run and lifetime statistics already accumulate from the gameplay event stream and persist to localStorage (src/game/runtime/runStats.ts, v1). This spec defines the next layer: the v2 stats schema (per-world bests, abandon semantics, active-time duration, run history, migration), and the first player-facing views of that data — an end-of-run summary, per-world badges on the world-select carousel, and a Chronicle scene with export/import.

Lineage. This spec claims the [STUB: run-stats-collection] exit point of the gameplay event stream spec; REQ-EVENTS-12 explicitly delegates aggregation formulas, storage schema, and migration strategy here. It also claims the world-select spec's deferred item "win/loss screen returning to WorldSelectScene." All decisions trace to the resolved brainstorm stats-persistence-and-player-views. Outcome vocabulary (won | lost | abandoned), single-source-of-truth, and subscriber-determinism rules are owned by REQ-EVENTS-8/10 and are referenced, not restated.
Stance cutover. Twice before, incompatible stored payloads were deliberately discarded as acceptable pre-release behavior (see telemetry-gaps note). That precedent ends with this spec: from v2 onward, any valid older-version payload must migrate forward. Discard remains correct only for malformed or unrecognized data.

Out of scope

  • Suspended-run save/resume — own design pass (stale-save policy pre-decided: invalidate on content-version mismatch).
  • Run-history browser UI, win streaks, fastest-win best — history is recorded (REQ-STATS-7) but nothing reads it yet.
  • New event types (heals, hazard destroy/exile tallies, energy spent — see telemetry-gaps); bests are defined against currently recorded values as-is.
  • Destiny unlock mechanics — the Chronicle is the proto-Destiny surface only; it grants nothing mechanical (vision anti-goal: no grind-past-the-ceiling).
  • Cloud sync, accounts, multiple profiles, seed sharing.

Schema and persistence (v2)

REQ-STATS-1 — v2 lifetime shape with per-world bests
LifetimeStats version 2 retains all v1 lifetime counters and extends each byWorld entry with two optional bests, absent until earned: fewestTurnsWin (lowest turns among won runs on that world) and mostProgressInRun (highest progressDealt in a single completed run on that world; see the completed-run definition in REQ-STATS-2). Bests update at fold time when a qualifying run beats or first sets the record.
Persisted-bests decision (brainstorm Q5). Known caveats accepted: turns undercount a final partial turn; progress includes overkill. Both apply uniformly, so comparisons stay fair.
REQ-STATS-2 — abandons count outcomes only
Definition used throughout this spec: a "completed run" is one with outcome won or lost; abandoned runs are not completed. From v2 onward, an abandoned run increments outcome tallies (lifetime and per-world runs/abandoned) but does not fold into activity counters (turns, cardsPlayed, progressDealt, damageTaken, hazards, discards) or play-time totals. Abandoned runs never set bests. lastRun and run history (REQ-STATS-7) still record abandoned runs in full.
Resolves the dilution issue flagged in the telemetry-gaps audit: lifetime activity totals and any future averages mean "per completed run."
REQ-STATS-3 — v1→v2 migration, lossless
Loading a payload that validates as v1 migrates it to v2 with no data loss: all counters and byWorld outcome tallies carry over, bests start absent, durationMs carries over as-is (it is historical wall-clock; see REQ-STATS-4), lastRun carries over. Key strategy: v2 persists under a new key (shattered-worlds/run-stats/v2); the exported RUN_STATS_STORAGE_KEY constant moves to the v2 value; after the first successful v2 persist, the v1 key is removed (the RunStatsStorage seam gains removeItem). Migrated activity counters may include pre-v2 abandoned-run activity that was folded under v1 rules; this is accepted as historical noise — the per-completed-run interpretation in REQ-STATS-2 applies only to counters accumulated under v2. Payloads that are malformed or claim an unknown version are still discarded with a console warning (unchanged v1 behavior).
Migration-at-first-schema-change decision (brainstorm Q1). This v2 is the migrate-don't-discard pattern's first customer. Key naming, old-key removal, and the legacy-noise stance are pinned so three implementers can't make three different data-loss choices.
REQ-STATS-4 — duration means active time
From v2 onward, recorded run duration is active time: the duration clock pauses while the document is hidden and resumes when visible. RunRecord gains activeDurationMs; lifetime play time accumulates activeDurationMs for completed runs. Wall-clock startedAt/endedAt timestamps remain on the record unchanged. Visibility is injected as an optional visibility?: () => boolean (true when the document is visible) plus a subscribe seam for change events on GameplayRuntimeOptions, parallel to clock, defaulting to the document visibility API in main.ts — tests never depend on real page visibility. The per-run clamp against backwards clock jumps is preserved.
Active-time decision (brainstorm Q2). Any screen labeled "time played" must be honest; a backgrounded tab must not accrue play time.
REQ-STATS-5 — abandon recording still works app-wide
The existing guarantees are preserved under v2: scene shutdown and pagehide still close open runs as abandoned exactly once, and those runs are recorded per REQ-STATS-2. Partial runs (collector attached mid-run) are still ignored rather than recorded understated.
Regression guard — the fold change must not disturb the close-exactly-once invariant from REQ-EVENTS-8.
REQ-STATS-6 — persistence seams and failure posture unchanged
All v2 persistence goes through the injected RunStatsStorage seam. Load failures and write failures remain non-fatal (warn, keep the in-memory copy). Validation of the v2 shape is as strict as v1's: a payload that does not fully check out is rejected wholesale, never partially folded.
The v1 hardening (garbage counters poisoning every future run) must survive the schema change.
REQ-STATS-7 — run history ring buffer
The most recent 100 RunRecords (all outcomes, including abandoned) persist across reloads, ordered most-recent-first, oldest evicted at the cap. History persists under its own key (shattered-worlds/run-history/v1), distinct from lifetime stats — that key separation is the isolation mechanism: a corrupted history payload is discarded with a warning while lifetime stats load normally, and vice versa. History load follows the same validate-or-discard posture as lifetime stats. No UI consumes history in this spec.
Can't backfill what wasn't recorded; this feeds the future history browser, streaks, and the Destiny brainstorm's Survivor's Journal / Feats lanes without a v3 rewrite.

End-of-run summary

REQ-STATS-8 — summary on every terminal outcome
When a run ends — won, lost, or abandoned via the in-game exit button — the bare win/loss overlays are replaced by a run summary showing: outcome, world name, turns, active duration, cards played, progress dealt, damage taken, hazards resolved, hazards discarded, and cards discarded. The exit affordance is the existing X button in TableScene; its behavior changes from jumping straight to WorldSelect to abandoning the run and showing the abandon summary first (dismiss then returns to WorldSelect per REQ-STATS-11). Tab-close abandons show nothing (no UI is possible at pagehide); they appear only in the Chronicle.
Full-recap-for-abandons decision (brainstorm Q3). The recap is the run's cooldown ritual. Naming the trigger matters: today the X button bypasses any end-of-run surface entirely.
REQ-STATS-9 — lifetime context and record callouts
The summary always shows the overall run number ("Run N" — well-defined from the very first run, where N = 1), and additionally shows the win count on this world when it is at least 1. If the run just set or beat a per-world best (REQ-STATS-1), the summary calls the new record out explicitly.
The single line that makes persistence felt; the record callout is the payoff for persisting bests at all.
REQ-STATS-10 — summary reads through RunStatsReader only
The summary sources its data from RunStatsReader (lifetime().lastRun plus lifetime/per-world fields). Views never write stats and never scrape renderer state. (Viable because RunEnded is emitted synchronously inside dispatch — lastRun is current before the overlay renders.)
Single source of truth per REQ-EVENTS-5/12; keeps the read seam the only coupling between stats and views.
REQ-STATS-11 — summary integrates with overlay conventions
The summary follows the established overlay conventions from the help-screen spec: centered container at overlay depth, full-canvas interactive background swallowing board input, hidden by default, the ? button hidden while shown. Dismissing the summary returns to WorldSelectScene — this closes the world-select spec's deferred "win/loss returns to WorldSelect" item. Its visual chrome uses the gold occult-cartography register (the Door/Destiny language per visual direction), not flat-design panels.
Amends existing conventions instead of inventing competing ones; the deferral chain from REQ-WS "Not in scope" closes here.

World-select badges

REQ-STATS-12 — per-world wins/attempts badge
Each world card on the world-select carousel shows that world's wins and attempts from byWorld (e.g., "2 / 7"). Worlds with zero recorded runs show no badge. Data arrives via RunStatsReader; the authored worldDisplayManifest is not involved (badges are live data, not authored text).
Converts the byWorld table into ambient motivation at zero navigation cost.
REQ-STATS-13 — badges respect the carousel and the color ration
Badges target the shipped shift-by-one carousel (not the superseded 3-card layout in the world-select spec): legible on the focused card, and on partial-opacity peek cards either legible or omitted entirely — never half-readable, defaulting to omit on peek cards when in doubt. Styling stays within the visual-direction color ration: at most the world's intrusionHue as a scarce accent, no new saturated chrome.
The carousel decision was never specced (new-world-concepts brainstorm); badges must not fight inset art per theme-authoring rules.

Chronicle scene

REQ-STATS-14 — dedicated scene with real navigation
A new ChronicleScene is reachable from a visible affordance on WorldSelectScene. Back navigation (on-screen control and Escape) returns to WorldSelect. Entering and leaving the Chronicle never disturbs run state or stats.
Dedicated-scene decision (brainstorm Q4): this is the surface that grows into the Destiny hub. Navigation between views is an implicit requirement from day one.
REQ-STATS-15 — Chronicle content
The Chronicle displays, read-only via RunStatsReader: lifetime totals (runs, wins, losses, abandons, turns, cards played, progress dealt, damage taken, hazards, play time as active time), a per-world breakdown (attempts, wins, losses, abandons, and earned bests from REQ-STATS-1), and the last run showing the same fields as the end-of-run summary (REQ-STATS-8 list; seed, appliedModifiers, and finalActIndex are not displayed). Durations render human-readable (no raw milliseconds).
The mirror: who has this Destiny been. Per-world bests are the rows that make it worth opening twice.
REQ-STATS-16 — honest empty state
With zero recorded runs the Chronicle renders a deliberate empty state in the Destiny voice — no zero-spam, no NaN, no empty table skeletons.
The first thing a new player who taps the shiny button sees; it must read as "nothing written yet," not "broken."
REQ-STATS-17 — Destiny register, zero mechanical surface
The Chronicle uses the gold occult-cartography linework register (visual direction Look 4) and is framed as the Destiny's memory. It grants, sells, and unlocks nothing; nothing in it reads as purchasable power. It is built so future Destiny content (journal, feats) can extend it without restructuring.
Vision anti-goals: no grind-past-the-ceiling, no clean flat-design UI. Proto-Destiny framing per brainstorm Q4 and the meta-progression brainstorm's "story persists" lane.
REQ-STATS-18 — export
From the Chronicle, the player can export their complete stored stats (lifetime v2 + run history) as a downloaded JSON file, exactly what import accepts.
Cheap durability backstop against script-writable storage eviction (Safari ITP); a debugging gift.
REQ-STATS-19 — import with full validation and confirm
From the Chronicle, the player can import a previously exported file. The file is validated with the same strictness as a storage load (including migration of older-version exports); an invalid file is rejected with a visible message and existing data is untouched. A valid import replaces stored stats wholesale only after an explicit in-UI confirmation that states current data will be overwritten; when the file is an older-version export, the confirmation additionally notes it will be upgraded to the current format. Migration happens only after confirmation, never before.
The only operation in this feature that can destroy data — it gets the only confirmation dialog.

Boundaries and quality gates

REQ-STATS-20 — core untouched, logic Phaser-free
This feature changes nothing in src/core/. All schema, migration, fold, bests, active-time, history, and export/import logic lives in src/game/ as pure TypeScript with no Phaser imports, unit-testable headless. Stats subscribers must not affect core determinism (REQ-EVENTS-10) and stats never feed back into game rules (consistent with REQ-VIS-1's renderer-only posture).
The lint-enforced core/game boundary; the deterministic core is the project's load-bearing invariant.
REQ-STATS-21 — tests ship with the implementation
Unit tests cover, at minimum: v1→v2 migration from a stored-v1 fixture (lossless carry, bests absent); malformed and unknown-version discard; abandon fold exclusion (outcome counts increment, activity counters do not, bests unset); bests set/beat/not-beaten transitions for both record types; ring buffer cap, ordering, and eviction at 100; history failure isolated from lifetime load; active-time clock under fake visibility transitions (hidden time excluded, backwards-clock clamp held); export/import round-trip; import rejection leaving stored data untouched. View wiring is covered by scene-level tests using the real collector (never mocked core), consistent with existing test conventions.
Tests alongside implementation, not after; migration and fold changes are exactly the kind of silent-corruption surface tests exist for.

AI Validation

How the AI verifies this spec is done. All commands from repo root.

Automated

  • bun test passes, including every test enumerated in REQ-STATS-21 (verify each scenario exists by name/assertion, not just that the suite is green).
  • bun run lint passes — confirms the core/game boundary (REQ-STATS-20) and that no Phaser import appears in the new runtime logic modules.
  • Grep checks: no scene or view module imports RunStatsCollector or calls fold/persist functions directly — scenes touch stats only through RunStatsReader-typed access (indirect writes via session.abandon() firing the subscriber are the designed path and are fine) (REQ-STATS-10); no worldDisplayManifest import in badge logic (REQ-STATS-12); no file under src/core/ imports anything from src/game/ — stats never feed back into game rules (REQ-STATS-20).
  • Migration proof: a unit test loads a verbatim v1 payload fixture (captured from the current shipped shape) and asserts the v2 result field-by-field (REQ-STATS-3).

Manual browser checks (required — visual requirements are not test-provable)

  • Win a run, lose a run, and exit a run: all three show the summary with the REQ-STATS-8 fields, a lifetime context line, and dismiss to WorldSelect. The ? button is hidden while the summary is up (REQ-STATS-11).
  • Set a record (first win on a fresh world) and confirm the new-record callout (REQ-STATS-9).
  • WorldSelect: badge shows wins/attempts on played worlds, absent on unplayed worlds, and is legible-or-omitted on carousel peek cards (REQ-STATS-12/13).
  • Chronicle: reachable from WorldSelect, Escape and back control both return, totals and per-world bests render with human-readable durations, fresh profile shows the empty state (REQ-STATS-14/15/16).
  • Pre-load localStorage with the v1 fixture under the v1 key, boot the app, play one run: stats appear intact, the v2 key now holds the migrated payload, and the v1 key is gone (REQ-STATS-3 in situ).
  • Export a file; wipe storage; import the file (confirming the overwrite prompt); stats restored. Then attempt importing a corrupted copy: visible rejection, data untouched (REQ-STATS-18/19).
  • Background the tab mid-run for ~a minute, finish the run, confirm the summary's duration excludes the hidden time (REQ-STATS-4).
