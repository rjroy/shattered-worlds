Spec · The Look, Made Real

Mundane reality, broken open

Turn the visual direction into renderer requirements: a per-world VisualTheme seam, card fronts that vary by world over a constant card back, a backdrop that slides toward intrusion as the core's intensity climbs, and The Walker rendered as a distant figure who steps to the foreground the turn you draw him. All presentation, layered on the proven core/render seam.

◇ Purpose & altitude

The world-deck slice made the game playable; this spec makes it look like itself. It builds the visual identity the direction sheet established, layered entirely on the existing Phaser renderer (src/game/) and the pure core it already consumes. Per the agreed altitude, it ships as a system plus seams: a per-world theming abstraction with exactly one concrete theme (Zombie Big-Box, the only world that exists), structured so additional worlds slot in without renderer surgery. It validates against the current slice.

Reality (mundane backdrop)→ intensity climbs → intrusion bleeds in→ Walker drifts nearer each act→ Draw The Walker → he's in your face→ the Door

The master gradient every reality maps onto, restated here because it is the load-bearing element:

REALITY · mundane · desaturatedINTRUSION · impossible · magenta → ember

This illustrates the master gradient shape, not the shipped theme's colors. Per the visual direction, the Zombie Big-Box theme's intrusion hue is fluorescent green (≈ #9bff6a → #39ff88), the toxic glow down the retail aisles. Each world claims its own intrusion hue over the same desaturated base.

Resolved decisions & assumptions (veto any of these)

  • One small core change, pure data only: a worldId string on game setup/state so the renderer knows which world is active. No colors, art, or presentation ever enter src/core/. (Assumed — the seam needs an identity to key on.)
  • Backdrop intrusion tracks intensity. The existing pure intensity(state) → [0,1] read-model drives how far the backdrop slides along the reality→intrusion ramp. Flavor only: it never feeds back into a dispatched action. (Confirmed.)
  • Walker proximity tracks actIndex, not intensity: far in Act 1, looming in Act 3. Two distinct signals on purpose. (Confirmed.)
  • The Walker is identified by his stable template identity (name === 'The Walker' in the existing catalog). "Drawn" means that world card is in state.hand. This is presentation of REQ-WDS-19/20, adding no rule. (From the world-deck slice.)
  • Card fronts vary by palette + frame treatment, not bespoke per-card art. The card back is constant canon (cardback.webp) across all worlds. (Confirmed.)
  • Source art for the Zombie Big-Box theme derives from the big-box panel of art-style.webp plus the existing card back. Producing final game-ready assets is real work but the art-authoring pipeline is out of scope; this spec requires the rendering system and that referenced assets resolve. (Assumed.)
  • Layout is inherited, not redesigned. The themed backdrop sits behind the existing world/hand rows and HUD from TableScene; this spec does not move the board. (Assumed.)
A · Scope & the boundary
REQ-VIS-1Visual identity is renderer-only; it adds no game rulesMust

Every artifact in this spec — themes, backdrops, the Walker presence, intrusion, escalation juice — lives under src/game/. It changes how the game looks, never what is true. No win/loss, card, effect, draw, or economy behavior is added or altered. The lint-enforced import boundary continues to forbid src/core/ from importing Phaser or src/game/, and the build fails if violated.

REQ-VIS-2Presentation reads from core; it never feeds backMust

The renderer derives all visuals from the core's existing surface: the immutable state, the pure intensity(state) read-model, and the events list from each dispatch. Nothing visual influences a dispatched Action: cosmetic randomness, theme choice, intrusion level, and the Walker's position never flow back into the core (the architecture's one-way rule). If the player skips animation, snapping to the authoritative state must yield an identical board.

B · World identity & the VisualTheme seam
REQ-VIS-3The core exposes a stable world identity as pure dataMust

Game setup carries a worldId (a plain string, e.g. 'zombie-big-box') readable from GameState. It is the single permitted core addition, and it is data, not presentation: it names the world, it carries no colors, art, or layout. createGame sets it for the shipped world. This is what lets a renderer-side registry pick a theme without the core knowing themes exist.

REQ-VIS-4A renderer-side VisualTheme descriptor, selected by worldIdMust

In src/game/, a VisualTheme descriptor captures a world's look, and a registry maps worldId → VisualTheme. The active theme is selected purely from state.worldId; an unknown id falls back to the zombie-big-box theme — the only shipped theme is the default — rather than crashing. The descriptor includes at least:

interface VisualTheme {
  worldId: string
  intrusionHue: string        // the world's signature intrusion color
  realityPalette: string[]    // the desaturated mundane base
  frameStyle: FrameStyle     // card-front frame / chrome treatment
  grain: GrainSpec           // distress / texture parameters
  backdrop: BackdropRefs      // reality + intrusion backdrop art layers
  walker?: WalkerArt          // present iff this world has a Walker
}

The concrete shape of the sub-types (FrameStyle, GrainSpec, BackdropRefs, WalkerArt) is a renderer implementation detail under src/game/; what this spec fixes is that they exist, are presentation-only, and are driven from the theme. The descriptor's backdrop and walker refs are also the authoritative source of asset paths (REQ-VIS-17). Exactly one theme ships (zombie-big-box); the seam is proven by one concrete theme cleanly driving the renderer, not by shipping many.

REQ-VIS-5The card back is constant canon, theme-independentMust

Every card back renders from cardback.webp regardless of world or theme. The back is the game's constant — the occult-cartography doorway — and is never recolored or re-themed per world. Only fronts vary (REQ-VIS-6).

C · Card fronts vary by world
REQ-VIS-6Card fronts are themed from the active VisualThemeMust

Card fronts draw their palette, intrusion hue, frame, and grain from the active theme, so a world's deck looks like the world you are standing in — the visual half of the vision's "worlds are identity, not wallpaper." The card layout and the self-describing face content are unchanged: theming retints and reframes; it does not move or rewrite the existing data-driven faces. Fronts vary by world; the back does not.

REQ-VIS-7Theming preserves card legibility and the player/world distinctionMust

Within a theme, player cards and world cards (Hazards) remain visually distinct (the slice's two-row, two-treatment separation survives). The theme tints, it never obscures: a Hazard's cost, keywords, penalty, and reward, and a player card's effect text, stay clearly readable against the themed front (vision Principle 5, legibility). Contrast is a pass/fail property, not a matter of taste: text-to-background contrast must meet WCAG 2.1 AA — at least 4.5:1 for body/effect text and 3:1 for large labels — measured against the busiest themed front. This is a floor for readability, independent of the distressed aesthetic; if a specific element cannot meet it without breaking the look, that is a finding to raise, not to silently ship under.

D · Backdrop & the reality→intrusion ramp
REQ-VIS-8A themed backdrop sits behind the boardMust

A backdrop layer renders behind the existing world/hand rows and HUD, expressing the active world's mundane reality base (the big-box store for the shipped theme). It is drawn from the theme's backdrop refs and never intercepts pointer input meant for cards or buttons.

REQ-VIS-9Backdrop intrusion is a pure mapping of intensity onto the rampMust

The backdrop's intrusion level is computed by a pure function of the core's intensity(state) ∈ [0,1], mapped onto the theme's reality→intrusion ramp: at low intensity the screen is desaturated and mundane; as intensity climbs (deeper act, lower HP, more held Hazards) it slides toward saturated intrusion in the world's signature hue. The transition is tweened across dispatches, not snapped. The tween is fire-and-forget: a new dispatch that changes the target intrusion level retargets the in-flight tween toward the new value rather than queueing or blocking, so the backdrop can never fall behind the authoritative state. This is renderer-only flavor (REQ-VIS-2): it dramatizes the state, it never alters it.

REQ-VIS-10Composition keeps the apocalypse localizedShould

Per the direction's load-bearing rule — apocalypse is localized, chaos has a radius — the backdrop preserves a visible boundary between the mundane and the intrusion even at high intensity; the board sits in the surviving-normal zone. The frame is never fully consumed edge-to-edge by catastrophe, because the contrast between normal and broken is the signature (a stated anti-goal in the visual direction).

E · The Walker presence
REQ-VIS-11The Walker is always present as a distant figure, nearer each actMust

For any world whose theme declares a walker (the shipped one does), a figure renders in the backdrop throughout play. His proximity is a pure function of state.actIndex: a tiny distant silhouette in Act 1 (actIndex 0), nearer in Act 2, looming at the edge in Act 3. He is the ambient dread of the run made literal, present long before he is ever in your hand. When actIndex advances (the ActAdvanced event), the figure tweens to the new act's proximity rather than snapping, so crossing into a deeper act reads as him closing in; this transition is suppressed while he is foregrounded (REQ-VIS-12), since the card in hand outranks the ambient position.

REQ-VIS-12Drawing The Walker collapses the distance to the foregroundMust

The moment the Walker world card (stable identity name === 'The Walker') enters the hand, the distant figure resolves to a full foreground presentation tied to that card — he is right in front of you. While the card is held in hand he stays foregrounded; once he leaves the hand (resolved or discarded per REQ-WDS-19/20), the presence returns to the act-appropriate backdrop state. Both exits share the same visual return by intent: the difference between resolving him (grants Summon Door) and discarding him (forces a Door) is carried entirely by the card mechanics, not by a distinct Walker animation. This is pure presentation of the existing mechanic; it introduces no new action, target, or rule.

REQ-VIS-13The Walker presence derives only from core state and eventsMust

Proximity reads state.actIndex; foregrounding reads hand membership and the dispatch events (e.g. CardsDrawn carrying the Walker's id, HazardResolved/HazardDiscarded removing it). The presence is never driven by a renderer-side timer or guess that could drift out of sync with the deck. The figure on screen and the card in the deck are always the same truth.

F · Escalation juice
REQ-VIS-14Additional juice scales from the same intensity signalShould

Beyond the backdrop, secondary flourish (grain density, particle activity, subtle shake on intrusion spikes) may scale from the same intensity read-model, realizing the architecture's escalation seam — mundane to insane. Escalation must be derived from the signal, not hand-scripted per moment, and like all visuals it never feeds back into state. Intensity of juice is a Should; correctness and legibility (REQ-VIS-7) outrank it.

G · Testing, boundary & build
REQ-VIS-15The pure visual mappers are unit-tested headlesslyMust

The decision logic is extracted into pure functions and tested with no Phaser/DOM/canvas: worldId → VisualTheme selection (including the unknown-id fallback), intensity → ramp position (clamped to the ramp ends, and monotonic non-decreasing — verified as a property, a ≤ b ⇒ ramp(a) ≤ ramp(b), over a few dozen random pairs, not a hand-picked sample of two or three), and actIndex → Walker proximity (the three act tiers). Phaser-touching glue stays a thin shell around these mappers, so the testable logic needs no engine to run (the project's dependency-injection testing discipline).

REQ-VIS-16The core stays free of presentationMust

The import-boundary lint passes. A grep of src/core/ finds no color, hue, theme, or Phaser reference, and the only addition there is the plain worldId string (REQ-VIS-3). All VisualTheme data and art references live under src/game/.

REQ-VIS-17Assets bundle and the build/deploy pipeline is unchangedMust

Theme art (the card back, backdrop layers, Walker art) bundles through the existing Bun + Vite toolchain; bun run build emits a self-contained dist/ on the correct Pages subpath base with no 404s on any asset, and CI gates lint + tests + build before deploy. No new pipeline, and the lockfile stays committed and reproducible. The VisualTheme descriptor is the authoritative source of asset paths (REQ-VIS-4): the 404 check tests exactly the paths the shipped theme references. The app has no asset directory today, so this spec establishes one under the renderer (e.g. src/game/assets/<worldId>/…, with the canonical card back at a shared path); the precise layout is an implementation decision, but it must be a committed convention future worlds follow, not an ad-hoc scatter.

REQ-VIS-18The identity is visibly correct in the browserMust

bun run dev serves the slice showing: themed card fronts over the constant card back; a backdrop that visibly slides toward intrusion as intensity rises (observable by advancing acts and taking damage); the Walker a distant figure in Act 1 who comes nearer by Act 3; and the Walker foregrounded whenever his card is in hand. The play loop from the world-deck slice still works unchanged underneath.

✕ Out of scope

Explicitly not in this spec: additional worlds or themes beyond the single Zombie Big-Box VisualTheme (the seam is proven by one); an art-authoring or asset-import pipeline, and the production of final art beyond what the theme references; per-card bespoke art (fronts vary by palette and frame, not by hand-drawn card illustration); audio and music; full animation polish beyond the specified backdrop tween, Walker foregrounding, and intensity-scaled juice; any change to game rules, the card catalog, or the turn loop (this spec is presentation only); and tuning the intensity curve itself, which belongs to the core. Generalizing the VisualTheme shape for worlds with radically different board structures is later design.
