Card data externalization & world-scoped templates

Date: 2026-06-04  |  Status: draft  |  Related: world-deck-slice, visual-identity

Problem Statement

The current card catalog is hardcoded as TypeScript objects in src/core/cards.ts. This has three concrete problems:

  1. Wrong abstraction layer. Card templates are content, not code. At hundreds of cards the TypeScript union of names and the exhaustiveness check become maintenance liabilities rather than safety nets.
  2. Single-world assumption. The catalog is a flat global list with no concept of which world a card belongs to. Zombie, Find Baseball Bat, and The Walker are Zombie Big-Box content, but the code treats them identically to starter cards.
  3. No card provenance. Cards gained as world rewards carry no record of where they came from. The renderer has no signal to apply world-appropriate card front art to a card that travels into future runs.
Locked decisions that constrain this spec
  • worldId: string on GameState is the active world and changes as the player traverses worlds. No art, colors, or presentation enters src/core/.
  • The VisualTheme registry in src/game/ already maps worldId → VisualTheme. Unknown worldId falls back to 'zombie-big-box', not a crash.
  • Card back (cardback.webp) is constant canon — never re-themed.
  • The preload() phase runs before create(). Multiple-world loading requires worldId passed as Phaser scene init data.
  • After this spec, mintCard takes an explicit catalog parameter: mintCard(catalog, state, templateId). It remains the only permitted id source — no ad-hoc card construction.

Requirements

Data Externalization
REQ-CARD-DATA-1 Card template data MUST be stored in JSON asset files, not in TypeScript source files. draft

No card name, cost, keyword list, effect, penalty, reward, or act composition array may be defined inline in .ts files. This includes the act specs currently hardcoded in src/core/world.ts. TypeScript files may reference template IDs and define schema types, but all data values live only in JSON.

REQ-CARD-DATA-2 All JSON required for the current run MUST be loaded via Phaser's asset loader before any card minting occurs. draft

"All JSON required for the current run" means starter.json and the active world's JSON (e.g., zombie-big-box.json). Loading happens in the preload phase. The game MUST NOT attempt to mint cards before both files have been loaded and parsed.

Attempting to mint from an unassembled or partial catalog is a runtime error (see REQ-CARD-DATA-14).

REQ-CARD-DATA-3 A failed JSON load MUST surface an error before any game state is created. The game MUST NOT proceed with a missing or partial catalog. draft

If starter.json or the world's JSON fails to load (missing file, network error, parse error), the game must not silently fall back to an empty catalog or partially constructed state. The minimum required behavior is an error message visible to the developer before the run begins. Graceful user-facing error UI is not required at this stage.

World Scoping
REQ-CARD-DATA-4 Card templates MUST be scoped per world. Each world's JSON defines the templates introduced by that world. draft

Zombie, Find Baseball Bat, The Walker, Listen, Baseball Bat, Regroup, and Summon Door are Zombie Big-Box content. They belong in a Zombie Big-Box JSON file, not in a global flat catalog. A card template does not exist in a world it was not defined in.

World JSON files are named by world ID (e.g., assets/worlds/zombie-big-box.json). The file name matches the worldId field inside the file.

REQ-CARD-DATA-5 Starter deck card templates MUST live in a dedicated JSON asset named starter.json, not scoped to any world. draft

Sprint, Explore, Barricade, Med Kit, Panic, and Adrenaline are world-agnostic starter cards. They ship in assets/worlds/starter.json, which is always loaded regardless of the active world. The file uses "worldId": "starter" — matching the sourceWorldId value stamped on starter deck cards (see REQ-CARD-DATA-9).

REQ-CARD-DATA-6 A world's JSON MUST include its deck composition as data alongside its card templates. draft

Deck composition specifies which templates appear in each act and in what quantities. This replaces the hardcoded act specs in src/core/world.ts. The composition is in a separate top-level key from card templates within the same file.

Minimal world JSON structure:

{
  "worldId": "zombie-big-box",
  "cardTemplates": {
    "Zombie": {
      "kind": "world",
      "name": "Zombie",
      "cost": 1,
      "keywords": ["Slow", "Creature"],
      "discardable": true,
      "penalty": { "kind": "Damage", "amount": 1 },
      "reward": { "kind": "None" }
    }
  },
  "deckComposition": {
    "acts": [
      { "cards": [{ "templateId": "Strange Sounds", "count": 2 }, { "templateId": "Zombie", "count": 3 }] },
      { "cards": [{ "templateId": "Find Baseball Bat", "count": 1 }, { "templateId": "Screams", "count": 2 }] },
      { "cards": [{ "templateId": "The Walker", "count": 1 }] }
    ]
  }
}

The starter.json file contains only worldId and cardTemplates — no deckComposition.

Catalog Assembly
REQ-CARD-DATA-7 The game layer MUST assemble a merged CardCatalog from loaded JSON before creating world state. draft

After JSON is loaded in preload, the Phaser scene (game layer) merges starter card templates and world card templates into a single CardCatalog object (Record<string, CardTemplate>). This merged catalog is the object passed to all core functions that require minting.

Template ID collisions between starter and world JSON are a data error and must be caught at assembly time, not silently overwritten.

REQ-CARD-DATA-8 The catalog MUST be passed as an explicit parameter to mintCard, createWorld, and dispatch. draft

The module-level CATALOG constant in src/core/cards.ts is removed. Functions that mint cards receive the catalog explicitly — no global, no module-level state. The intended signatures:

mintCard(catalog: CardCatalog, state: GameState, templateId: string): [PlayerCard | WorldCard, GameState]
createWorld(catalog: CardCatalog, seed: number): GameState
dispatch(catalog: CardCatalog, state: GameState, action: Action): { state: GameState; events: GameEvent[] }

Effects inside src/core/effects.ts that currently call mintCard via the global will receive the catalog threaded through from dispatch.

Card Provenance
REQ-CARD-DATA-9 Each PlayerCard instance MUST carry a sourceWorldId: string field set at mint time. draft

New field on the PlayerCard type in src/core/types.ts. Set once at mint time, never mutated. mintCard derives it from state.worldId at the moment of minting.

REQ-CARD-DATA-10 The initial GameState MUST have worldId = 'starter' before a world is entered. draft

Starter deck cards are minted before a world is active. Setting the initial worldId to 'starter' means mintCard stamps them with sourceWorldId: 'starter' naturally — no special-casing needed in the minting logic. When createWorld begins a run, it sets worldId to the active world's ID.

REQ-CARD-DATA-11 PlayerCards minted as world rewards MUST have sourceWorldId equal to state.worldId at the moment of minting. draft

When a reward effect mints a player card (e.g., Listen from Strange Sounds, Baseball Bat from Find Baseball Bat), mintCard reads state.worldId and stamps it. The card retains this ID for the duration of the run and across future runs.

REQ-CARD-DATA-12 WorldCard instances MUST NOT carry a sourceWorldId field. draft

World cards — including world cards minted by penalty effects at runtime (e.g., the Door added by The Walker's discard penalty) — are always of the current world, resolved within a run, and never travel between worlds. They are rendered using state.worldId directly. sourceWorldId on a WorldCard is misleading and must not be added.

Rendering
REQ-CARD-DATA-13 The card front renderer MUST use card.sourceWorldId to select visual theme for player cards. draft

Wherever a player card's front is rendered — hand, discard pile, reward selection — the renderer looks up VisualTheme by card.sourceWorldId, not by state.worldId. A card gained in Zombie Big-Box displays Zombie Big-Box card front art regardless of which world is currently active.

REQ-CARD-DATA-14 An unrecognized sourceWorldId MUST fall back to 'zombie-big-box' theme without crashing. draft

This is a rendering-time lookup, not a mint-time error. Consistent with the existing VisualTheme registry fallback from the visual-identity spec. The fallback is silent; no console warning is required at this stage.

REQ-CARD-DATA-15 The card back visual MUST NOT vary by sourceWorldId or any other per-card signal. draft

cardback.webp is canon and constant across all worlds. This is load-bearing for game integrity — players must not be able to identify face-down cards by back art. Inherited from the visual-identity spec.

Type Migration
REQ-CARD-DATA-16 Minting a card with a template ID absent from the catalog MUST throw a typed error. draft

Runtime validation replaces the compile-time exhaustiveness check lost by moving to JSON. mintCard throws a typed error with the invalid ID and current state if the template ID is not found — consistent with the existing error contract from the core loop spec (illegal actions throw, no silent no-ops).

REQ-CARD-DATA-17 CardTemplateId MUST become string. The TypeScript union of all template names is removed. draft

The current union type 'Sprint' | 'Explore' | 'Zombie' | ... cannot survive JSON-driven catalogs where templates are defined at runtime. Safety is provided by runtime validation at mint time (REQ-CARD-DATA-16) rather than the type system.

The type alias is retained as type CardTemplateId = string for documentation value. All internal references to the old union are updated.

Out of Scope

  • A content-authoring pipeline or card editor. JSON files are edited by hand.
  • Card art assets per world (covered by visual-identity; this spec only wires the sourceWorldId signal to the existing theme registry).
  • Meta-progression persistence across sessions. When run persistence is added, sourceWorldId must be included in the serialized PlayerCard — it is not ephemeral.
  • On-demand world JSON loading at world-selection time. All JSON required for the current run is loaded upfront in preload.
  • JSON schema validation at load time (desirable but deferred).
  • Adding a second world. This spec migrates existing Zombie Big-Box content; it does not ship new world content.

AI Validation

How to verify this spec is implemented

  1. No card data in TypeScript: grep -rn "cost:" src/core/ returns zero results. Also confirm src/core/world.ts contains no hardcoded act spec arrays.
  2. JSON assets exist: assets/worlds/starter.json and assets/worlds/zombie-big-box.json exist and contain valid template and (for the world file) deck composition data.
  3. World scoping: Zombie, Find Baseball Bat, The Walker, Listen, Baseball Bat, Regroup, Summon Door appear only in the Zombie Big-Box JSON — not in starter.json and not in any .ts file.
  4. Starter card provenance (unit): Mint a Sprint card with a state whose worldId = 'starter'. Assert card.sourceWorldId === 'starter'.
  5. Starter card provenance (integration): Call createWorld. Assert all cards in the initial playerDraw and opening hand have sourceWorldId === 'starter'.
  6. Reward card provenance: Simulate a Strange Sounds reward during a Zombie Big-Box run. Assert the minted Listen card has sourceWorldId === 'zombie-big-box'.
  7. Invalid template ID throws: Call mintCard with a template ID not in the catalog. Assert a typed error is thrown — not a silent undefined or null.
  8. Load failure surfaced: Remove or corrupt starter.json. Confirm the game surfaces an error before world state is created, not a silent empty catalog.
  9. Renderer uses sourceWorldId: With a Zombie Big-Box world active, render a hand containing a card with sourceWorldId === 'starter'. Confirm it renders with the starter/fallback card front, not the Zombie Big-Box card front.
  10. All existing tests pass: bun test green. Migrating data format must not break the game loop.
