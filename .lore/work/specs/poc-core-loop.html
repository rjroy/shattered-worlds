POC · Ultra-basic core loop ● approved REQ-POC-N 18 requirements Shattered Worlds
Spec · Proof of Concept

The smallest loop that exercises the seam

A deck of ten, draw five, play to sum, end the turn to record and redraw. No win, no loss. Its only real job is to prove the approved core/render architecture works end to end.

◇ Purpose & the loop

This POC is deliberately not a game. It is the thinnest vertical slice that drives the seam from the approved core/render architecture: a pure synchronous core that returns { state, events }, seeded RNG and shuffle, and a Phaser renderer that observes state and plays events. If this slice is typed, tested, deterministic, and visibly playable, the architecture is proven and the real game can build on it.

New game (shuffle 10, draw 5)→ Play cards in order, summing→ End Turn→ Discard hand · record total · draw 5↺

Resolved decisions & assumptions (veto any of these)

  • End Turn discards the whole hand. Both played and unplayed cards go to discard, then a fresh 5 is drawn. (Confirmed with user.)
  • Playing a card is a one-way commit within a turn. No un-play, no reorder once played. Order played = order selected. (Assumed — simplest action set.)
  • "Record total" keeps an ordered history of per-turn totals on the state; the running total resets to 0 each new turn. (Assumed.)
  • No FX/juice is required. The renderer may be visually plain. This POC proves the seam, not the feel. (Escalation/intensity is explicitly out of scope.)

The two assumed items above are not yet confirmed by the user. Raise them for sign-off before implementation begins; do not let an assumption silently become a bug.

A · Architecture & the seam
REQ-POC-1All game logic lives in the pure coreMust

Every rule of this loop is implemented under src/core/ with zero Phaser imports and no DOM/canvas access. The renderer under src/game/ may import the core; the core may never import the renderer or Phaser. The boundary is enforced by a lint rule, not convention.

REQ-POC-2The core is a synchronous dispatch functionMust

The core exposes dispatch(action) → { state, events } that returns immediately and never awaits animation or I/O. Given the same prior state and action it always returns the same next state and the same ordered event list. State is treated as immutable (each dispatch yields a new snapshot; the prior is not mutated).

REQ-POC-3All randomness is seeded and lives in the coreMust

createGame(seed) constructs a game whose every random decision (initial shuffle, reshuffle) flows through a seeded PRNG carried in the state. Same seed + same action sequence ⇒ byte-identical state and events. The renderer introduces no randomness that can affect state.

B · Cards & state
REQ-POC-4The deck is exactly ten cards in a fixed compositionMust

A new game's library is five value-1 cards, three value-2 cards, two value-3 cards (ten cards, value sum 13). Every card carries a unique stable id so otherwise-identical cards (the five 1s) can be tracked individually through hand, played, and discard, and used as render keys.

REQ-POC-5State models the four zones, the running total, and historyMust

GameState represents: the draw pile (ordered), the hand, the played-this-turn pile (ordered by play order), the discard pile, the current running total for the turn, the ordered history of completed turn totals, and the PRNG cursor (whatever opaque state the chosen PRNG requires, not assumed to be a single integer). The ten cards are conserved: at every point the four zones together contain all ten, no duplicates, none lost.

C · The turn loop
REQ-POC-6New game shuffles and deals the opening handMust

createGame(seed) builds the ten-card library, shuffles it with the seeded PRNG, draws an opening hand of 5, sets the running total to 0, and leaves the turn-total history empty. Draw pile then holds 5, discard 0. (The design's optional config parameter is intentionally omitted for this POC; the exact PRNG algorithm and seed type remain deferred to implementation, per the approved design.)

REQ-POC-7Drawing reshuffles the discard when the draw pile runs dryMust

A draw of 5 pulls from the top of the draw pile. If the draw pile empties before 5 are drawn, the discard pile is shuffled (seeded) to form a new draw pile and drawing continues until 5 are in hand. With ten cards total this always succeeds. Reshuffle emits a DeckShuffled event.

REQ-POC-8PlayCard moves a card to the played pile and adds its valueMust

A PlayCard(cardId) action moves that card from hand to the played-this-turn pile, appended in play order, and adds its value to the running total. Only cards currently in hand are legal targets. Playing is one-way within the turn (no un-play). Emits a CardPlayed event carrying the card id, its value, and the new running total.

Illegal actions throw. Dispatching PlayCard with an id that is not currently in hand (unknown, already played, or already discarded) throws a typed error carrying the offending action and the current state. The core never silently no-ops and never returns an error event for legal-vs-illegal; dispatch's { state, events } return is for valid actions only. (This is the single error contract for every action in this POC.)

REQ-POC-9EndTurn discards the hand, records the total, and redrawsMust

An EndTurn action, in this strict order: (1) appends the current running total to the turn-total history; (2) moves every card in both the hand and the played pile to the discard pile; (3) resets the running total to 0; (4) draws a fresh 5 (reshuffling per REQ-POC-7 as needed). Steps (1)–(3) must complete before step (4) begins, so the draw step always sees the full discard pool. With ten cards and a draw of five, step (4) empties the draw pile and therefore triggers exactly one seeded reshuffle per End Turn; this is expected, not an edge case.

If both hand and played piles are empty (an End Turn with nothing to discard), the action still records the running total (which is 0) and draws normally. Emits an ordered event sequence TurnEnded{total}, CardsDiscarded{cardIds}, optional DeckShuffled, CardsDrawn{cardIds}; each card-moving event carries the specific card ids it moved, so the renderer can animate per card (REQ-POC-11).

REQ-POC-10There is no win, loss, or end conditionMust

The loop runs indefinitely. There is no score target, no enemy, no failure state. The only notion of a "turn" is the record-total step in REQ-POC-9. EndTurn is always legal even with an empty played pile (records a total of 0).

D · Events & renderer
REQ-POC-11Every dispatch returns an animation-sufficient event logShould

The events returned alongside each new state are an ordered, semantic list (card played, cards discarded, deck shuffled, cards drawn, turn ended with total). Card-moving events carry the specific card ids involved, and TurnEnded carries the recorded total, so the renderer can drive per-card animation from the log alone, without diffing two states. The renderer remains free to ignore the log and snap to state; note that the renderer's display obligations in REQ-POC-12 are satisfied from state, so this requirement is "Should" (it enriches animation) while REQ-POC-12 stays "Must".

REQ-POC-12A Phaser renderer makes the loop playableMust

A Phaser scene observes core state and displays: the hand (clickable to play), the played pile with the running total, the draw and discard pile counts, the turn-total history, and an End Turn button. Clicking a hand card dispatches PlayCard; the button dispatches EndTurn. The renderer owns the clock: it reflects the resulting state (animation of events is allowed but not required for the POC). Visual polish is explicitly not required.

E · Testing, CI, and the second driver
REQ-POC-13The core is unit-tested headlessly and gated in CIMust

Core tests run with no Phaser, no DOM, no canvas and cover: deck composition (REQ-POC-4), deterministic shuffle from seed (REQ-POC-3), draw and reshuffle (REQ-POC-7), play summation (REQ-POC-8), the full end-turn cycle (REQ-POC-9), the ten-card conservation invariant (REQ-POC-5), and replay equivalence (same seed + actions ⇒ identical state & events). CI runs the import-boundary lint and the core test suite on every change.

REQ-POC-14A headless sim runner drives the core without the rendererShould

A small headless runner uses only the core to dispatch a long sequence of random-legal actions at full speed (no Phaser). Action policy: at each step, if the hand is non-empty, choose uniformly among playing each individual hand card or ending the turn; if the hand is empty, end the turn. Because a fresh hand of 5 is dealt after every End Turn, the runner never dispatches an illegal action, so it must complete without throwing while the ten-card conservation invariant holds throughout. This proves the "one core, two drivers" property the architecture depends on for future balance data.

F · Build, serving & deploy

Phaser is a browser app, so "serving" means bundling to a static web app (HTML + JS + assets) that runs client-side. There is no server-side runtime for the game; hosting is static file hosting.

REQ-POC-15Toolchain is Bun + Vite + strict TypeScriptMust

The project uses Bun as runtime and package manager, Vite to bundle Phaser + TypeScript, and the core test suite runs under bun test. TypeScript runs in strict mode. Per reproducible-build rules, dependency versions are pinned and the lockfile is committed, so bun install reconstructs the exact dependency graph.

REQ-POC-16A local dev server runs the playable gameMust

bun run dev starts the Vite dev server and serves the Phaser game in the browser with hot reload. This is the command behind the "visibly playable" validation of REQ-POC-12.

REQ-POC-17A static build emits a self-contained bundleMust

bun run build produces a self-contained static dist/ (HTML, JS, assets) that runs from any static host with no server-side code. Because the target is a GitHub Pages project site served from a subpath, Vite's base is set to that repository subpath so asset URLs resolve correctly (the classic Pages 404-on-assets trap is avoided by configuration, not by hand-editing output).

REQ-POC-18CI builds, tests, and deploys to GitHub PagesMust

A CI workflow on push to the default branch runs, in order: the import-boundary lint and core test suite (REQ-POC-13) and the static build (REQ-POC-17) as gates, and only on success deploys dist/ to GitHub Pages. A failing lint, test, or build blocks deploy. The deployed URL serves the same playable loop as local dev. This makes REQ-POC-13's CI gate and the deploy one pipeline.

✓ AI Validation

How the AI confirms this POC is done. Each check is runnable or observable, not a matter of opinion.

▸Tests & lint pass headlessly. The core test suite and the import-boundary lint run green with no Phaser/DOM in scope. REQ-POC-1, 13
▸No engine in the core. A search of src/core/ for phaser / canvas / DOM imports returns nothing. REQ-POC-1
▸Deck composition is exact. A fresh game's ten cards are five 1s, three 2s, two 3s (value sum 13), each with a unique id. REQ-POC-4
▸Determinism holds. Tests use at least two distinct seeds: with one seed, building two games and applying the same action sequence yields deep-equal final state and deep-equal event logs; the second, different seed produces a different initial shuffle order. REQ-POC-3, 13
▸Dispatch is synchronous. dispatch's return type is { state, events }, not a Promise/thenable; a test asserts the result is not awaitable. REQ-POC-2
▸ Cards are conserved. Across an automated run of many turns, the four zones always total exactly ten distinct cards, never more, fewer, or duplicated. REQ-POC-5, 14
▸Summation is correct. After any sequence of plays, the running total equals the sum of the played cards' values; after EndTurn the history gains exactly one entry equal to that total and the running total is 0. REQ-POC-8, 9
▸Reshuffle works. Driving enough turns to empty the draw pile triggers a seeded reshuffle (observable via a DeckShuffled event) and drawing still reaches a hand of five. REQ-POC-7
▸The sim runner survives a long run. Dispatching ≥1000 random-legal actions through the core alone completes without error and preserves the conservation invariant throughout. REQ-POC-14
▸It is visibly playable locally. bun run dev serves the game; the browser shows a hand, a running total, pile counts, the turn-total history, and a working End Turn button; clicking cards and ending turns advances the loop. REQ-POC-12, 16
▸The bundle builds reproducibly. bun install from the committed lockfile then bun run build produces a static dist/ with no errors. REQ-POC-15, 17
▸Assets resolve on the Pages subpath. Serving the built dist/ under the project subpath (locally via vite preview or on Pages) loads the game with no 404s on JS/assets. REQ-POC-17
▸CI deploys only on green. The workflow runs lint + core tests + build as gates and deploys dist/ to GitHub Pages on success; a deliberately failing test blocks the deploy. The live URL serves the playable loop. REQ-POC-18

✕ Out of scope

Explicitly not in this POC, to keep the slice thin: card acquisition or removal, enemies or combat, Destiny / worlds / between-run economy, scoring beyond raw summation, any win or loss condition, save/load, content authoring pipeline, FX / juice / the intensity escalation signal, and animation polish. These belong to later specs. The structural seam they will all sit on is what this POC validates.
