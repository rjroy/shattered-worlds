Help screen

Spec · 2026-06-08 · status: draft
Related: brainstorm, world-select spec

A Phaser overlay triggered by a ? button in the top-right corner of the canvas. Shows base rules, a keyword glossary, and world-specific mechanic notes for the active shard. Designed to be consulted mid-run without leaving the canvas.

Trigger — ? button

REQ-HELP-1 A ? button is visible during active gameplay in the top-right corner of the canvas Must

A standalone ? button is created in TableScene.create(), positioned near the top-right corner of the 900×600 canvas (approximately x=860, y=22). It is not part of the HUD container — it is a separate game object in the scene.

The button is visible while state.status === 'playing'. It is hidden when the win or loss overlay is shown. The button may remain visible during a modal chooser (branch-selection UI for certain cards) — opening help from that state is allowed, and dismissing help returns to the modal chooser unaffected. On pointerup, it shows the help overlay.

Visual treatment: a small rounded rectangle or text button labeled "?", clearly legible against the dark backdrop. A hover state (brightness or scale change) is sufficient affordance.

Overlay architecture

REQ-HELP-2 Help overlay is a Phaser container at depth 1000, initially hidden Must

A createHelpOverlay(scene, worldId, totalActs) factory in src/game/view/overlays.ts creates the overlay and returns the container. The container is positioned at (CANVAS_W / 2, CANVAS_H / 2), depth 1000, and setVisible(false) by default — matching the pattern of createWinScreen and createLossScreen.

TableScene calls createHelpOverlay(this, worldId, state.totalActs) during create() once the GameCore instance is available. It stores the returned container reference and toggles its visibility in response to the ? button and dismiss controls.

REQ-HELP-3 While the overlay is visible, game board interaction is suspended Must

The overlay's background rectangle calls setInteractive(). Phaser's default topOnly behavior means that interactive objects below the topmost interactive object under the pointer do not fire events — so the overlay background, sitting at depth 1000, swallows all pointer events that would otherwise reach cards and action buttons at lower depths.

No explicit per-card disableInteractive loop is required. The depth-ordering guarantee is sufficient for current content. If topOnly behavior is ever disabled globally, revisit this requirement.

REQ-HELP-4 Overlay dismisses on X button click or Escape key Must

An X button is positioned in the top-right of the overlay panel. Clicking it hides the container and restores normal gameplay. Pressing the Escape key has the same effect. The Escape handler is registered in TableScene and only fires if the help overlay is visible.

After dismissal, the ? button remains visible and can reopen the overlay.

Canvas is 900 × 600. The overlay covers the full canvas. All content is in a centered panel with 50px horizontal padding.

REQ-HELP-5 Overlay contains three sections in order: Base Rules, Keywords, In This World Must

The panel is divided into three labeled sections, separated by visual dividers. Section order is always Base Rules → Keywords → In This World. Each section has a distinct uppercase header label.

All three sections must be visible simultaneously without scrolling for the current content volume (see REQ-HELP-13). The first section header label reads "HOW IT WORKS"; the second reads "KEYWORDS"; the third reads "IN THIS WORLD — [name]".

Content — base rules

REQ-HELP-6 Base rules section shows four static facts: HP, energy, hand size, act count Must

The Base Rules section displays the following, on one or two lines:

  • HP: 10 (drawn from WORLD_CONSTS.startHp)
  • Energy: +1 per turn
  • Hand: 6 cards (drawn from WORLD_CONSTS.maxHandSize)
  • Acts: 3 per run (passed in as totalActs to the factory — see REQ-HELP-2)

HP and hand size are read from WORLD_CONSTS (accessible inside overlays.ts without additional parameters). Act count is passed in from state.totalActs at the call site in TableScene.create(). None of these values are hard-coded as literals in the overlay factory.

Content — keyword glossary

REQ-HELP-7 Keywords section defines seven terms in a two-column layout Must

The seven terms and their definitions (authoritative source for implementers):

TermDefinition
Energy Resource earned at the start of each turn (+1). Spent to play cards with an energy cost. Some cards grant additional energy mid-turn.
Progress Damage accumulated on a hazard. When progress equals or exceeds the hazard's cost, the hazard is cleared and its reward fires.
Hazard A world card dealt into your hand. Three things can happen to it: Clear it by filling the progress ring (fires its reward). Discard it voluntarily (fires its discard effect — may cost HP, add bad cards, or occasionally help). Or survive the turn with it still in hand (fires its end-of-turn effect, usually a penalty). A hazard's card face shows all three.
Hidden Keyword on a hazard. Cards like Explore and Listen deal bonus progress against Hidden hazards.
Creature Keyword on a hazard. Cards like Baseball Bat and Fire Axe deal bonus progress against Creature hazards.
Slow Keyword on a hazard. Sprint's movement branch deals bonus progress against Slow hazards.
ForceDestroy Queues one random player card to be permanently destroyed from your next hand. Multiple charges stack.

The two-column layout places Energy, Progress, Hazard, Hidden in the left column and Creature, Slow, ForceDestroy in the right column. With 50px horizontal padding on each side, the usable panel width is ~800px. Each column is approximately 380px wide with a ~40px gap. These widths are the wordWrapWidth constraint for the text objects — without an explicit wrap width, Phaser text objects do not auto-wrap into columns. Keyword names are styled distinctly from their definitions (e.g., accent color or bold).

The Hazard entry is intentionally multi-line (it covers three interaction paths: clear, discard, end-of-turn). It will be taller than the other entries. The left column should be sized to accommodate Hazard's wrapped definition without overflowing into the section below. Row heights in the two-column layout are not equal — the implementer should position each entry below the previous one rather than assuming a fixed row height.

Keyword definitions are static strings in overlays.ts — they are game-rules prose, not data-driven. If definitions need per-world variation in the future, that is a new requirement.

Content — in this world

REQ-HELP-8 "In This World" section title shows the world's display name; content is world-aware Must

The section header reads "IN THIS WORLD — [name]" where [name] is the world's WorldDisplayData.name from worldDisplayManifest (e.g. "The Big Box").

Each mechanic note is rendered as a title line followed by a detail sentence. The title is styled distinctly (accent color or bold); the detail is standard body text.

If worldHelpManifest[worldId] is undefined at runtime, throw immediately with a clear error: "No help entry for worldId: <id>". The completeness test (REQ-HELP-12) is the primary guard; the throw is belt-and-suspenders.

Content authoring model

REQ-HELP-9 WorldHelpData type and worldHelpManifest are defined in src/data/worldHelpManifest.ts Must

A new file src/data/worldHelpManifest.ts exports:

export interface WorldMechanicNote {
  title:  string   // Short label, e.g. "Zombies grow"
  detail: string   // 1–2 sentences explaining the mechanic
}

export interface WorldHelpData {
  mechanics: readonly WorldMechanicNote[]
}

export const worldHelpManifest: Record<string, WorldHelpData>

This file has no game-logic dependency and must NOT be imported by core or engine modules. It mirrors the separation established by worldDisplayManifest.ts.

REQ-HELP-10 zombie-big-box has four mechanic notes covering its escalation behaviors Must

The 'zombie-big-box' entry must cover these four mechanics:

  • Zombies grow — A Zombie in your hand deals 1 damage and adds another Zombie to the world deck at end of every turn. Clear them before they multiply.
  • Corpses transform — A Corpse becomes a Zombie at end of turn. Corpses cannot be discarded; they must be cleared or endured.
  • Costly discards — Discarding a Zombie costs 5 HP. Know your limits before you cut your losses.
  • The Walker and the Door — The Walker (Act 3, cost 10) is the final hazard. Clearing it adds Summon Door to your hand. Play Summon Door to place the Door (a Hidden hazard, cost 4). Clear the Door to survive the world.

Exact wording is up to the implementer as long as each note accurately describes the mechanic named above.

REQ-HELP-11 bird-building and highway-volcano have placeholder mechanics entries Must

'bird-building' and 'highway-volcano' each have at least one WorldMechanicNote in their mechanics array. Placeholder notes acknowledge the world without misleading the player — for example:

  • title: "Mechanic notes coming"
  • detail: "This world's mechanics are still being tuned. Content will be added after rebalancing."

The placeholder text must be non-empty strings. An empty array would fail the completeness test (REQ-HELP-12). Full mechanic notes for these worlds are deferred until rebalancing is complete.

Tests

REQ-HELP-12 A unit test verifies all worlds have a worldHelpManifest entry with at least one mechanic note Must

A unit test confirms:

  • Every key in Object.keys(worldManifest) has a corresponding entry in worldHelpManifest. The test iterates keys only — it must not invoke the builder functions (calling them throws CatalogError without the Phaser asset cache).
  • Each entry's mechanics array has at least one note.
  • Each note has non-empty title and detail strings.

This test guards against adding a new world without a help entry, which would produce a runtime crash when the player opens help in that world.

Layout constraint

REQ-HELP-13 All three sections fit within the 900×600 canvas without scrolling at current volume Must

For zombie-big-box (the fullest world), all three sections must be simultaneously visible with no vertical overflow beyond the 600px canvas height. The layout sketch above fits: header (~50px) + base rules (~80px) + keywords (~155px, accounting for Hazard's multi-line entry) + in-this-world (~180px) + internal padding ≈ 565px — within budget.

If future content growth pushes beyond this constraint, scrolling may be added in a separate spec. At current volume, Phaser text objects are sufficient — no DOM overlay is needed. This is an explicit revisit trigger: if mechanic notes for all worlds ever exceed ~6 entries total, revisit the layout.

Not in scope

Scrollable glossary — Phaser text with drag/mask. Deferred; current volume fits without scrolling. Revisit if glossary grows past ~15 keyword terms.
First-run help prompt — Automatically showing help on the first run of a world. Deferred; requires first-run state tracking.
Keyword tooltips on cards — Hovering a keyword term on a card face shows its definition inline. Deferred; requires card-face text to carry structured keyword links.
Full bird-building and highway-volcano mechanic sections — Complete "In This World" content for these worlds. Deferred pending rebalance. Bird-building's notable mechanics (Gripping Talon's ForceDestroy on discard, dual Slow+Creature keyword) should be documented when that world is locked.
Pause menu integration — If a pause screen is ever added, help should live there. For now, the standalone ? button is sufficient.
