What

Wire the front-art lookup to the per-card provenance signal, with a pure, headless-testable helper.

  • src/game/render.ts — extract a pure helper: 
    export function selectCardFrontKey(
  card: Card,
  activeTheme: VisualTheme,
  resolveTheme: (worldId: string) => VisualTheme,
): string
    World card → activeTheme.worldCardfrontKey ?? 'cardfront' (themed by the active world). Player card → resolveTheme(card.sourceWorldId); since no theme defines a player-card front (out of scope), this resolves to the generic 'cardfront'. Unknown sourceWorldId falls through selectTheme's existing 'zombie-big-box' fallback without throwing.
  • src/game/render.ts — createCardObject(scene, card, x, y, activeTheme, resolveTheme) uses the helper in place of the inline card.kind === 'world' ? ... : 'cardfront' at lines 50–51.
  • src/game/TableScene.ts — pass selectTheme (from theme.ts) as resolveTheme at the createCardObject call site (~line 252). Frame/highlight styling stays on the active theme.
Seam, not a visible change. Player cards keep rendering the generic front today; per-world player art is out of scope. The point is that the lookup is now keyed by sourceWorldId for the future. The card back (cardback.webp) is never touched here.

Validation — automated (headless)

  • World card: selectCardFrontKey(worldCard, activeTheme, resolveTheme) returns the active theme's worldCardfrontKey.
  • Player card, foreign provenance: a player card with sourceWorldId === 'zombie-big-box' while the active world differs returns generic 'cardfront', never the zombie world front key.
  • Unknown provenance: a player card with an unrecognized sourceWorldId returns a key without throwing (relies on selectTheme's 'zombie-big-box' fallback).
  • Suite: bun test green; new selectCardFrontKey tests pass.

Validation — manual (runtime)

  • AI-Validation #9: with Zombie Big-Box active, a hand containing a 'starter' player card renders the starter/fallback front, not the Zombie Big-Box world front.
  • Card back constant: face-down cards still show cardback.webp regardless of any per-card signal.

Why

  • REQ-CARD-DATA-13 front renderer uses card.sourceWorldId to select theme for player cards.
  • REQ-CARD-DATA-14 unrecognized sourceWorldId falls back to 'zombie-big-box' without crashing.
  • REQ-CARD-DATA-15 card back must not vary by any per-card signal.
