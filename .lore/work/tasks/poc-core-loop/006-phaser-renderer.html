Phaser renderer makes it playable

REQ-POC-12REQ-POC-11REQ-POC-16

What

A plain but playable Phaser 3 scene that observes core state and turns clicks into actions. Visual polish is explicitly out of scope; the bar is "visibly playable".

  • src/game/main.ts — Phaser 3 Game bootstrap (config, canvas size, single scene), mounted into index.html.
  • src/game/TableScene.ts — holds one GameCore (createGame(seed)). Renders the hand (clickable ⇒ dispatch(PlayCard)), the played pile with live running total, draw and discard counts, the turn-total history, and an End Turn button (⇒ dispatch(EndTurn)). After each dispatch it re-reads state and redraws. Event-log animation allowed, not required.
  • src/game/render.ts (optional) — small pure helpers (card label, layout positions) to keep the scene under the size heuristic.

The renderer is the sole importer of phaser. Write against the pinned Phaser 3.x scene API — do not assume an older version's signatures. The renderer owns the clock; for the POC it may snap straight to state after each dispatch.

✓ Validation (observable + automated)

▸bun run typecheck and bun run lint pass; the boundary lint confirms no phaser/DOM import has leaked into src/core/.
▸bun run dev serves the game; the browser shows hand, running total, pile counts, history, and a working End Turn button; clicking cards and ending turns advances the loop. Confirm via the verify/run flow with a screenshot.

Why

REQ-POC-12 requires a Phaser renderer that makes the loop playable from state, with clicks dispatching actions; REQ-POC-16 requires bun run dev to serve it with hot reload. The renderer consuming id-carrying events satisfies the consumption side of REQ-POC-11. This is the "visibly playable" proof the seam works end to end. See plan Step 6.
