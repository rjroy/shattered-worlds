Seeded PRNG & deterministic shuffle

REQ-POC-3

What

The single source of all game randomness — purely functional and serializable — so same seed + same actions reproduce byte-identical runs.

  • src/core/rng.ts (imports RngState from types.ts):
    • createRng(seed: number): RngState — expand the numeric seed through splitmix32 four times to fill {a,b,c,d}.
    • nextFloat(rng): [value, next] — one sfc32 step; value in [0,1). Pure: never mutates the input.
    • shuffle<T>(items: readonly T[], rng): [shuffled, next] — Fisher-Yates threading rng state through each swap.

Returning [value, nextState] rather than mutating keeps the RNG inside the immutability story and makes it testable without injection or module mocking.

✓ Validation (bun test, no DOM)

▸Determinism: two createRng(sameSeed) states produce identical nextFloat sequences over N draws.
▸Shuffle determinism: same seed + same input array ⇒ identical permutation; a different seed ⇒ a different permutation.
▸Purity: calling nextFloat(state) twice on the same input returns the same value, input unchanged.

Why

REQ-POC-3 requires all randomness seeded and inside the core, with same seed + same actions yielding byte-identical state and events. A purely functional PRNG whose cursor lives in state is the mechanism the determinism and replay guarantees of later steps depend on. See plan Step 3.
