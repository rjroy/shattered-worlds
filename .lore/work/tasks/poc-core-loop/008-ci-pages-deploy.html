CI: gate then deploy to GitHub Pages

REQ-POC-18REQ-POC-13

What

One pipeline on push to master: lint + typecheck + core tests + build as gates, deploy dist/ to Pages only on success.

  • .github/workflows/ci.yml — on: push: branches:[master] (plus PR for the gate-only run). Steps: checkout → oven-sh/setup-bun → bun install --frozen-lockfile → bun run lint → bun run typecheck → bun run test → bun run build. Then, gated on success: actions/configure-pages → actions/upload-pages-artifact (path dist) → actions/deploy-pages.

typecheck is a gate too: ESLint catches import-boundary breaks, but only tsc --noEmit enforces strict-mode type errors (REQ-POC-15).

⚠ Manual prerequisites (flag to user — workflow can't self-serve)

  • Set Pages → Source = GitHub Actions in repo settings. Deploy fails without it.
  • Grant the workflow pages: write and id-token: write permissions and target a github-pages deployment environment. A first-time Pages-via-Actions deploy can fail for these reasons even with the Source set.

✓ Validation

▸The workflow runs green and deploys; the live rjroy.github.io/shattered-worlds/ URL serves the same playable loop as local dev.
▸A deliberately failing test (on a throwaway branch, then reverted) blocks the deploy job — proving the gate.

Why

REQ-POC-18 requires CI on the default branch to run lint + core tests + build as gates and deploy dist/ to Pages only on success, making REQ-POC-13's CI gate and the deploy a single pipeline. A failing gate must block deploy. See plan Step 8.
