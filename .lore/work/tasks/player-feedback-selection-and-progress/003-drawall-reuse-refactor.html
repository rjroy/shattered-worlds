S3

Refactor drawAll() to reuse card containers

pending highest risk enabling refactor (FEEDBACK-1/2/7/8/9)

Dependencies: none, but foundational — S4, S5, S6, S7, S9, S10 all build on persistent containers. Confirmed architecture choice (reuse over cache-and-tween).

What

Stop destroying every card container each cycle. Reconcile this.cardObjects against the new hand: update existing cards in place, create only newly-present cards, destroy only cards that left the hand. Behavior must be identical to today after this step — no new feedback yet.

  • render.ts: keep createCardObject for the static face (name/effect/cost text — immutable per card id). Factor mutable visuals into update helpers: position, highlight stroke (applyCardHighlight exists), dim (dimCard exists).
  • drawAll: compute the desired ordered card list per row. For each desired card: if cardObjects.has(id), reuse and re-position; else createCardObject, attach pointerdown/over/out handlers once, add to the map. After layout, destroy any container whose id is no longer desired (call scene.tweens.killTweensOf on its animated children first).
  • Handlers capture only the stable id and read this.sel at call time — attach-once stays correct.
  • Reposition a reused card by setting x/y directly (no position tween required).
  • Track the hovered card id on the scene so the update path can re-assert base transform on non-hovered cards (consumed by S9).
Highest-risk step. Rewrites the hot path. Heed the visual-identity rollout retro: never hold a stale Tween reference — use killTweensOf + scene.tweens.add. Keep the change behavior-preserving.

Validation

  • Full bun test green — no regression below the 149 baseline; tsc/eslint clean.
  • Manual smoke (dev build, Zombie Big-Box run): select a card, target a hazard, commit, discard a hazard, end turn, draw — every interaction works exactly as before. No orphaned or duplicated cards; no dead clicks.
  • Structural invariants (also the S3 fresh-eyes review scope): (1) every card id present before a cycle has exactly one container after; (2) pointer handlers are attached once, not re-bound per cycle; (3) killTweensOf runs before any container is destroyed; (4) no code path destroys a card still in state.hand.
  • Record whether the open clicking/input bug from the rollout is affected (do not assume a fix).

Why

Persistent containers are the precondition for the ring animations (FEEDBACK-8/9) and for hover emphasis/connector not racing the rebuild (FEEDBACK-1/2). This refactor replaces the spec's suggested cache-and-tween mechanism, per the user-confirmed architecture decision in the plan. Per plan step S3.
