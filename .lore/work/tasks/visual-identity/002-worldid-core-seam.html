Add worldId to the core as pure data

REQ-VIS-3161

What

The single permitted core change: name the active world so the renderer can pick a theme. Data only — no colors, no presentation, nothing the rules read.

  • src/core/types.ts: add worldId: string to the GameState interface as a required field.
  • src/core/world.ts: set worldId: 'zombie-big-box' in the skeleton state inside createWorld.
  • Fix every full-literal GameState construction the compiler now flags. Grep both src/core/ and src/game/ first: fixtures that fork createWorld(42) (e.g. core/golden.test.ts makeState) inherit the field, but hand-rolled literals do not — src/game/describe.test.ts builds its own makeState and will need worldId added.

Validation

  • bun run typecheck, bun run lint, and bun test all pass (the full core suite is green).
  • The import-boundary lint stays green — worldId is plain data, no presentation pulled into core.
  • A grep of src/core/ finds no color / hue / theme / Phaser reference; the only addition is the worldId string.
  • createWorld(42).worldId === 'zombie-big-box'.

Why

The renderer-side theme registry (S3) keys off state.worldId without the core ever knowing themes exist — that is what keeps colors out of the pure layer (REQ-VIS-3) and the boundary intact (REQ-VIS-1, 16). Required (not optional) is the typed-discipline choice. See plan Step 2.
