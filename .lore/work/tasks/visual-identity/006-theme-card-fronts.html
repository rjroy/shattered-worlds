Theme the card fronts

REQ-VIS-67

What

Make fronts vary by world without touching layout or the self-describing faces.

  • In src/game/render.ts, replace the hardcoded COLORS object with values pulled from the active VisualTheme (frameStyle, realityPalette, intrusion-hue accents). Thread the theme through: createCardObject(scene, card, x, y, theme).
  • In TableScene, compute the active theme once (selectTheme(this.game_.state.worldId)) and pass it into each createCardObject call in layoutRow.
  • Preserve the existing text layout and the describeEffect / describePenalty / describeReward content verbatim — retint and reframe only, do not move or rewrite faces.
  • Keep player vs world cards visually distinct (the two-row, two-treatment split survives).
  • Choose front text colors that clear WCAG AA against the busiest themed background.

Validation

  • In the browser (bun run dev): fronts carry the big-box palette; the face content and its layout are unchanged from before theming.
  • Player and world cards remain clearly distinguishable at a glance.
  • Contrast measured with a tool: cost, keywords, penalty, reward, and effect text all meet WCAG 2.1 AA (≥ 4.5:1 body, ≥ 3:1 large) against the themed front. Not eyeballed.
  • bun run typecheck / lint pass; the boundary lint stays green (render.ts is allowed to import Phaser; it must not import from a deeper core path than core/index).

Why

Fronts varying by world is the visual half of "worlds are identity, not wallpaper" (REQ-VIS-6); legibility is a hard floor regardless of aesthetic (REQ-VIS-7, vision Principle 5). See plan Step 6.

Files

  • src/game/TableScene.ts edit (compute + pass theme)
