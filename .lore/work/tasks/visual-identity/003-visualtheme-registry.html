The VisualTheme registry

REQ-VIS-415

What

The renderer-side theme abstraction and its one concrete theme. Plain data, no Phaser, so the test runs headless under bun test.

export interface VisualTheme {
  worldId: string
  intrusionHue: string            // '#9bff6a' for big-box (fluorescent green)
  realityPalette: string[]        // desaturated slate base
  frameStyle: FrameStyle         // front frame/chrome colors + stroke
  grain: GrainSpec               // distress params (numbers) — STATIC for now
  backdrop: { realityKey: string; intrusionKey: string }
  walker?: { textureKey: string }   // present iff the world has a Walker
}
export function selectTheme(worldId: string): VisualTheme
  • Define VisualTheme and its sub-types (FrameStyle, GrainSpec) as plain data. Asset references are string keys (realityKey, intrusionKey, textureKey) — never imported URLs. The key→URL mapping lives only in the Phaser glue (S5).
  • Build the zombie-big-box theme: green intrusion hue (#9bff6a), slate reality palette, and front colors chosen so card text clears WCAG AA against the themed background (the actual contrast check happens in S6, but pick AA-capable colors here).
  • walker: { textureKey: 'walker' } for this theme; backdrop: { realityKey: 'bigbox-reality', intrusionKey: 'bigbox-intrusion' } (keys must match the S5 manifest).
  • selectTheme is total: an unknown worldId returns the zombie-big-box theme (the only shipped theme is the default).
  • GrainSpec is a static per-theme constant — define and populate it but do not wire it to anything. Driving grain from intensity is REQ-VIS-14, deferred.
  • No Phaser import anywhere in this file.

Validation

  • theme.test.ts green under bun test with no Phaser/DOM in scope.
  • selectTheme('zombie-big-box') returns the theme; selectTheme('does-not-exist') returns the same zombie-big-box default (assert deep-equal or identity).
  • A grep confirms theme.ts imports no phaser and no .webp.
  • The theme's backdrop/walker keys are plain strings (no URL, no import).

Why

This is the seam that lets fronts and backdrops vary by world while the core stays theme-blind (REQ-VIS-4). Keeping it pure and key-based is what makes the mappers/registry headlessly testable (REQ-VIS-15) — the load-bearing testability guarantee. See plan Step 3.
